   1. Agent 能力正在模块化，`基于 Claude Code SDK，Cursor 级别的 Agent 体验已经可以快速搭建，不再是护城河`
   2. **传统 IDE 能力被集体忽视**，所有玩家都在卷 Agent，但跳转、搜索、LSP、重构这些"基本功"普遍拉胯

如果说第三方应用是套壳，有的壳是柱子，模型能力涨上来后，柱子就没了。有的壳是船，模型能力提升后，水涨船高。随着模型能力的提升，你的产品体验是否变得更好？这是核心。不过，是否是船也是相对的，在模型能力大变化的下，船也可能不是船。”之前的水是模型能力，现在的水变成 Agent 能力了.
需要判断水会涨多少，造船厂的灯塔有时候也是有必要的，灯塔可能会被淹没，但是造出来的船不会。

将“模型能力”比作水，将“产品形态”比作船或柱子，揭示了在 AI 浪潮中开发者定位的本质区别。

结合 JetBrains Air 和 Claude Code SDK，可以从以下三个层面理解这一隐喻：

### 1. 柱子 vs 船：从“补丁”到“放大器”

- **柱子（Pillar）**：这类产品本质上是在**填补模型当前的缺陷**。例如，早期的长文本处理工具、简单的代码翻译器。当模型原生支持 200K 上下文或具备更强的推理能力时，这些“柱子”就被淹没了，失去了存在的意义。
- **船（Boat）**：这类产品是**模型能力的载体和放大器**。JetBrains Air 就是典型的“船”。它不卷 Agent 算法本身，而是提供 Agent 运行所需的**顶级基础设施**（LSP、索引、重构引擎）。模型越强，这艘船跑得越快，因为更强的模型能更精准地调用这些复杂的 IDE 工具。

### 2. “水”的演变：从 LLM 到 Agentic Workflow

- **过去的水（模型能力）**：主要是文本生成、逻辑推理。
- **现在的水（Agent 能力）**：是**自主决策、工具调用和闭环执行**。
- **理解变化**：Claude Code SDK 的出现，实际上是把“Agent 循环”变成了基础设施（水）。如果你的产品还在辛苦地手写 `ReAct` 循环，你就是在造一根随时可能被淹没的“柱子”。而 Air 选择直接套用 Claude Code SDK，是直接把船开到了“Agent 能力”这股更急、更深的水流上。

### 3. 为什么 JetBrains 的“船”更稳？

在模型能力大变化的背景下，船也可能翻。但 JetBrains 找准了**“非智能部分的护城河”**：

- **确定性 vs 随机性**：Agent 的决策是随机的，但代码跳转、符号索引、语法树分析是确定性的。
- **环境即护城河**：Agent 需要一个“身体”来感知和操作代码世界。JetBrains 20 年积累的 IDE 能力，就是为 Agent 提供的最强“外骨骼”。
- **水涨船高**：当 Claude 的推理能力从 Sonnet 提升到 Opus 级别时，它对 IDE 工具的利用率会呈指数级增长。Air 的体验会因为模型升级而自动进化，这就是典型的“船”式产品。

### 总结

**“做船”的核心在于：你的产品是否为模型提供了它无法自我生成的“外部环境”和“专业工具”。**

如果一个产品的功能，模型通过简单的 Prompt 或原生更新就能实现，那它就是柱子；如果一个产品能让模型在其中“如鱼得水”，且模型越强，对该环境的依赖和利用就越深，那它就是一艘能承载 AI 浪潮的船。JetBrains Air 的逻辑正是：**让 Anthropic 负责“大脑”（水），自己负责“身体”和“甲板”（船）。**
