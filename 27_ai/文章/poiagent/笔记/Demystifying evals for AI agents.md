# [Demystifying evals for AI agents](https://www.anthropic.com/engineering/demystifying-evals-for-ai-agents)

2026.1.9

以下是对 Anthropic 官方文章《Demystifying evals for AI agents（揭秘 AI Agent 的评估）》的精简讲解：

### 核心观点

随着 AI Agent 从单轮对话演变为多轮自动化操作，传统的测试方法已失效。**如果没有一套完善的自动化评估系统（Evals），Agent 的开发将陷入“盲飞”状态**——只能被动等待用户报错，且修复一个 Bug 往往会引发新 Bug。构建 Evals 不仅是质量保障，更是加速研发和快速适配新模型的利器。

### 评估系统 (Evals) 的核心组件

- **任务 (Task)**：带有明确输入和成功标准的单个测试用例。
- **轨迹 (Transcript)**：Agent 尝试完成任务的完整运行记录（包含所有的 API 调用、工具使用和内心思考）。
- **结果 (Outcome)**：经过运行后环境的最终真实状态（例如：数据库里是否真的生成了订单）。
- **打分器 (Grader)**：判断任务是否成功的逻辑。分为三种：
  1. **代码驱动**（精准、便宜，如单元测试、正则匹配，但缺乏灵活性）。
  2. **模型驱动 / LLM 裁判**（灵活，适合开放式任务，但需防幻觉并用人类标准校准）。
  3. **人工评估**（最准确的“黄金标准”，但极其昂贵且缓慢）。

### 两类不同的评估目标

1. **能力评估 (Capability Evals)**：“它能做到多好？”——专门针对 Agent 不擅长的难题，初始通过率通常很低，用于指引研发方向。
2. **回归评估 (Regression Evals)**：“它原来的功能还在吗？”——针对已掌握的能力，要求接近 100% 的通过率，防止代码或提示词修改导致原有功能退化。

### 针对不同类型 Agent 的评估策略

- **编码 Agent**：最好测。主要使用“代码分析”与“单元测试（跑测验证）”这种确定性打分器。
- **对话 Agent（客服/销售）**：需要评估多维指标（任务是否解决、对话轮数、语气态度）。通常需要**引入另一个 LLM 来扮演挑剔的用户**以模拟多轮交互。
- **研究 Agent**：最主观。没有绝对正确的答案。需组合多个模型打分器来分别评估其：信息来源的真实性、内容的覆盖面、来源的权威性。
- **GUI / 电脑操作 Agent**：不仅要看它怎么点鼠标，关键要通过沙盒环境检测**最终的底层系统状态**（例如提取页面后端数据验证状态，而不仅仅看截图）。

### 必懂的非确定性指标

由于大模型的输出不稳定，跑一次通过不代表永远通过：

- **pass@k**：试 $k$ 次只要成功 $1$ 次就算过。用于评估模型的绝对上限（“它到底能不能解出这题”）。
- **pass^k**：连续试 $k$ 次必须**次次都成功**。用于评估面向用户的商用产品的稳定性（“它到底可不可靠”）。

### 从 0 到 1 构建 Evals 的关键建议

1. **尽早开始，不要等完美**：别一开始就想着攒几百个测试。从你平时靠手工测试的 20~50 个真实痛点案例开始即可。
2. **构建“平衡”的测试集**：不仅要测“触发”的情况，也要测“不触发”的情况。比如搜索 Agent，既要测需要联网搜的问题，也要测直接回答即可（不该联网）的问题，防止模型过度使用工具。
3. **隔离环境状态**：每一次测试都必须是一个干净的沙盒，防止上一次测试遗留的缓存文件或 Git 历史“帮”模型作弊。
4. **给结果打分，而不是给过程打分**：尽量不要死板地限制模型必须按“特定顺序”调用工具，只要最终结果（Outcome）是对的，允许 AI 发挥创造力。
5. **一定要人工阅读轨迹 (Read the transcripts)**：不要迷信冷冰冰的分数。当测试失败时，一定要去读运行日志，看看是模型真笨，还是你的打分器太死板误判了有效的解决方案。
