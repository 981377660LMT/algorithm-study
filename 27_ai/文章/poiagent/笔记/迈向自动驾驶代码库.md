# [迈向自动驾驶代码库](https://cursor.com/cn/blog/self-driving-codebases)

2026.2.5

以下是对 Cursor 官方博客文章《迈向自动驾驶代码库》的精简讲解：

### 核心实验与突破

作为对上一篇“扩展自主编码能力”的延续，Cursor 团队部署了**数千个 Agent** 连续运行一周，使用 Rust 从零构建一个完整的 Web 浏览器。项目峰值吞吐量达到**每小时 1000 次代码提交 (Commit)**。团队在此过程中摸索出了管理海量 Agent 的终极架构。

### 架构的四步演进

1. **初代“自我协同”（失败）**：让所有 Agent 平级，通过共享状态文件和“锁”来协作。导致严重拥堵（大量时间在等锁），且 Agent 会互相甩锅，逃避复杂任务。
2. **“按角色编排”（遇颈）**：引入流水线（规划器 -> 执行器 -> 工作者 -> 裁判）。解决了权责不清，但由于“过度前置规划”，导致应对新问题时系统非常僵化。
3. **“持续执行器”（崩溃）**：将规划和执行合一，动态下发任务。但这个“超级节点”因为要同时兼顾规划、探索、审查和合并，最终不堪重负出现随机休眠和停工。
4. **最终设计：“递归规划”与“闭门造车”（成功）**：
   - **根规划器 (Root Planner)**：负责总体目标，只拆解、下发任务，绝对不写代码。
   - **子规划器 (Sub-planners)**：在遇到复杂任务时递归生成，独立承包某一小块领域。
   - **执行者 (Workers)**：只负责低头干活。对系统全局一无所知，不在乎其他人在干嘛，写完代码带着“交接说明”提交收工。
   - **成果**：这套类似人类软件团队层级分包的架构，实现了最强的稳定性和扩展性。

### 关键的“反直觉”经验（Trade-offs）

1. **为了吞吐量，放弃 100% 的单次正确率**：如果要求每次 Commit 必须绝对正确，系统会因为微小错误或拼写错误陷入大面积停工。**允许少量错误、相信系统后续会自我修复**，是实现每小时 1000 次提交的关键。
2. **砍掉“代码集成审查者 (Integrator)”**：原本设立审查者是为了防冲突和把控质量，结果却成了系统的最大物理瓶颈（“繁文缛节”）。不如接受 Agent 偶尔同时修改同一个文件的“乱流”，让它们后续自然收敛。
3. **硬盘 I/O 成了最大瓶颈**：随着几百个 Agent 同时在单一代码库上执行编译，极大的读写量让系统的物理限制从 CPU/内存 转移到了磁盘。

### Prompt 提示词工程的进阶发现

- **为了防止“失忆”与“偏离”**：不要让 Agent 无限向下追加笔记，而是强制它们定期**重新从头写**草稿盘（Scratchpad），以此保持上下文的“新鲜度”。
- **约束比指令更有效**：说 `No TODOs`（不准留 TODO）比说 `remember to finish`（记得完成）更管用。
- **给出具体的量化范围**：让模型 `Generate tasks`（生成任务），它出于求稳只会生成极少数；如果说 `Generate 20-100 tasks`（生成 20 到 100 个），它就能展现出宏大的规划野心。
