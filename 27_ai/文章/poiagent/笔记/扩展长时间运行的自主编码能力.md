# [扩展长时间运行的自主编码能力](https://cursor.com/cn/blog/scaling-agents)

2026.1.14

**其实还是 subagent**

以下是 Cursor 官方博客文章《扩展长时间运行的自主编码能力》的精简讲解：

### 核心实验与目标

Cursor 团队尝试让**数百个编码 Agent 持续运行数周且完全自主工作**，以测试多智能体并行在人类通常需要耗时数月的大型复杂项目上能走多远。

### 架构演进路线

1. **单 Agent 的局限**：在专注的小任务上表现好，但面对复杂大型项目过于缓慢。
2. **初代协同架构（失败）**：
   - **方法**：采用扁平化对等结构，Agent 之间通过共享文件和“锁机制（Lock）”动态协调。
   - **痛点**：导致严重的相互等待（20 个 Agent 降级成两三个的吞吐量）、死锁、系统脆弱。即使换成“乐观并发控制”，Agent 仍会变得**极度规避风险**，只做安全的小修小补，无人敢啃硬骨头。
3. **分层流水线架构（成功）**：
   抛弃让单个 Agent 兼顾所有的思路，拆分成职能闭环：
   - **规划者 (Planners)**：只负责探索代码库并进行任务拆解和分发（可递归派生子规划者）。
   - **执行者 (Workers)**：领取任务并**闷头执行到底**。不看全局，也不与其他 Worker 协调，只负责写代码和提交（自己解决代码冲突）。
   - **评审者 (Reviewer)**：每个周期结束时判断进展，并确保下一轮迭代从干净的上下文重新开始。

### 震撼成果

在这个架构下，Agent 消耗了数万亿 Token，取得了真实且巨大的进展：

- **从零构建浏览器**：运行近一周，在 1000 多个文件中写出超 100 万行代码。
- **框架级重构**：历时 3 周，在 Cursor 真实代码库中将 Solid 迁移至 React，代码增删达 45 万行。
- **重写底层内核**：把其视频渲染底层用 Rust 重写，实现了 25 倍加速及新动效，并即将上线生产环境。

### 关键经验与启发

1. **模型分工“因材施用”**：
   - **GPT-5.2 系列**在长时间自主任务中远胜于其他模型，展现出极强的专注度、指令遵循和规划能力，是最好的“规划者”。
   - 相反，**Opus 4.5** 倾向于走捷径和偷懒，总想尽快结束任务把控制权交还给用户。
2. **做减法比做加法更重要**：曾尝试设立专职的“合并集成 (Integrator)”Agent 处理冲突，结果反而变成系统瓶颈。最后发现执行者自行解决冲突效率更高。
3. **适度的结构化**：过度规划会让系统极其脆弱，完全自由又会重复劳动和跑偏。寻找分布式调度的中间地带最佳。
4. **提示词依然是王道**：在长周期的多智能体系统中，Prompt 依旧是防止模型异常行为和保持专注的最核心杠杆。

### 总结

要想通过堆砌成百上千个 Agent 来实现大型代码工程的突破，最好的办法**不是建立复杂的网状通信，而是采用简单的黑盒分层（规划与执行分离）**。尽管目前系统仍需定期清空上下文重启以对抗“目标漂移”和“隧道视野”，但**“靠堆 Agent 来解决几个月人工开发量”的可行性已被证实。**
