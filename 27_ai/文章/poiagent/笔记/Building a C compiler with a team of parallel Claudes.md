# [Building a C compiler with a team of parallel Claudes](https://www.anthropic.com/engineering/building-c-compiler)

2026.2.5

以下是对 Anthropic 文章《Building a C compiler with a team of parallel Claudes（用并行 Claude 团队构建 C 编译器）》的精简讲解：

### 核心实验

Anthropic 的研究员采用了一种名为 **“智能体团队 (Agent Teams)”** 的新思路：让 16 个 Claude Opus 4.6 实例在没有任何人类干预的情况下，并行工作了两周。

- **目标**：从零开始使用 Rust 编写一个能够编译 Linux 内核的 C 编译器。
- **结果**：花费了约 2 万美元 API 成本（消耗 20 亿输入 Token，1.4 亿输出 Token），经过近 2000 个会话，最终产出了一个 **10 万行代码**的编译器。该编译器成功在 x86、ARM 和 RISC-V 架构上编译了 Linux 6.9、QEMU、FFmpeg 等大项目，甚至能编译并运行经典游戏《毁灭战士 (Doom)》。

### 系统架构与实现方式

这并非一个中心化调度的系统，而是一个基于最基本工具的去中心化协作循环：

1. **死循环运行**：把 Claude 扔进一个简单的 Bash `while true` 循环中，一旦完成一个任务，立即领下一个，直到系统被强制停止。
2. **基于 Git 的并行防撞锁**：没有复杂的通信中枢。16 个沙箱容器挂载同一个 Git 仓库。Agent 想做某个任务时，会创建对应的 `txt` 文件作为“锁”。如果发生 Git 冲突，Agent 足够聪明去自动解决合并冲突或换个任务。
3. **角色专业化**：当基础可用后，让不同的 Agent 担任不同的角色。比如：一个专职重构除重、一个专职优化编译器性能、一个以 Rust 专家的身份审查代码质量、一个写文档。

### 让 Agent 独立工作的三个关键发现

因为人不在场，如何让 Agent 不迷失方向至关重要：

1. **极高质量的测试环境**：Agent 只能通过报错来认路。必须编写精准的自动化测试和持续集成 (CI) 管道，防止 AI 在加新功能时弄坏老代码（回归错误）。
2. **站在 Claude 的角度设计反馈（防瞎防聋）**：
   - **防止上下文污染**：测试脚本绝不能打印几千行无用日志。出错只输出几行提要，并写到特定文件里让 Agent 用 `grep` 查。
   - **时间盲区**：AI 没时间概念，可能会花满整个测试去跑无关用例。因此加入了 `--fast` 随机抽样跑 1% 测试的机制来快速验证猜想。
3. **消除大任务瓶颈（引入 Oracle）**：在试图编译庞大的 Linux 内核时，所有的 Agent 都卡在了同一个 Bug 上，互相覆盖代码。作者通过引入 **GCC (标准编译器) 作为“神谕”**解决了这个问题：将大部分内核文件用 GCC 编译，只抽一小部分给 Claude 编译。这样 Agent 就可以分散开，各自去修复少数几个文件中的不同 Bug。

### 局限性与思考

尽管成果震撼，但此次实验也摸到了当前大模型（Opus 4.6）的极限天花板：

- 它没能独立写出 x86 16位真实模式下微巧（小于32kb）的编译器，最后不得已部分调用了 GCC “作弊”。
- 还没有自己的汇编器和链接器，且生成的代码运行效率不如不开优化的 GCC。
- 代码的 Rust “风味”一般，和顶级人类程序员写的有差距。

**总结**：这标志着 AI 从“人类主导的结对编程助手”，向着“全自动长程研发工程团队”迈出了重要一步。但同时也带来了隐患——如果人类完全不去肉眼审计 AI 写出的海量代码直接部署，将会带来真实且巨大的安全风险。
