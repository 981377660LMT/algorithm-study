# [为本地代理实现安全沙箱](https://cursor.com/cn/blog/agent-sandboxing)

2026.2.18

以下是对 Cursor 官方博客文章《为本地代理实现安全沙箱》的精简讲解：

### 核心痛点与解决方案

- **痛点**：对于 AI 代理执行的终端命令，自动批准会带来极高的安全风险（可能删库或泄密），而手动逐条审批会导致用户的“审批疲劳”，最终降低效率并削弱了审核的意义。
- **解决方案**：Cursor 为本地 Agent 引入了**操作系统级沙箱**。Agent 可以在受控的“安全区”内自由运行命令，只有在涉及高风险操作（如访问外部网络）时才会弹窗请求人工审批。
- **成果**：成功将 Agent 的**运行中断频率降低了 40%**，极大节省了用户时间。

### 各操作系统的沙箱实现差异

为了在安全与低延迟之间取得平衡，Cursor 针对不同平台采用了不同的底层控制原语：

1. **macOS**：放弃了虚拟机（太慢）和容器，采用了 **Seatbelt (`sandbox-exec`)**。它可以基于用户的 `.cursorignore` 和工作区配置，动态生成细粒度的拦截策略（拒绝特定的读写和系统调用）。
2. **Linux**：直接调用内核级的 **seccomp**（拦截危险系统调用）和 **Landlock**（限制文件系统）。通过建立 Overlay 文件系统，直接让受保护的文件对沙箱内的 Agent “隐身”且不可读写。
3. **Windows**：目前的折中方案是在 **WSL2** 中运行 Linux 沙箱。原生的 Windows 沙箱不够通用，Cursor 目前正与微软联合推进相关底层支持。

### 让大模型学会“戴着镣铐跳舞”

给系统加上沙箱后，必须对大模型（Agent）进行针对性调优，否则它会在遇到权限拦截时陷入死循环：

- **注入边界意识**：在 Shell 工具的 Prompt 描述中，明确告诉模型它正处于沙箱中，且面临文件和网络的访问限制。
- **动态报错引导**：如果 Agent 执行了越权命令被沙箱拦截，系统会向 Agent 返回带有明确解释的报错信息，并引导它“向用户申请权限升级”，而不是让它在原地反复重试相同的错误命令。

### 总结

该机制通过系统底层的权限隔离，解决了多智能体并行工作时的“人工打断”瓶颈。Cursor 的长远目标是开发出直接在约束边界内进行训练的“**沙箱原生 Agent**”，赋予它们在安全隔离区内随意编写和执行复杂程序的绝对自由。
