# 深入分析：将 LLM 视为可编程计算机

## 核心洞见

这篇文章的核心命题是一个元编程隐喻：**提示词是代码，.json/.md 文件是状态**。作者 Mario Zechner 提出将 LLM 不再视为"聊天伙伴"，而是视为一台**极慢、不可靠、但可编程的通用计算机**。

---

## 问题诊断：为何现有方式在大型代码库上失败

作者精准地识别了四个根本性问题：

### 1. 上下文缺失

LLM 无法掌握项目全貌。要么你没提供概览，要么上下文窗口太小。

### 2. 执行流理解能力不足

即使有"推理"能力，LLM 仍然无法追踪复杂的执行流——多进程、IPC、客户端-服务器架构、并发执行。它生成的代码往往**不符合你系统的真实架构**。

### 3. 缺乏"品味"

> LLMs also lack taste.

LLM 生成的是训练数据的"统计均值"。高级工程师追求优雅、精简的方案，而 LLM 倾向于过度工程化的"最佳实践"垃圾代码。

### 4. 上下文退化

超过约 100k token 后，模型就开始丢失关键细节。无论基准测试多好看，**实际使用中大上下文窗口并不可靠**。

---

## 编程模型映射

这是文章最有洞察力的部分——将传统编程概念映射到 LLM 交互：

| 传统编程概念   | LLM 编程对应                      |
| -------------- | --------------------------------- |
| **程序**       | 提示词（自然语言编写）            |
| **输入**       | 预处理信息 + 用户输入 + 工具输出  |
| **状态**       | 序列化到磁盘的 JSON/Markdown 文件 |
| **输出**       | 工具调用产生的代码、diff、文件等  |
| **函数库**     | 工具描述和预定义的命令模板        |
| **控制流**     | 顺序步骤、循环、条件、甚至 goto   |
| **变量赋值**   | "存储这些值以供后续使用"          |
| **字符串插值** | `${targetRuntime}-conventions.md` |

### 关键设计决策：状态持久化

```
上下文 = 临时存储（会被压缩清除）
JSON 文件 = 持久存储（可通过 jq 精确读写）
Markdown = 非结构化持久存储
```

**核心好处：可以用全新的上下文从任意断点恢复，完全绕开上下文压缩问题。**

## 实战案例分析：Spine 运行时移植

### 问题规模

- 8 种语言的运行时 + 10+ 平台集成
- 两个版本间：79 文件变更，4820 行增加，4679 行删除
- 手动移植需要 2-3 周

### 程序架构

整个系统由以下部分组成：

```
port.md (程序主体)
├── 数据结构定义 (porting-plan.json 的 schema)
├── 工具库定义 (vs-claude、jq 命令、编译测试)
├── 初始化流程 (Setup)
└── 主循环 (Port Types)
    ├── 查询下一个待处理类型
    ├── 打开文件供人类审查
    ├── 人工确认
    ├── 读取源文件到上下文
    ├── 执行移植
    ├── 人工确认结果
    ├── 更新持久状态
    └── 更新移植笔记
```

### 五个关键工程决策

#### 1. 预生成 vs LLM 探索

作者选择用 `generate-porting-plan.js` 脚本预先生成移植计划，而不是让 LLM 自行探索代码库。理由：

- **确定性**：相同输入 → 相同计划
- **上下文效率**：节省大量 token 和工具调用轮次
- **速度**：脚本秒级完成 vs LLM 分钟级探索

> 这将一个开放式的探索问题转化为结构化的数据处理。

#### 2. 精确的函数调用而非自由发挥

所有 jq 查询都是**预先编写和测试过的**：

```bash
# 获取下一个待处理类型
jq -r '.portingOrder[] | {file: .javaSourcePath, types: .types[] | select(.portingState == "pending")} | ...' porting-plan.json | head -1
```

这比让 LLM 自己构造查询要可靠得多。

#### 3. 约定文件作为编码风格的"类型系统"

`spine-cpp-conventions.md` 详尽记录了目标代码库的所有约定——命名、内存管理、文件组织、错误处理等。这相当于给 LLM 一套**编码风格的类型约束**，防止它生成不合规的代码。

#### 4. 移植笔记作为增量学习

`porting-notes.md` 记录了移植过程中发现的所有边界情况和特殊模式。关键示例：

> - Java `pose` vs C++ `_applied` 的对应关系
> - `toString()` 方法不移植到 C++
> - Java `Vector2` 参数在 C++ 中展开为独立浮点参数

这解决了一个根本问题：**LLM 没有跨会话记忆，但通过写入文件，知识得以累积**。

#### 5. 人工检查点作为"断言"

程序在多个关键位置要求人工确认：

- 开始移植前：确认是否要移植这个类型
- 移植完成后：确认是否标记为完成
- 循环结束时：确认是否继续下一个

这不仅是安全机制，也是**给人类提供干预机会**——可以跳过、修正或中止。

## 深层洞见总结

### 1. "非确定性计算机"的防御性编程

> Though with our non-deterministic computer, there's always a chance it'll try anyway.

对不确定性的处理策略：

- 明确说明**不要做什么**（如不要编译单个 TypeScript 文件）
- 状态持久化到磁盘而非依赖上下文
- 预定义精确的工具调用而非让 LLM 自由发挥

### 2. 上下文是最宝贵的资源

整个程序设计的核心原则是**上下文管理**：

- 只读入当前任务需要的文件
- 用 jq 精确查询 JSON 而非载入整个文件
- 每完成一个类型就可以开始新会话

### 3. 人机协作而非完全自动化

作者没有试图让 LLM 完全自主工作，而是设计了一个**人在环中（human-in-the-loop）**的工作流。LLM 处理机械性的翻译工作，人类处理需要判断力的决策。

### 4. 从"对话"到"工程"的范式转移

> Think "inputs, state, outputs" instead of "chat with the AI"

这是全文最重要的思维转变。作为程序员，我们习惯用精确的编程语言编码规范。与 LLM 交互时，自然语言的模糊性让我们忘记了可以应用同样的结构化思维。

### 5. 实际效果

> 之前需要 2-3 周的工作，现在只需 2-3 天。

约 **5-7 倍的效率提升**，而且关键在于：这个提升来自**工作流设计**，而非模型能力的提升。

---

## 方法论的局限性

文章也诚实地承认了局限：

1. LLM 仍无法处理复杂类型层次结构（如 Java 泛型 → C++ 模板）
2. 非确定性意味着 LLM 可能偶尔不遵循指令
3. 子代理编排缺乏可观测性和通信通道
4. 不是万能方案，不适用于所有问题

---

## 对实践的启示

如果你在大型代码库上使用 AI 编码工具，这篇文章提供了一套可操作的框架：

1. **把你的手动工作流编码为结构化提示词**
2. **用脚本预生成确定性数据，不要让 LLM 探索**
3. **用 JSON 文件管理状态，确保可恢复**
4. **预写精确的工具调用命令，不要让 LLM 即兴发挥**
5. **设置人工检查点，保持控制权**
6. **累积领域知识到文件中，跨会话复用**
