# [你可能根本不需要 MCP](https://mariozechner.at/posts/2025-11-02-what-if-you-dont-need-mcp/)

## 文章概览

这篇文章由 Mario Zechner 撰写，核心论点是：**在很多场景下，你不需要 MCP（Model Context Protocol）服务器，简单的 Bash 脚本 + 代码组合更高效、更灵活、更可组合。**

- MCP 不具有可组合性，而 Bash 和代码是可组合的：Agent 可以写一个 bash 脚本把多个操作串起来，结果存到文件里，后续用代码处理——数据不必经过 Agent 的上下文窗口。这在处理大量数据时是决定性的优势。
- 工具本身可以由 Agent 生成。MCP 服务器做不到这一点。
- MCP 的流行部分源于对"标准化"的迷信。标准化在多 Agent、多提供商互操作时有价值，但对于单个开发者的日常使用，Unix 哲学（小工具 + 管道组合）更加实用。

## 核心洞见

### 1. MCP 服务器的三大问题

| 问题           | 具体表现                                                                                                                                                         |
| -------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **上下文膨胀** | Playwright MCP 有 21 个工具消耗 13.7k tokens（占 Claude 上下文的 6.8%）；Chrome DevTools MCP 有 26 个工具消耗 18k tokens（9.0%）。工具越多，Agent 越容易"迷路"。 |
| **不可扩展**   | 想加功能？你得理解整个 MCP 代码库，再让 Agent 也理解它。                                                                                                         |
| **不可组合**   | MCP 的返回结果必须经过 Agent 上下文才能持久化或与其他结果结合，无法像 Unix 管道那样自由串联。                                                                    |

### 2. "拥抱 Bash 和代码" 的哲学

作者的核心洞见可以概括为一句话：

> **模型已经知道如何写代码和用 Bash，利用它们已有的知识比喂给它们一堆工具描述要高效得多。**

他只用了 4 个极简的 Node.js 脚本（start、nav、eval、screenshot），加一个 **225 tokens** 的 README，就覆盖了浏览器自动化的核心需求。对比 MCP 的 13,000-18,000 tokens，这是 **60-80 倍的效率提升**。

### 3. 最小工具集设计原则

作者的工具设计体现了极致的"最小充分"原则：

- **start.js** — 启动浏览器（可选加载用户 profile）
- **nav.js** — 导航到 URL（当前 tab 或新 tab）
- **eval.js** — 在页面上下文执行 JS（这是杀手锏——Agent 天生会写 DOM 代码）
- **screenshot.js** — 截图并返回文件路径

关键设计决策：`eval.js` 让代码在**页面上下文**中运行，而非 Puppeteer 上下文。这意味着 Agent 不需要学习 Puppeteer API，只需要用它已经精通的 DOM API。

### 4. 可组合性是真正的杀手级优势

这是文章中最深刻的洞见。MCP 的架构是：

```
工具调用 → 结果回到 Agent 上下文 → Agent 决定下一步
```

而 Bash 脚本的架构是：

```
工具调用 → 结果可以直接写文件/管道/链式处理 → 绕过 Agent 上下文
```

Agent 可以写一个 bash 脚本把多个操作串起来，结果存到文件里，后续用代码处理——**数据不必经过 Agent 的上下文窗口**。这在处理大量数据时是决定性的优势。

### 5. "渐进增强"的工具扩展模式

作者展示了两个即兴添加工具的例子：

- **pick.js**（元素选择器）：让用户直接在浏览器中点击 DOM 元素，Agent 获得元素信息。这是**人机协作**的典范——人擅长视觉定位，Agent 擅长代码生成。
- **cookies.js**：需要 HTTP-only cookies 时，"不到一分钟"就让 Claude 生成了这个工具。

这揭示了一个重要模式：**工具本身可以由 Agent 生成**。MCP 服务器做不到这一点。

### 6. 跨 Agent 复用方案

作者的方案极其朴素但有效：

```bash
alias cl="PATH=$PATH:/Users/badlogic/agent-tools/browser-tools:<other-tool-dirs> && claude --dangerously-skip-permissions"
```

- 工具脚本加前缀避免命名冲突（如 `browser-tools-start.js`）
- 通过 `PATH` 环境变量使脚本全局可用
- 用 `/add-dir` 和 `@README.md` 按需引入工具文档

## 更深层的启示

### 对 Agent 工具设计的启示

1. **利用模型已有知识，而非替代它**。模型知道 DOM API、知道 Bash、知道文件操作——给它一个薄薄的"桥接层"就够了，不要给它一个厚重的抽象层。

2. **工具描述的 token 成本是隐性税**。每次对话都要为所有注册的 MCP 工具付出 token 代价，即使这次根本不用它们。而 README 可以按需引入。

3. **代码即工具接口**。与其定义复杂的 JSON Schema 来描述工具参数，不如让 Agent 直接写代码调用。代码的表达力远超参数列表。

### 对 MCP 生态的批评

文章隐含了一个尖锐的批评：**MCP 的流行部分源于对"标准化"的迷信**。标准化在多 Agent、多提供商互操作时有价值，但对于单个开发者的日常使用，Unix 哲学（小工具 + 管道组合）更加实用。

### 适用边界

作者也隐含了这种方法的适用边界：

- ✅ 技术用户自己使用
- ✅ 单 Agent 场景
- ✅ 需要高度定制的场景
- ❌ 非技术用户（MCP 提供了更好的开箱即用体验）
- ❌ 需要标准化 API 的多 Agent 协作场景
- ❌ 需要跨语言/跨平台统一接口的场景

---

## 一句话总结

**最好的 Agent 工具不是功能最全的，而是最薄的——薄到几乎只是 Agent 已有能力的一个"出口"。** MCP 是锦上添花，Bash + 代码才是基本功。
