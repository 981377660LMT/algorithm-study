# [MCP vs CLI: Benchmarking Tools for Coding Agents](https://mariozechner.at/posts/2025-08-15-mcp-vs-cli/)

## 文章概述

这是 Mario Zechner（开发者、教练、演讲者）于 2025 年 8 月 15 日发表的一篇实证研究文章，核心问题是：**对于 AI 编程代理（coding agents），MCP 服务器和 CLI 工具哪种方式更好？**

MCP 不是银弹，在大多数情况下 CLI 工具是更好的选择，尤其当模型已经"认识"这些工具时。MCP 的价值在于填补 CLI 无法覆盖的缝隙，而非替代 CLI。

---

## 一、研究背景与动机

### MCP 的现状问题

MCP（Model Context Protocol）服务器当下非常火热，但实践者普遍持怀疑态度，原因是：

1. **很多 MCP 只是 CLI 的薄包装**——就像"很多会议其实可以是邮件"一样，很多 MCP 完全可以用 CLI 调用替代
2. **上下文污染**——MCP 输出大量不必要内容、暴露过多工具，会**毒害上下文窗口**，降低代理性能
3. **具体案例**：
   - **GitHub MCP Server**：重新实现了 GitHub CLI 已有的功能，效果反而更差
   - **Context7**：提供代码片段帮助用 LLM 使用库，但作者发现自己库的片段"毫无用处"。更好的做法是让 Claude 克隆依赖仓库，用子代理分析并生成简明用户手册

### MCP 仍有价值的场景

- 没有易用的 CLI 工具时
- CLI 工具输出过于冗长时
- LLM 客户端没有内置 shell 工具时
- **有状态工具**更适合 MCP（MCP 服务器天然有状态）

### 研究目标

量化 MCP 与 CLI 的差异，具体评估：

- **哪个成本更高**
- **哪个耗时更多**
- **哪种成功率更高**
- **模型对标准工具的内在知识 vs 上下文学习全新工具**，哪个更有效

## 二、实验设计：terminalcp 工具

### 工具本身

作者构建了 **terminalcp**——一个 tmux 的轻量级替代品，具有 MCP 服务器和 CLI 两种接口，让 AI 代理可以控制终端应用（调试器、REPL、vim、htop 等），类似 Playwright 控制浏览器。

**架构设计的关键洞见**：

| 组件              | 职责                                                             |
| ----------------- | ---------------------------------------------------------------- |
| `TerminalManager` | 核心：用 `node-pty` 生成伪终端进程，用 `xterm.js` 渲染 ANSI 序列 |
| `TerminalServer`  | 持久化守护进程，通过 Unix socket 通信                            |
| `TerminalClient`  | MCP/CLI 与服务器之间的通信桥梁                                   |
| `AttachClient`    | 允许用户实时附加到运行中的会话                                   |

### Token 效率的设计哲学

这是文章中非常重要的一个工程洞见：

1. **纯文本响应而非 JSON**——减少 token 消耗（MCP 版本仍不可避免地被包裹在 JSON-RPC 中）
2. **单一工具暴露**——MCP 服务器只暴露一个 `terminalcp` 工具而非每个命令一个工具。Anthropic 曾在旧版 Claude Code 文档中提到工具数量有上限，否则 LLM 会混乱
3. **选择性输出**——`stdout` 只返回最后 N 行；`stream` 可以只返回上次调用后的增量输出（`since_last`），还可以剥离 ANSI 码

## 三、评估框架设计

### 评估矩阵

```
3 个任务 × 4 个工具 × 10 次重复 = 120 次评估运行
```

**四个工具**：

1. **terminalcp MCP** — 全新工具，通过 MCP 协议
2. **terminalcp CLI** — 全新工具，通过命令行
3. **tmux** — 标准 CLI 工具，模型训练数据中存在
4. **screen** — 标准 CLI 工具，模型训练数据中存在

**三个任务**：

| 任务                  | 复杂度 | 考察点                                                 |
| --------------------- | ------ | ------------------------------------------------------ |
| **LLDB 调试**         | 中     | 启动调试器、发送命令、读取状态、分析崩溃根因           |
| **Python REPL**       | 低-中  | 多步骤交互、多行输入（函数定义）、输出验证             |
| **OpenCode 项目分析** | 高     | TUI 导航、模型切换（`/models` 命令）、滚动捕获完整输出 |

### 评估流程（精妙之处）

```
┌────────────────────┐
│ 1. 运行测试        │  每个组合启动全新 Claude Code 实例（无自定义 MCP/CLAUDE.md）
│    使用 TerminalManager 驱动                                           │
├────────────────────┤
│ 2. 捕获所有输出     │  prompt、终端滚动缓冲区、原始 ANSI 流            │
├────────────────────┤
│ 3. 提取统计数据     │  成功/失败标记、成本、时间、各模型 token 使用     │
├────────────────────┤
│ 4. Claude 作为裁判  │  第一轮：逐个评判每个任务/工具组合               │
│                    │  第二轮：跨工具排名 + 分析 + 洞见 + 建议           │
└────────────────────┘
```

这里有一个**递归性的巧妙设计**：作者用 terminalcp 本身来驱动评估框架中的 Claude Code 实例，如果 Claude 停滞，框架会"催促"它继续。

### 工具定义的公平性

作者刻意让所有工具定义**遵循完全相同的结构**：

- Quick Start（典型工作流）
- Core Commands（核心命令）
- Important Notes（注意事项）

每个定义都包含：特殊键操作、多行输入、滚动缓冲区处理等等。这种**控制变量**的做法确保了比较的公平性。

值得注意的是，`screen` 工具定义中有一个 **v4.0.3 的 hardcopy bug** 需要 `expect` 变通，这是一个天然劣势。

## 四、结果分析（基于展示的部分数据）

虽然文章在此处截断了完整结果，但从展示的 `evaluation-summary.json` 中可以提取关键发现：

### debug-lldb 任务的排名

| 排名 | 工具               | 平均成本 | 平均时间 | 成功率 |
| ---- | ------------------ | -------- | -------- | ------ |
| 1    | **tmux**           | $0.3729  | 1m 28.7s | 100%   |
| 2    | terminalcp CLI     | $0.3865  | 1m 37.2s | 100%   |
| 3    | **terminalcp MCP** | $0.4804  | 1m 22.2s | 100%   |
| 4    | screen             | $0.6003  | 1m 46.2s | 100%   |

### 关键解读

1. **MCP 比 CLI 更贵但更快**：terminalcp MCP 的成本比 CLI 高约 24%（$0.4804 vs $0.3865），但时间却短约 15%（1m 22s vs 1m 37s）。这说明 **MCP 的 JSON-RPC 包装确实增加了 token 消耗**，但减少了代理的"认知负担"，让它更快完成任务。

2. **tmux 是总冠军**：最低成本、第二快速度。这验证了一个重要假设——**模型对标准工具的内在知识（训练数据中见过 tmux）胜过对全新工具的上下文学习**。

3. **screen 表现最差**：最贵、最慢。原因很明确：v4.0.3 的 hardcopy bug 需要 `expect` 变通，增加了复杂度和 token 消耗。多次运行中代理还会不必要地先读源码再调试。

4. **所有工具成功率均为 100%**：说明对于相对标准的调试任务，工具选择不影响成功率，**差异化体现在效率上**。

## 五、核心洞见与深层思考

### 洞见 1：MCP 的真正代价是 token 税

MCP 协议的 JSON-RPC 包装、工具描述（MCP 服务器自动报告的 description 很长）都会消耗额外 token。从数据看，terminalcp MCP 的 `cacheWrite` 明显高于 tmux，说明上下文更大。

### 洞见 2：训练数据中的知识是免费的先验知识

tmux 作为最小化上下文开销的标准工具，性能最好。模型已经"知道"tmux，不需要花费 token 去学习它。这给工具选择带来一个**实用准则**：

> **如果有标准 CLI 工具能做到，就用标准工具，不要用 MCP。**

### 洞见 3：MCP 的价值在于"不可替代性"

文章暗示 MCP 的真正场景是：

- 没有现成 CLI 可用
- **需要有状态的交互**
- 客户端没有 shell 工具
- CLI 输出太冗长需要结构化

### 洞见 4：工具设计原则

从 terminalcp 的设计中可以提炼出面向 LLM 的工具设计原则：

1. **最少工具数量**——一个工具+action 参数 > 多个独立工具
2. **纯文本优先**——减少 JSON 包装的 token 开销
3. **选择性输出**——让代理只获取需要的信息（最后 N 行、增量输出）
4. **幂等且可清理**——每个工具定义包含 cleanup 命令

### 洞见 5：评估方法论的价值

作者的评估框架虽然自称"hacky"，但设计思路非常值得借鉴：

- **控制变量**（统一工具定义结构、全新 Claude Code 实例）
- **10 次重复**处理 LLM 非确定性
- **两级判断**（统计数据 + LLM-as-judge）
- **全部可复现**（保存所有中间产物）

## 六、实际建议

基于这篇文章的分析，我总结的实践准则：

| 场景                 | 建议                                       |
| -------------------- | ------------------------------------------ |
| 有成熟 CLI 工具      | **直接用 CLI**（tmux、git CLI、gh CLI 等） |
| 无 CLI 或 CLI 太繁琐 | 考虑 MCP                                   |
| 需要有状态交互       | MCP 更合适                                 |
| 客户端无 shell       | MCP 是唯一选择                             |
| 追求成本效率         | CLI > MCP                                  |
| 追求执行速度         | MCP 可能因降低认知负担而更快               |
| 工具数量             | **严格控制暴露给 LLM 的工具数量**          |

这篇文章的核心结论很明确：**MCP 不是银弹，在大多数情况下 CLI 工具是更好的选择**，尤其当模型已经"认识"这些工具时。MCP 的价值在于填补 CLI 无法覆盖的缝隙，而非替代 CLI。
