# [Prompts are code, .json.md files are state](https://mariozechner.at/posts/2025-06-02-prompts-are-code/)

这篇文章由 Mario Zechner 撰写，深入探讨了在大型代码库中有效使用大语言模型（LLM）的工程化方法。文章的核心理念是将 LLM 视为一种“缓慢且不可靠的通用计算机”，并通过严谨的工作流对其进行“编程”。

- 这是一种奇特的元编程形式：我们以提示词的形式编写"代码"，这些提示词在 LLM 上执行，生成在真实 CPU 上运行的实际代码。
- 当我们与 LLMs 交互时，自然语言的模糊性让我们忘记了可以应用相同的结构化思维。这个框架弥补了这一差距：想想"输入、状态、输出"而不是"与 AI 聊天"，突然间你就更接近工程化解决方案，而不仅仅是寄希望于最好的结果。
- 关键
  - Follows the program: Reads port.md and executes each step methodically
    遵循程序：读取 port.md 并有条不紊地执行每个步骤
  - Manages state: Uses precise jq queries to read and update porting-plan.json
    管理状态：使用精确的 jq 查询来读取和更新 porting-plan.json
  - Loads context strategically: Reads only necessary files when needed
    战略性地加载上下文：仅在需要时读取必要的文件
  - Maintains resumability: Updates persistent state after each completed type
    保持可恢复性：在每个完成的类型后更新持久状态
  - Implements human checkpoints: Asks for confirmation before proceeding
    实现人工检查点：在继续之前请求确认
- 未来工作
  - 测试和调试：这些工作流的确定性特性为传统软件工程实践开辟了可能性。我们可以测试在`给定特定输入和状态的情况下，我们的程序是否产生预期的输出`。对于调试，我们可以在提示词中添加工具来在关键点写入状态信息，创建执行跟踪，帮助识别工作流在何处偏离预期。
  - 子代理编排：当前的工具（如 Claude Code）生成子代理时缺乏可观测性和通信通道。要扩展这种编程模型，我们需要结构化的方式来编程子代理并监控其执行。`主代理应该为子代理定义显式工作流，而不是临时生成它们，通过与主工作流相同的结构化方法来确保可靠性。`

---

### 1. 核心痛点

- **上下文缺失与退化**：LLM 难以全盘掌握关联紧密的复杂项目架构；在长对话中会遗忘关键细节。
- **缺乏系统设计品味**：LLM 倾向于生成过度设计或仅符合统计平均规律的代码，容易引入难以维护的逻辑。
- **非确定性**：传统的“聊天式”交互缺乏可重复性和可控性。

### 2. 工程化抽象：提示词即代码，文件即状态

作者提出使用传统的计算机科学思维来管理 LLM 代理：

- **程序 (Program)**：写在 Markdown 文件（如 `port.md`）中的系统级提示词。它规定了初始输入、工具调用说明以及带有条件和循环的控制流。
- **状态 (State)**：为避免上下文窗口溢出，将状态持久化到磁盘（如 `json` 或 `md` 文件）。通过 `jq` 等工具精准读写，实现任务的随时中断与无缝恢复。你可以从任何时刻用全新的上下文恢复，完全规避了令人担忧的压缩问题。
  > jq 是一个轻量级且灵活的命令行 JSON 数据处理工具。
- **输入/输出 (I/O)**：通过预先准备的数据（分析脚本的输出）、工具回调（VS Code 插件）以及终端命令来与物理世界交互。

### 3. 实践案例：Spine Runtimes 代码移植

在将 Spine 的 Java 核心库移植到 C++ 的真实任务中，作者构建了以下系统：

1.  **数据初始化**：预先用`脚本扫描`代码变更并生成一份带有待处理队列的 `porting-plan.json`。LLM 不必浪费 `Token` 探索代码库、并且是`确定`的、`速度`更快。
2.  **建立规范**：LLM 会读取 `.md` 格式的编码规范和注意事项，确保生成的 C++ 代码符合项目的固定风格。
3.  **严格的工作流与检查点 (Checkpoints)**：
    - LLM 通过 `jq` 从 JSON 中读取下一个需要移植的类。
    - LLM 通过 VS Code 工具打开对应文件，等待人工确认（`y/n`）。
    - LLM 完整读取源文件并进行局部代码移植。
    - 完成一小部分后，保存状态回 JSON 文件。

### 4. 总结

这种方法将不可靠的自然语言交互转化为**结构化、确定性的数据处理流程**。
**通过将“探索代码”和“管理状态”的负担从 LLM 上卸下并转移到本地文件系统中**，开发者能够安全、高效地在大型项目中利用 AI 完成繁重的机械性重构和代码移植工作。
