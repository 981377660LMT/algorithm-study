## 一、最核心的一个认知

**对 AI 产出的质量管理，应该从"凭感觉"升级到"用数据"——和软件工程对代码质量的管理方式一样。**

skill-creator 本质上是在回答一个问题：**怎么系统化地度量和改进一段自然语言指令的效果？**

这和传统软件工程的演进路径完全一致：

- 代码质量：手动 review → 单元测试 → CI/CD → coverage 报告
- Prompt/Skill 质量：手动试几下 → **自动化 eval + A/B 基线 + 量化指标 + 迭代循环**

**可迁移结论**：任何你在"调 prompt"的场景（RAG 系统、Agent 工具描述、System Prompt），都可以套这个框架。

## 二、四个值得内化的设计模式

### 1. Draft-Test-Evaluate-Improve 闭环（Evaluation-Driven Development）

类比 TDD，但评估对象是自然语言产出物：

| TDD          | Skill-creator                            |
| ------------ | ---------------------------------------- |
| 先写测试用例 | 先写 evals.json（prompt + expectations） |
| 红-绿-重构   | Draft → Test → Evaluate → Improve        |
| 测试覆盖率   | pass_rate + trigger_rate                 |
| 回归测试     | benchmark.json 跨 iteration 对比         |

**关键约束**：退出条件不是"最后一轮"而是"test 集上得分最高的那轮"——防止过拟合。

### 2. 永远带基线的 A/B 测试

每次测试同时跑 `with_skill` 和 `without_skill`，否则你无法区分：

- Skill 真的有帮助？
- 还是 Agent 本来就能搞定，Skill 只增加了 token 开销？

**可迁移**：评估任何 prompt 改进、RAG 检索策略、工具选择策略时，永远保留一个 baseline。delta 比绝对值重要得多。

### 3. 渐进式加载（Progressive Disclosure）的三层信息架构

| 层级 | 内容                             | 始终加载？ | 类比     |
| ---- | -------------------------------- | ---------- | -------- |
| L1   | name + description (~100 tokens) | ✅         | 函数签名 |
| L2   | SKILL.md body (≤500 行)          | 触发时加载 | 函数实现 |
| L3   | scripts/ references/             | 按需读取   | 外部依赖 |

**核心洞察**：L1 是门卫。门卫判错，后面再精彩也没用。所以 skill-creator 专门搞了一整套描述优化循环来打磨 L1。

**可迁移**：设计任何 Agent 工具描述 / API 文档 / 函数注释时，想一下"哪一层是触发决策的入口？那一层的信噪比够不够高？"

### 4. 评估系统的三层分离

| Agent      | 视角               | 类比                        |
| ---------- | ------------------ | --------------------------- |
| Grader     | 单次运行，逐条断言 | 单元测试 assert             |
| Comparator | 单次运行，盲目 A/B | 代码 review（不知道谁写的） |
| Analyzer   | 跨运行聚合，找模式 | 性能分析 / flaky test 检测  |

Grader 最精妙的一步：**评估"评估"本身**——质疑断言质量。"一个弱断言的 PASS 比 FAIL 更危险，因为它制造虚假信心。" 这句话值得贴在墙上。

## 三、描述优化循环的机器学习思维

这是整篇文章技术含量最高的部分，本质上是把 **ML 的训练范式** 应用到了自然语言优化上：

| ML 概念                      | skill-creator 对应                             |
| ---------------------------- | ---------------------------------------------- |
| 训练集/测试集分离            | 60/40 分层采样，改进模型看不到 test 结果       |
| 避免过拟合                   | 最终选 test 得分最高的 iteration，不是最后一轮 |
| 分层采样                     | 正样本（应触发）和负样本（不应触发）按比例切分 |
| Early stopping               | 训练集全过 → 提前退出                          |
| Extended thinking = 更多计算 | 给 10000 tokens 思考预算去分析失败原因         |

**关键 prompt 工程技巧**：明确告诉改进模型"不要把失败查询的关键词硬塞进 description，要泛化出更通用的意图模式"——**防止 description 退化成一个枚举列表**。

## 四、三条写 Skill/Prompt 的实操原则

1. **description 聚焦用户意图，不聚焦实现细节**
   - ❌ "Uses PyMuPDF to extract text"
   - ✅ "Use when user needs to read, fill, merge, or create PDF files"

2. **解释 why，不堆砌 MUST/NEVER**
   - 看到全大写的 ALWAYS/NEVER 就是黄旗——改成讲道理
   - 让模型理解原因后自己做判断，比硬规则鲁棒得多

3. **稍微激进一点**
   - Claude 当前倾向是"宁可不用也不乱用"（undertrigger）
   - 所以 description 要主动列出边缘触发场景

## 五、元认知：自举（Self-hosting）

skill-creator 本身是一个 Skill，能创造和改进 Skill（包括自己）。这和 GCC 用 C 编写、能编译自己是同一个模式。

**更深一层**：当 Agent 能系统化地评估和改进自己的知识模块时，"人写 prompt → AI 执行"的模式开始转向"AI 写 prompt → AI 评估 → AI 改进"。人的角色从作者变成审查者。

## 一、文章的真正主题不是 Skill，是「如何工程化地管理不确定性产出」

表面上这篇文章在讲 skill-creator 的架构。但底层在回答一个更普遍的问题：

> **当产出物的质量不是二值的（对/错），而是连续的、概率性的、依赖上下文的，你怎么做质量管理？**

传统软件的产出是确定性的——同样的输入永远得到同样的输出，所以单元测试就够了。但 LLM 的产出是随机的——同一个 prompt 跑 10 次可能有 7 种不同结果。skill-creator 的所有设计选择都在解决这个根本矛盾：

| 确定性世界的做法    | skill-creator 的适配                                |
| ------------------- | --------------------------------------------------- |
| 跑 1 次测试就下结论 | 每个 query 跑 3 次，用 `mean ± stddev`              |
| 通过/失败二值判定   | `trigger_threshold=0.5`（3 次里触发 2 次才算 pass） |
| 单一指标            | pass_rate + time + tokens 三维度                    |
| 一个评审者          | 三个 Agent 从不同角度评估，减少单点偏差             |

**内化点**：任何涉及 LLM 输出的评估，都应该用**统计语言**（均值、方差、置信度）而不是**逻辑语言**（对/错）来描述。

## 二、五个被忽视的精妙设计细节

### 1. 「趁等待的空隙干活」的并发哲学

文章多次提到这个模式：测试跑起来后，趁 Subagent 执行的时间去写 assertions；启动仪表板后，先让人看着，AI 同步开始评估。这不是小技巧，而是一种**人机协作的流水线设计**——人和 AI 的等待时间被交错填充，总时长接近 `max(人的时间, AI的时间)` 而非两者之和。

### 2. Grader 的 Step 4「提取隐含声明」是防止 Goodhart's Law 的关键

预定义断言天然有覆盖盲区。如果只查预定义断言，系统会退化成"只要满足这几条就算好"——典型的 Goodhart's Law（指标成为目标后就不再是好指标）。Step 4 通过自动发现预定义之外的声明来抵抗这种退化。

**类比**：这就像代码审查不只看"测试是否通过"，还要看"有没有明显的 bad smell 是测试覆盖不到的"。

### 3. 触发检测用流式中断而非完整执行

`run_eval.py` 用 `--include-partial-messages` 做流式检测，一旦发现 Claude 调用了 Skill 工具就立即 return True，**不等它真正执行完**。这把每个 query 的评估时间从「完整执行时间」压缩到「首次工具调用时间」——可能快 10 倍以上。

**洞见**：触发率评估只关心"是否触发"，不关心"执行结果"。这两个问题的评估粒度不同，应该分开测。skill-creator 确实把它们分成了两个独立流程（`run_eval.py` 测触发率，Grader 测执行质量）。

### 4. description 优化的 prompt 里有一条反模式约束

> "DO NOT repeat these — try something structurally different"

这条约束在 prompt engineering 领域极其重要。没有这条约束，模型在迭代过程中会产生**描述漂移**——每轮都在上一轮的基础上微调措辞，越调越相似，最终收敛到一个局部最优。要求"结构性不同"迫使模型跳出局部最优去探索。

**ML 类比**：这相当于训练时增加 exploration rate，防止策略过早收敛。

### 5. `max(1, ...)` 这行代码背后的工程直觉

```python
n_trigger_test = max(1, int(len(trigger) * holdout))
```

如果 eval set 很小（比如只有 2 个正样本），`int(2 * 0.4) = 0`，test 集会为空，整个评估框架就废了。`max(1, ...)` 确保极端情况下系统依然可用。这种"小数据场景下的鲁棒性"贯穿整个设计——`runs_per_query=3` 也是同一个思路：不追求统计显著性（那需要几百次），而是在实用时间内取一个"比跑 1 次好得多"的折中。

## 三、最值得内化的思维模型：「门卫-执行者-审计员」三层架构

文章的渐进式加载（L1/L2/L3）不只是一个 Skill 加载策略，它是一个通用的**信息过滤架构**：

```
L1 门卫层：用最少的信息做最快的决策（要不要进？）
L2 执行层：进来之后给出完整指导（具体怎么做？）
L3 资源层：只在真正需要时才加载重型资源（用什么工具？）
```

这个模式到处都是：

| 领域           | L1 门卫         | L2 执行           | L3 资源             |
| -------------- | --------------- | ----------------- | ------------------- |
| HTTP           | URL + Method    | Request Handler   | Database / File I/O |
| 编译器         | 词法分析        | 语法分析          | 代码生成            |
| 搜索引擎       | 倒排索引匹配    | 排序模型          | 完整网页渲染        |
| 人的认知       | 标题/摘要       | 正文阅读          | 参考文献            |
| **Agent 工具** | **description** | **SKILL.md body** | **scripts/**        |

**核心原则**：L1 的错误无法被 L2/L3 弥补。门卫放错了人，执行层再完美也是白搭；门卫拒绝了对的人，审计层永远不会知道。因此**优化 ROI 最高的永远是 L1 层**。

skill-creator 用 5200 行代码里的很大一部分（`run_loop.py` + `run_eval.py` + `improve_description.py`）专门优化 L1，正是这个原则的体现。

## 四、「评估评估」的元认知递归

Grader 的 Step 6（评估断言本身的质量）是整个系统最深刻的设计。它引入了一个**元评估层**：

```
Level 0: Agent 执行任务，产出结果
Level 1: Grader 评估结果是否满足断言
Level 2: Grader 评估断言本身是否有意义
```

这解决了一个根本性问题：**100% pass rate 可能是个坏消息**。如果所有断言都太弱（"文件存在"而非"文件内容正确"），通过率再高也没有信息量。

原文那句话值得反复品味：

> "A passing grade on a weak assertion is worse than useless — it creates false confidence."

**可迁移**：在任何评估体系里（代码 review、绩效考核、模型 eval），都应该定期质疑评估标准本身。度量系统的退化速度往往比被度量对象更快。

## 五、从「手艺人」到「工厂主」的范式转换

这篇文章最大的 meta-insight 是：

**当一个工具能系统化地制造和评估其他工具时，制造者的角色就发生了转变——从「亲手做」变成「设计评估标准 + 审查产出」。**

| 阶段             | 人的角色       | AI 的角色                 |
| ---------------- | -------------- | ------------------------- |
| 手写 Skill       | 作者           | 执行者                    |
| skill-creator v1 | 编辑（审稿）   | 起草人                    |
| skill-creator v2 | 评审委员会主席 | 起草 + 自测 + 自评 + 自改 |
| 理论终态         | 只定义评估标准 | 全自动闭环                |

这和软件行业的历史轨迹一致：手写汇编 → 编译器 → IDE → CI/CD → 现在的 AI 写代码。每一步都是把「做」的部分交给机器，人退到更高的抽象层去做「判断」。

**skill-creator 的真正意义**：它不只是一个工具，它是 Anthropic 对"AI 系统如何自我改进"这个问题的一个工程化回答。而且这个回答本身是可递归的（self-hosting），这意味着改进的速度可以加速。

## 六、三个可以立刻用的行动项

1. **给你的任何 prompt/工具描述加 A/B 基线**。下次修改 system prompt 时，不要只跑新版本看效果——同时跑一遍旧版本（或无 prompt），算 delta。
2. **用「门卫层」思维审视你的 API/工具设计**。找到信息过滤链的入口层，确认它的信噪比。如果入口层有问题，优先修入口层，不要在下游补救。
3. **给你的评估断言加元评估**。写完 test case，问自己：「如果这个测试通过了，我真的能确定功能是对的吗？还是这个测试本身就太弱了？」
