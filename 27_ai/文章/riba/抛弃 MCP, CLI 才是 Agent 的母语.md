# [抛弃 MCP, CLI 才是 Agent 的母语](https://blog.riba2534.cn/blog/2026/%E6%8A%9B%E5%BC%83mcp-cli%E6%89%8D%E6%98%AFagent%E7%9A%84%E6%AF%8D%E8%AF%AD/)

这篇文章的论证质量很高——数据驱动、立场明确但不极端、有自我修正。逐层拆解。

## 一、文章的论证结构（值得学习的写法）

整篇文章的骨架是一个**递进式论证**，每层回答一个更深的问题：

| 层级 | 问题                     | 论证方式                                                    | 结论                         |
| ---- | ------------------------ | ----------------------------------------------------------- | ---------------------------- |
| 1    | 风向变了吗？             | 社会证据（CEO/CTO 集体表态 + 工具开发者行动投票）           | 是结构性转变，不是噪音       |
| 2    | 贵不贵？                 | **硬数据**（Playwright 114K→27K tokens, Anthropic 150K→2K） | 数量级差距，不是优化能解决的 |
| 3    | 为什么 LLM 用 CLI 更顺？ | **第一性原理**（训练数据分布决定能力分布）                  | CLI 是母语，MCP 是外语       |
| 4    | 安全吗？                 | 攻击向量分析（Tool Poisoning / Rug Pull / Shadowing）       | MCP 安全模型有结构性缺陷     |
| 5    | 所以 MCP 该死？          | **自我修正**——区分"连接"和"使用"两个问题                    | 不该死，但该换位置           |

**写法上最值得学的**：先把观点推到极端（MCP 各方面都不如 CLI），然后在第五层主动回撤，给出更精确的定位。这比一开始就说"各有优劣"有力得多——先建立势能，再精确落点。

## 二、三个核心洞见

### 洞见 1：训练数据分布决定能力分布

> "你没法发明一种新语言，然后指望它比模型的母语说得更好。"

这是全文最有穿透力的观点。它揭示了一个**通用原则**：

- LLM 的能力边界本质上是训练数据的投影
- CLI 命令在训练语料中出现了数十亿次，MCP 的 JSON-RPC schema 是 2024 年才发明的
- 这不是工程问题（可以修），是**统计问题**（训练数据的分布无法快速改变）

**可迁移推论**：设计任何 Agent 工具接口时，优先选择训练数据中已大量存在的交互模式。不要发明新协议——尽量复用模型已经"见过"的模式。

同理：写 prompt 时用常见的编程术语和惯用表达，比用生造的 DSL 效果好。

### 洞见 2：区分"连接"和"使用"

|            | CLI                      | MCP                            |
| ---------- | ------------------------ | ------------------------------ |
| 解决的问题 | **使用**工具（执行操作） | **连接**工具（发现+认证+路由） |
| 前提条件   | 工具已安装在本地         | 工具可以在远端                 |
| 最佳场景   | 本地操作、已有工具       | 远程服务、多租户、沙箱         |

这个区分精准地解释了为什么争论双方都有道理——他们在讨论不同层的问题。

**可迁移**：遇到"A 好还是 B 好"的争论时，先问"它们是不是在解决同一个问题？"很多时候争论的本质是层级错位。

### 洞见 3：Smart Money 的集体移动比个体判断更值得关注

文章用了一个投资分析框架来判断技术趋势：

1. **信号聚合**：不是一两个人吐槽，而是不同背景的人在同一时期指向同一方向
2. **行动投票**：不是嘴上说 CLI 好，而是工具开发者（Playwright、Anthropic）自己在用行动转向
3. **内部人士的行为最可信**：Playwright MCP Server 的开发者自己写"CLI 更好"——这比任何外部评测都有说服力

> "MCP 的亲爹绕开了 MCP。这个信号够强了。"

## 三、数据论证的方法论

文章最强的地方是**用数据说话的纪律**。每个关键判断都有量化支撑：

| 判断               | 数据                                                                                    |
| ------------------ | --------------------------------------------------------------------------------------- |
| CLI 更省 token     | Playwright: 114K → 27K（4x），Anthropic Code Exec: 150K → 2K（75x）                     |
| Schema 开销大      | GitHub MCP: 55K tokens 初始化；Playwright MCP 26 tools = 3600 tokens vs CLI = 68 tokens |
| CLI 更可靠         | 任务完成率 100% vs 72%                                                                  |
| CLI 是主流使用方式 | 自己统计：Bash 78%, MCP 12%, 文件读写 10%                                               |

**方法论价值**：当你要论证"A 优于 B"时，不要只说"我觉得 A 更好"——找到可比较的场景，量化差距，让数据说话。特别注意**同一团队、同一产品**的前后对比（Playwright MCP → CLI），这消除了"不同团队/不同标准"的干扰变量。

## 四、安全分析的结构

Simon Willison 的"致命三连"是一个很好的安全分析框架：

| 攻击                           | 本质                                     | 为什么 CLI 天然更安全                       |
| ------------------------------ | ---------------------------------------- | ------------------------------------------- |
| **Tool Poisoning**（工具投毒） | description 中嵌入对用户不可见的恶意指令 | CLI 没有隐藏的 description 层；命令就是命令 |
| **Rug Pull**（地毯抽拉）       | MCP Server 随时更新行为，Agent 继续信任  | CLI 工具版本固定，更新需要显式操作          |
| **Shadowing**（影子工具）      | 恶意 Server 注册同名 tool 覆盖合法工具   | Unix 进程隔离模型，命令在独立进程中运行     |

根本差异：**MCP 的信任是隐式的、无条件的**（所有 Server 共享上下文和权限）；**CLI 的信任是显式的、有边界的**（进程隔离、文件权限、用户组）。

50 年攻防锤炼出的安全模型 vs 2 年的新协议。时间是安全的最终裁判。

## 五、一句话总结 + 可行动的外带

> **CLI 赢在三点：token 经济学（便宜 10-75x）、训练数据兼容性（母语 vs 外语）、安全模型成熟度（50 年 vs 2 年）。MCP 的真正价值在"连接"不在"使用"——做 Gateway，不做执行层。**

**外带行动项**：

1. 给 Agent 加能力时，优先找 CLI 工具，MCP 作为 fallback
2. 写工具描述 / prompt 时，用训练数据中大量存在的模式，不发明新格式
3. 评估任何"新协议"时，问三个问题：比现有方案贵多少？模型见过多少？安全模型经过多久的验证？
