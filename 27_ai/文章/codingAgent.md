- AI coding 痛点与需求上下文
  - 痛点：当前 AI 编程工具如 Claude Codex、Copilot 等功能强大，但在实际开发中，无法让 AI 直接完成需求开发。
    主要存在两个痛点
    一是作为 “AI 的文档搬运工”，面临**上下文缺失问题**，如 PRD 质量不佳、后端技术文档和接口协议需手动提供等；
    二是作为 “AI 的指令拆解员”，难以在一个会话中让 AI 完成整个需求，还存在代码生成后无法运行、**缺少验证**等问题。
  - 前端需求上下文：对于前端需求开发，需要的上下文包括(5个) **PRD、Figma 设计稿、后端技术文档、仓库知识和内部物料知识**。PRD 包含需求背景、功能点和边界情况；Figma 设计稿除页面样式和交互外，还可能补充需求功能点；后端技术文档明确接口定义、字段变化和前后端交互边界；仓库知识影响 AI 生成符合规范的代码；内部物料知识对于内网开发至关重要。
- 解决 AI coding 问题的策略
  - 三大支柱(对的上下文、合适的运行环境、可靠的验证闭环)
    - 构建强大的 agent runtime，包括使用 Claude code 等强大的 coding agent 以及云端沙箱和各种 skill 工具的调用和验证能力
    - 二是精准提供上下文，接入 PRD、仓库等上下文，并根据开发阶段和场景召回精准信息
    - 三是分阶段推进和校验 AI 研发流程，将**需求解析、技术方案、代码修改和验证反馈**环节串联起来，形成闭环。
  - 知识底座与研发主线：将解决方案分为知识底座和研发主线两个模块。知识底座包含前端仓库知识和物料相关知识，为 AI 提供全局的静态上下文；研发主线则通过动态注入 PRD、Figma 和接口文档等上下文信息，支撑整个研发流程。
- 知识底座核心功能点
  - 仓库知识管理：为 agent 提供仓库的 “地图”，让其知道相关知识的位置。通过仓库管理、项目管理和项目内知识管理，为 AI 生码提供重要输入，如 agent MD 和页面相关知识。
  - 物料知识处理：在 AI coding 中，AI 对内部物料的封装逻辑理解困难，容易出现使用错误。为解决这一问题，将物料进行结构化处理，并按场景召回合适的用法示例。通过解析物料，生成包含物料名称、描述、指导说明、组件信息和使用示例的结构化数据，在实际研发中，根据需求场景召回相关文档内容，避免上下文过大和误召回问题。
- 研发主线流程详解
  - 需求解析阶段：对原始输入的 PRD 进行解析，提取背景、收益、交互说明等信息，转化为结构化的功能点和页面线索，并关联到对应的页面和路径。通过需求拆解和图片关联，结合仓库 wiki 知识，让 agent 判断功能点对应的页面，同时识别新增页面。
  - 技术方案阶段：该阶段主要规划如何实现需求，指导后续的人工拆分、代码修改和预览交付。基于仓库环境，使用云端沙箱和 coding agent 生成技术文档，通过定制 prompt 和 skill 能力，收集上下文信息，制定并审核技术方案。在上下文管理方面，采用 DIFF 形式提供后端接口信息，避免信息过大，提高模型对需求的理解。
  - 代码修改阶段：此阶段主要进行执行和验证工作。由于需求复杂度不同，直接让 agent 一次性完成代码修改可能导致质量下降。因此，将研发任务按 coding agent 会话维度拆分，保证每次对话工作量适中。在代码生成过程中，通过任务之间的上下文传递，总结与技术方案的偏差，为后续任务提供参考。同时，利用仓库的校验和预览能力，为 agent 提供反馈，帮助其修改代码。
- 底层能力技术底座
  - Coding agent 架构：基于开源的 new work new wait code 进行修改，核心结构包括底层的核心引擎层，具备 agent Loop、messagebus、记忆系统和配置能力；中间的工具和能力层，包含内置工具、网络获取、用户交互等功能，支持串联外部 MCP、skill 和 subagent；顶层的交互层，提供 CLI、IDE 插件和无头模式等接口，以及莱特查 Server 连接沙箱。
  - 沙箱能力：沙箱是 agent 在云端运行的重要支撑，负责准备各种环境，实现预览、回收、重启和恢复等功能。主要分为容器调度、容器池管理和容器实例逻辑层三个部分，通过 RPC 服务、消息队列和数据库存储进行串联，实现资源调度、容器启动、环境预热、任务执行和生命周期管理等流程。
