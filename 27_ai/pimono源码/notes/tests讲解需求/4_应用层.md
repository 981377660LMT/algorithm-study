# 应用层（mom / pods / web-ui）代码功能全面讲解

**mom（Slack Bot）**：频道隔离机制、Docker 沙箱执行、两阶段日志系统（log.jsonl + context.jsonl）、事件系统（Cron 定时）、5 个内置工具、自管理 Skills、system prompt 结构

**pods（GPU CLI）**：Pod 生命周期（setup → start → stop）、SSH 封装、GPU 智能分配算法、vLLM 三种版本策略、预定义模型配置与 VRAM 校验

**web-ui（浏览器组件）**：ChatPanel ↔ AgentInterface 分层、9 种 Artifact 类型、iframe 沙箱 Runtime Providers、JS REPL、IndexedDB 持久化存储、工具渲染注册表

> 结合 README.md 与源码，拆解 pi-mono 的三个应用层包。
> 它们都建立在底层库（pi-ai → pi-agent-core → pi-coding-agent / pi-tui）之上，面向不同的使用场景。

## 二、mom（Master Of Mischief）— Slack AI 机器人

### 2.1 项目定位

mom 是一个 **自管理的 Slack 机器人**。它不预装任何业务逻辑，而是通过 LLM + bash 工具**自行**安装依赖、编写脚本、配置凭据。本质是把 pi-coding-agent 的能力搬到了 Slack 上。

### 2.2 核心架构（6 个模块）

```
┌──────────────────────────────────────────────────────┐
│                    main.ts                            │
│  CLI 入口：解析参数 → 创建 SlackBot → 启动事件监听    │
└───────────────┬──────────────────────────────────────┘
                │
┌───────────────▼──────────────────────────────────────┐
│              slack.ts — SlackBot                      │
│  Socket Mode 连接 / 频道消息路由 / 队列化处理         │
└───────────────┬──────────────────────────────────────┘
                │
┌───────────────▼──────────────────────────────────────┐
│              agent.ts — AgentRunner                   │
│  创建 pi-agent-core Agent / 构建 system prompt /     │
│  加载 memory & skills / 执行对话循环                  │
└───────────────┬──────────────────────────────────────┘
                │
┌───────────────▼──────────────────────────────────────┐
│  sandbox.ts          store.ts          events.ts     │
│  Host/Docker 执行器   JSONL 日志存储    Cron 定时触发  │
```

### 2.3 关键机制详解

#### 2.3.1 频道隔离

每个 Slack 频道（包括 DM）有独立的状态：

```typescript
// main.ts — 每个 channelId 独享一个 ChannelState

  running: boolean // 是否正在处理消息
  runner: AgentRunner // 独立的 Agent 实例
  store: ChannelStore // 独立的日志存储
  stopRequested: boolean // 用户发 "stop" 时中断
}
const channelStates = new Map<string, ChannelState>()
```

对应的文件系统布局：

```
./data/                         # 宿主机目录
  ├── MEMORY.md                 # 全局记忆（跨频道共享）
  ├── skills/                   # 全局自制 CLI 工具
  └── C123ABC/                  # 每个频道一个目录
      ├── MEMORY.md             # 频道专属记忆
      ├── log.jsonl             # 完整消息历史（无 tool results）
      ├── context.jsonl         # LLM 上下文（由 SessionManager 管理）
      ├── attachments/          # 用户上传的文件
      ├── scratch/              # mom 的工作目录
      └── skills/               # 频道级自制工具
```

#### 2.3.2 Docker 沙箱（sandbox.ts）

```typescript
// 两种执行环境，通过 Executor 接口屏蔽差异
export type SandboxConfig =
  | { type: 'host' } // 直接在宿主机运行（不推荐）
  | { type: 'docker'; container: string } // Docker 容器内运行（推荐）

// DockerExecutor 实现：
// 所有 bash 命令自动包装为 docker exec <container> sh -c '...'
class DockerExecutor implements Executor {
  async exec(command: string, options?: ExecOptions): Promise<ExecResult> {
    const dockerCmd = `docker exec ${this.container} sh -c ${shellEscape(command)}`
    return new HostExecutor().exec(dockerCmd, options)
  }
  getWorkspacePath(): string {
    return '/workspace'
  } // 容器内统一看到 /workspace
}
```

关键安全设计：

- Docker 模式下 mom 只能访问挂载的 data 目录
- 输出限制 10MB 防内存爆炸
- 支持超时和 AbortSignal 中断
- 进程树级 kill（`killProcessTree`）

#### 2.3.3 消息流转（两阶段日志）

```
用户在 Slack @mom
│
▼ SlackBot.setupEventHandlers()
│ 识别 mention / DM，创建 SlackEvent
│
▼ store.logMessage()
│ 写入 log.jsonl（纯文本历史，可 grep）
│
▼ context.syncLogToSessionManager()
│ 将 log.jsonl 中新消息同步到 SessionManager（LLM 上下文）
│ ↳ 去重逻辑：对比已有 content 避免重复注入
│
▼ agent.run()
│ 构建 system prompt（含 memory、skills、频道/用户列表）
│ 调用 AgentSession.prompt() 触发 LLM 循环
│
▼ LLM 返回工具调用 → 执行 → 返回结果 → LLM 继续...
│
▼ 最终文本通过 slack.postMessage / updateMessage 发送
│ 工具细节通过 respondInThread 放入线程
```

#### 2.3.5 事件系统（events.ts）

mom 支持三种事件触发方式（通过 JSON 文件）：

```typescript
// 立即执行
{ type: "immediate", channelId: "C123", text: "检查服务状态" }

// 一次性定时
{ type: "one-shot", channelId: "C123", text: "发送日报", at: "2024-12-01T09:00:00+08:00" }

// 周期定时（Cron）
{ type: "periodic", channelId: "C123", text: "每日站会提醒",
  schedule: "0 9 * * 1-5", timezone: "Asia/Shanghai" }
```

使用 `croner` 库实现 Cron 调度，用 `fs.watch` 监听 events 目录变更，热加载/取消事件。

#### 2.3.6 工具集（5 个内置工具）

## 三、pods — GPU Pod 管理 CLI

```
               │
┌──────────────▼──────┐  ┌──────────────────────────┐
│  config.ts          │  │  ssh.ts                   │
│  本地 JSON 配置管理  │  │  SSH/SCP 命令封装         │
```

### 3.3 类型系统（types.ts）


### 3.4 Pod 生命周期

#### 3.4.1 Setup（setupPod）

```
pi pods setup dc1 "ssh root@1.2.3.4" --mount "sudo mount ... /mnt/hf-models"
│
├─ 1. 验证 HF_TOKEN 和 PI_API_KEY 环境变量
├─ 2. 测试 SSH 连通性
├─ 3. SCP 上传 pod_setup.sh 脚本
├─ 4. SSH 执行 setup 脚本（安装 vLLM、CUDA 等，2-5 分钟）
├─ 5. nvidia-smi 检测 GPU 配置
└─ 6. 保存到 ~/.pi/pods.json
```

#### 3.4.2 模型部署（startModel）

```
pi start Qwen/Qwen2.5-Coder-32B-Instruct --name qwen
│
├─ 1. 查找 pod 和可用端口（从 8001 递增）
├─ 2. 如果是已知模型：自动查找最佳 GPU 配置
│     └─ 智能 GPU 分配：按使用量排序，选最空闲的
├─ 3. 如果有 --vllm 自定义参数：直接透传
├─ 4. SSH 运行 model_run.sh 脚本（后台 nohup）
├─ 5. 轮询等待模型加载完成（health check）
└─ 6. 更新 pods.json 中的模型记录
```

### 3.5 GPU 智能分配

```typescript
// 轮询选择最空闲的 GPU（round-robin）
const selectGPUs = (pod: Pod, count: number): number[] => {
  // 统计每个 GPU 被多少模型使用
  const gpuUsage = new Map<number, number>()
  for (const model of Object.values(pod.models)) {
    for (const gpuId of model.gpu) {
      gpuUsage.set(gpuId, (gpuUsage.get(gpuId) || 0) + 1)
    }
  }
  // 按使用量排序，取最少的
  return Array.from(gpuUsage.entries())
    .sort((a, b) => a[1] - b[1])
    .map(e => e[0])
    .slice(0, count)
}
```

```typescript
  document.body.appendChild(sandbox)

  const result = await sandbox.execute(sandboxId, code, runtimeProviders, [], signal)
  // 返回 console 输出 + 返回值 + 生成的文件
}
```

特点：

- 沙箱隔离（iframe sandbox）
- 可访问 Artifacts 和附件
- 支持 AbortSignal 取消
- 捕获 console 输出和返回值

## 六、核心设计理念

1. **分层复用**: pi-ai → pi-agent-core → 应用层，每层都可独立使用
2. **自管理**: mom 不预装业务逻辑，让 LLM 自己安装工具、编写脚本
3. **沙箱隔离**: mom 用 Docker，web-ui 用 iframe sandbox，安全执行不可信代码
4. **频道/会话隔离**: 每个对话空间完全独立，互不干扰
5. **持久化记忆**: MEMORY.md（mom）/ IndexedDB（web-ui）让 AI 跨会话记住上下文
6. **OpenAI 兼容**: pods 部署的所有模型都暴露标准 OpenAI API，任何支持 OpenAI 接口的客户端都能直接使用
