### 第一阶段：`packages/ai` — 统一 LLM API 层

这是整个项目的地基，其他所有包都依赖它。建议花 3-4 天精读。

#### 阅读顺序 & 代码分析

**Step 1: 类型系统（最重要，先建立心智模型）**

读 ai/src/types.ts（316 行，必须完整读完）

核心类型关系图：

```
Context { systemPrompt, messages: Message[], tools: Tool[] }
    │
    ├── Message = UserMessage | AssistantMessage | ToolResultMessage
    │       │
    │       ├── UserMessage { role:"user", content: string|(TextContent|ImageContent)[] }
    │       ├── AssistantMessage { role:"assistant", content: (TextContent|ThinkingContent|ToolCall)[], usage, stopReason }
    │       └── ToolResultMessage { role:"toolResult", toolCallId, content, isError }
    │
    ├── Tool { name, description, parameters: TSchema }   ← TypeBox schema，运行时可验证
    │
    └── Model<TApi> { id, name, api, provider, baseUrl, cost, contextWindow, maxTokens, compat? }
```

关键设计决策：

- `Api` vs `Provider` 的区别：`Api` 是 **协议类型**（如 `"anthropic-messages"`、`"openai-completions"`），`Provider` 是 **商家品牌**（如 `"anthropic"`、`"groq"`）。多个 Provider 可以共享同一个 Api（例如 Groq 用 `"openai-completions"` 协议）
- `AssistantMessageEvent` 是一个精心设计的**有限状态机**事件：`start → (text_start → text_delta* → text_end)* → (toolcall_start → toolcall_delta* → toolcall_end)* → done|error`
- `Model<TApi>` 的泛型参数 `TApi` 让 TypeScript 能在编译期关联 Model 和它对应的 Provider

**Step 2: EventStream 异步流**

读 ai/src/utils/event-stream.ts（~90 行）

这是整个项目最核心的基础设施之一。它实现了一个**生产者-消费者模式的异步可迭代对象**：

```typescript
class EventStream<T, R> implements AsyncIterable<T> {
    private queue: T[] = [];                    // 事件缓冲队列
    private waiting: ((value) => void)[] = [];  // 等待中的消费者 resolve 回调

    push(event: T): void {
        // 有人等 → 直接交给等待者；无人等 → 入队
        const waiter = this.waiting.shift();
        if (waiter) waiter({ value: event, done: false });
        else this.queue.push(event);
    }

    async *[Symbol.asyncIterator]() {
        while (true) {
            if (queue有数据) yield queue.shift();
            else if (done) return;
            else yield await new Promise(resolve => this.waiting.push(resolve));
            // ↑ 关键：没数据时挂起，等 push() 唤醒
        }
    }

    result(): Promise<R> { ... }  // 等待 done/error 事件提取最终结果
}
```

`AssistantMessageEventStream` 继承自 `EventStream<AssistantMessageEvent, AssistantMessage>`，在 `done`/`error` 事件时提取 `AssistantMessage` 作为最终结果。

**Step 7: 深入一个 Provider（Anthropic 最典型）**

精读 ai/src/providers/anthropic.ts（881 行，需要分段读）

**核心流程**（`streamAnthropic` 函数，约第 200-420 行）：

```
创建 AssistantMessageEventStream → 异步 IIFE → {
    1. 初始化空 AssistantMessage output
    2. 创建 Anthropic SDK client
    3. 构建请求参数 (buildParams)
    4. 发起流式请求 client.messages.stream()
    5. push({ type:"start", partial: output })
    6. for await (event of anthropicStream) {
        message_start     → 采集 input token 数
        content_block_start → 根据类型(text/thinking/tool_use) 创建 block，push start 事件
        content_block_delta → 增量更新 block，push delta 事件
        content_block_stop  → push end 事件
        message_delta      → 更新 stopReason 和 usage
    }
    7. push({ type:"done", message: output })
    8. stream.end()
}
return stream;  // 立即返回，异步填充
```

亮点设计：

- **"Stealth mode"**（第 72-88 行）：当使用 OAuth token（Claude Pro 订阅）时，自动将 tool 名映射为 Claude Code 的规范命名（`read` → `Read`），伪装成 Claude Code 客户端以避免被厂商识别
- **Fuzzy JSON 解析**（`parseStreamingJson`）：ToolCall 的参数在流式传输中是不完整 JSON，用增量解析器处理
- **Adaptive Thinking**（第 430-470 行）：Opus 4.6/Sonnet 4.6 使用 `effort` 参数，旧模型用 `thinkingBudgetTokens`

读 `streamSimpleAnthropic`（第 465-500 行）—— 看它如何把统一的 `reasoning: "medium"` 映射为 Anthropic 特定参数。

**Step 8: 消息转换器**

### 第二阶段：`packages/agent` — Agent 运行时

在 `ai` 之上添加状态管理和 Tool 执行循环。约 1200 行，建议 2 天。

#### 阅读顺序 & 代码分析

**Step 1: 类型（理解 AgentMessage 的精妙设计）**

读 agent/src/types.ts（195 行）

关键类型：

```typescript
// 1. 可扩展的消息类型（通过 declaration merging）
interface CustomAgentMessages {} // 空接口，外部模块可以扩展
type AgentMessage = Message | CustomAgentMessages[keyof CustomAgentMessages]

// 2. AgentTool 比 Tool 多了 execute 函数
interface AgentTool<TParams, TDetails> extends Tool<TParams> {
  label: string
  execute: (toolCallId, params, signal?, onUpdate?) => Promise<AgentToolResult<TDetails>>
}

// 3. AgentLoopConfig — agent-loop 的全部配置
interface AgentLoopConfig extends SimpleStreamOptions {
  model
  convertToLlm
  transformContext?
  getApiKey?
  getSteeringMessages?
  getFollowUpMessages?
}
```

注意 `AgentMessage` 的设计：coding-agent 后来通过 declaration merging 往 `CustomAgentMessages` 里加了 `bashExecution`、`custom`、`compactionSummary`、`branchSummary` 四种自定义消息。

**Step 2: agent-loop（核心循环，最重要的 300 行代码）**

精读 agent/src/agent-loop.ts（418 行，全部读完）

`runLoop()` 是双层循环：

```
外层 while(true): 检查 followUp 消息
  │
  └── 内层 while(hasMoreToolCalls || pendingMessages): 处理 tool calls 和 steering
        │
        ├── 1. 注入 pending messages (steering/followUp)
        ├── 2. streamAssistantResponse()
        │       ├── transformContext(messages)     // AgentMessage[] → AgentMessage[] (裁剪)
        │       ├── convertToLlm(messages)         // AgentMessage[] → Message[] (过滤自定义类型)
        │       └── streamSimple(model, llmContext) // 调用 LLM
        │
        ├── 3. 检查 toolCalls
        │       └── executeToolCalls()
        │           ├── 逐个执行 tool (validateToolArguments → tool.execute)
        │           ├── 每执行完一个 tool 检查 steering messages
        │           └── 如果有 steering，跳过剩余 tools (skipToolCall)
        │
        └── 4. turn_end，检查下一轮 steering

外层检查 getFollowUpMessages()，有则继续
```

`streamAssistantResponse()` 的事件桥接（约第 220-275 行）是精华：

```
LLM 的 event → 映射为 AgentEvent
    "start"         → message_start
    "text_delta"等  → message_update (携带原始 AssistantMessageEvent)
    "done"/"error"  → message_end
```

注意它如何在 context.messages 中**原地更新** partial message：第 242 行 `context.messages[context.messages.length - 1] = partialMessage`。

**Step 3: Agent 类**

精读 agent/src/agent.ts（560 行，全部读完）

`Agent` 是对 `agentLoop` 的有状态封装：

- `state: AgentState` — 状态存储
- `subscribe(fn)` — 发布-订阅模式
- `steeringQueue` / `followUpQueue` — 两种消息队列

```
prompt("Hello") → _runLoop(msgs)
    │
    ├── 创建 AgentLoopConfig {
    │       getSteeringMessages: () => dequeueSteeringMessages(),
    │       getFollowUpMessages: () => dequeueFollowUpMessages(),
    │   }
    │
    ├── messages ? agentLoop(msgs, context, config) : agentLoopContinue(context, config)
    │
    └── for await (event of stream) {
            switch: 更新内部 state + emit 给 listeners
        }
```

关键设计：**Steering vs FollowUp**

- `steer(msg)` — 在 tool 执行间隙被检查，能**中断**未执行的 tool
- `followUp(msg)` — 只在 agent 完成所有工作后检查，用于排队追加消息

`steeringMode: "one-at-a-time" | "all"` 控制每次取一条还是全部取出。

**Step 6: Session 持久化**

精读 coding-agent/src/core/session-manager.ts（1411 行，分段读）

Session 文件是 **JSONL 格式**（每行一个 JSON），append-only：

```json
{"type":"session","id":"uuid","timestamp":"...","cwd":"/..."}
{"type":"message","id":"1","parentId":null,"message":{"role":"user",...}}
{"type":"message","id":"2","parentId":"1","message":{"role":"assistant",...}}
{"type":"thinking_level_change","id":"3","parentId":"2","thinkingLevel":"medium"}
{"type":"compaction","id":"4","summary":"...","firstKeptEntryId":"3"}
```

核心概念：

- 每个 entry 有 `id` + `parentId` → 形成**链表/树结构**
- `branch` 是从某个 entry 回溯到根的路径
- `buildSessionContext()` 重建 AgentMessage[] 用于恢复会话
- compaction entry 标记"从这里开始是有效消息"

### 核心架构模式总结

| 模式                    | 在哪用                              | 怎么实现                                                 |
| ----------------------- | ----------------------------------- | -------------------------------------------------------- |
| **策略注册**            | api-registry                        | `Map<Api, Provider>` + `registerApiProvider()`           |
| **异步可迭代流**        | EventStream                         | `push()`/`async *[Symbol.asyncIterator]()` 生产者-消费者 |
| **装饰器层叠**          | `stream → streamSimple → agentLoop` | 每层加一点逻辑，向上传递事件                             |
| **Declaration Merging** | CustomAgentMessages                 | 外部包通过 `declare module` 扩展接口                     |
| **可插拔 Operations**   | tools 的 ReadOps/EditOps/BashOps    | 接口 + 默认实现，可替换为 SSH 远程版                     |
| **JSONL Append-Only**   | session-manager                     | 每条 entry 一行，parentId 形成树                         |
| **生命周期事件链**      | extension runner                    | 事件依次传递给每个扩展的 handler                         |
| **双队列中断**          | Agent.steer/followUp                | steering 中断 tool 执行，followUp 等 agent 闲置          |
