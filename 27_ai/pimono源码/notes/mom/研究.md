# Mom (Master Of Mischief) 源码深度解析

## 一、项目概览

Mom 是一个 **Slack Bot**，背后由 LLM (Claude Sonnet 4.5) 驱动。它的核心理念是 **"自管理"** —— 不预设任何工具，Bot 自己安装依赖、编写脚本、配置环境，逐步把自己打造成用户需要的样子。

### 架构总览

```
用户 (Slack) ←→ SlackBot (slack.ts) ←→ Handler (main.ts) ←→ AgentRunner (agent.ts) ←→ LLM (Claude)
                     ↑                                              ↑
                ChannelStore (store.ts)                    Tools (bash/read/write/edit/attach)
                EventsWatcher (events.ts)                  Sandbox (sandbox.ts)
                Context (context.ts)                       Memory (MEMORY.md)
```

---

## 二、每个文件的职责

### 1. main.ts — 入口与粘合层

**职责**：解析命令行参数、初始化所有组件、定义消息处理流程。

关键设计：

- **per-channel 状态管理**：每个 Slack 频道维护独立的 `ChannelState`（含 runner、store、running 标志）
- **SlackContext 适配器**：main.ts 将 Slack 事件包装为 agent 可消费的统一接口
- **消息更新机制**：用 `accumulatedText` 累积文本，通过 `updatePromise` 链保证 Slack API 调用的顺序性

```typescript
// 核心状态结构
interface ChannelState {
  running: boolean // 是否正在处理
  runner: AgentRunner // LLM agent 执行器
  store: ChannelStore // 持久化存储
  stopRequested: boolean // 用户是否请求停止
}
```

**消息截断**：主消息限制 35K 字符（Slack 限制 40K），线程消息限制 20K。

### 2. agent.ts — Agent 核心（最重要的文件）

**职责**：构建 system prompt、创建 LLM Agent、管理 session、事件订阅、消息流转。

#### System Prompt 构建

agent.ts 是理解 mom 行为的关键。它包含：

| 部分           | 内容                                 |
| -------------- | ------------------------------------ |
| 角色定义       | "You are mom, a Slack bot assistant" |
| Slack 格式指令 | mrkdwn 格式（非 Markdown!）          |
| 频道/用户映射  | 所有频道 ID ↔ 名称、用户 ID ↔ 用户名 |
| 环境信息       | Docker 或 Host 模式描述              |
| 工作区布局     | 目录结构说明                         |
| Skills 系统    | 可扩展的自定义 CLI 工具              |
| Events 系统    | 定时/即时/周期事件调度               |
| Memory 系统    | 持久化记忆                           |
| 工具说明       | bash/read/write/edit/attach          |

#### Agent 生命周期

```
getOrCreateRunner() → 创建Runner(每频道一个，持久化)
    └→ createRunner()
        ├→ 创建 Executor (host 或 docker)
        ├→ 创建 Tools (bash/read/write/edit/attach)
        ├→ 创建 SessionManager (context.jsonl)
        ├→ 创建 Agent (pi-agent-core)
        ├→ 创建 AgentSession (pi-coding-agent)
        └→ 订阅事件 (subscribe)

run() 被调用时:
    1. syncLogToSessionManager() — 同步离线消息
    2. 重载 context.jsonl
    3. 刷新 system prompt（最新 memory + channels + users + skills）
    4. 构建 user message（带时间戳、用户名前缀）
    5. session.prompt() — 调用 LLM
    6. 处理响应/错误/[SILENT] 标记
```

#### 事件订阅（Agent 内部的事件流）

agent.ts 监听以下事件：

- `tool_execution_start` → 在 Slack 显示 `_→ label_`
- `tool_execution_end` → 将工具结果发到线程，错误在主消息显示
- `message_start/end` → 将 LLM 文本响应发给用户
- `auto_compaction_start/end` → 上下文压缩通知
- `auto_retry_start` → 重试通知

### 3. context.ts — 上下文管理

**职责**：管理两个独立的日志文件，确保消息不丢失。

- **context.jsonl**：结构化的 LLM API 消息，SessionManager 用来恢复上下文
- **log.jsonl**：人类可读的频道历史（用户消息 + bot 响应，不含工具调用细节）

#### 核心函数：`syncLogToSessionManager()`

这是**消息同步**的关键。当 mom 重启或者在忙碌时有消息到来，这些消息会被写入 log.jsonl。下次 run() 时，此函数会：

1. 收集 SessionManager 中已有的消息内容（去重用）
2. 读取 log.jsonl，找到不在 context 中的用户消息
3. 按时间排序后追加到 SessionManager

```typescript
// 消息去重的关键：标准化处理
let normalized = content.replace(/^\[\d{4}-\d{2}-\d{2} \d{2}:\d{2}:\d{2}[+-]\d{2}:\d{2}\] /, '')
// 去掉时间戳前缀再比较
```

### 4. store.ts — 持久化存储

**职责**：频道目录管理、附件下载、消息日志。

关键特性：

- **后台异步下载队列**：`processDownloadQueue()` 异步下载 Slack 附件
- **去重机制**：`recentlyLogged` Map 记录最近 60 秒内已记录的消息 ts，防止重复写入
- **附件命名**：`{timestamp_ms}_{sanitized_filename}` 确保唯一性

```typescript
// 下载队列设计
private pendingDownloads: PendingDownload[] = [];
private isDownloading = false;

processAttachments()  → 入队
processDownloadQueue() → 消费（带互斥锁 isDownloading）
downloadAttachment()   → 执行（Bearer token 认证下载）
```

### 5. slack.ts — Slack 集成层

**职责**：Socket Mode 连接、事件路由、消息回填、用户/频道管理。

#### 事件处理流程

```
Slack WebSocket
  ├→ app_mention 事件（频道 @提及）
  │   ├→ logUserMessage() — 同步写日志
  │   ├→ 过滤旧消息（pre-startup）
  │   ├→ "stop" → handleStop()
  │   └→ 排队 → handler.handleEvent()
  │
  └→ message 事件（所有消息 + DM）
      ├→ 跳过 bot 消息、编辑等
      ├→ logUserMessage() — 所有消息都记日志
      ├→ DM → 触发处理
      └→ 非 DM 非 @提及 → 仅记录，不触发
```

#### 消息回填（Backfill）

启动时自动回填所有已交互过的频道的历史消息：

```typescript
backfillAllChannels()
  → 只回填有 log.jsonl 的频道
  → 从最新已记录的 ts 开始拉取（增量）
  → 最多拉 3 页 × 1000 条
  → 写入 log.jsonl
```

#### 频道队列

每个频道有独立的 `ChannelQueue`，**严格顺序执行**，防止同一频道的并发处理。

### 6. events.ts — 事件调度系统

**职责**：文件系统监听驱动的事件调度。

三种事件类型：

| 类型        | 触发方式       | 用途                   |
| ----------- | -------------- | ---------------------- |
| `immediate` | 文件创建即触发 | webhook、外部程序通知  |
| `one-shot`  | 指定时间触发   | 提醒、定时任务         |
| `periodic`  | cron 表达式    | 周期任务（邮件检查等） |

工作原理：

1. `fs.watch()` 监听 `events/` 目录
2. 新 JSON 文件 → 解析 → 根据类型调度
3. `immediate`：立即执行（忽略进程启动前的陈旧事件）
4. `one-shot`：`setTimeout()` 延迟执行，过期则丢弃
5. `periodic`：`croner` 库的 cron 调度
6. 执行时创建合成 `SlackEvent`，通过 `slack.enqueueEvent()` 进入队列

**防抖**：同一文件的变更事件在 100ms 内只处理一次。

### 7. sandbox.ts — 沙箱执行层

**职责**：提供统一的命令执行接口，支持宿主机和 Docker 两种模式。

```typescript
// 两种执行器
HostExecutor  → spawn("sh", ["-c", command])
DockerExecutor → docker exec <container> sh -c '<command>'
```

安全特性：

- 输出限制 10MB（防止内存溢出）
- 支持超时和 AbortSignal
- 进程树清理：`killProcessTree()` 用 `SIGKILL` 杀死整个进程组

### 8. Tools — 工具集

#### bash.ts — 命令执行

- 输出截断：尾部保留 2000 行 或 50KB
- 超限时写入临时文件，告知 LLM 完整输出路径

#### read.ts — 文件读取

- 支持文本和图片（jpg/png/gif/webp → base64 → ImageContent）
- 头部截断（保留前 2000 行 / 50KB）
- 支持 offset/limit 分页读取

#### write.ts — 文件写入

- 自动创建目录
- 使用 `printf '%s'` 加 shell 转义处理特殊字符

#### edit.ts — 精确编辑

- 严格匹配 `oldText`（必须精确唯一）
- 返回 unified diff 格式差异
- 多重校验：不存在、多处匹配、无变化 → 全部报错

#### attach.ts — 文件上传到 Slack

- 通过 `setUploadFunction()` 注入宿主路径转换逻辑

### 9. log.ts — 控制台彩色日志

用 chalk 做彩色输出：绿色=用户消息、黄色=工具调用、红色=错误。

### 10. download.ts — 频道历史导出

`mom --download <channel-id>` 模式：拉取完整频道历史+线程回复，输出为文本。

## 三、核心功能特性深入

### Memory 实现

Mom 的记忆系统是**文件驱动**的，分两级：

```
workspace/
├── MEMORY.md          ← 全局记忆（所有频道共享）
└── <channelId>/
    └── MEMORY.md      ← 频道特定记忆
```

agent.ts 在每次 run() 时读取这两个文件，拼接后注入 system prompt：

```typescript
function getMemory(channelDir: string): string {
  // 1. 读 workspace-level MEMORY.md （全局）
  // 2. 读 channel-specific MEMORY.md
  // 3. 拼接返回，注入到 system prompt 的 "### Current Memory" 区域
}
```

**LLM 自行管理记忆**：system prompt 指示 LLM "Write to MEMORY.md files to persist context across conversations. Update when you learn something important"。LLM 通过 write/edit 工具直接修改 MEMORY.md。

### 完整的消息处理工作流

```
1. 用户在 Slack 发消息 @mom
       ↓
2. slack.ts 的 app_mention 处理器收到事件
       ↓
3. logUserMessage() — 同步写入 log.jsonl + 下载附件
       ↓
4. 检查 startupTs → 老消息只记录不处理
       ↓
5. 检查 "stop" → 直接 abort
       ↓
6. 检查 isRunning → 忙则回复 "Already working"
       ↓
7. 放入 ChannelQueue → handler.handleEvent()
       ↓
8. main.ts handleEvent:
   - 创建 SlackContext 适配器
   - 设置 typing 状态
   - 调用 runner.run(ctx, store)
       ↓
9. agent.ts run():
   a. syncLogToSessionManager() — 补齐离线消息
   b. 重载 context.jsonl
   c. 刷新 system prompt（最新 memory/skills/channels/users）
   d. 构建用户消息 "[timestamp] [username]: text"
   e. session.prompt(userMessage) — 调用 Claude
       ↓
10. Claude 执行循环:
    - 生成文本 → 发到 Slack 主消息
    - 调用工具 → 工具细节发到 Slack 线程
    - 继续推理...
    - 直到返回最终响应
       ↓
11. 处理最终结果:
    - [SILENT] → 删除消息（静默完成）
    - 正常文本 → replaceMessage 更新最终内容
    - 错误 → 显示错误
    - 记录 token 使用量和成本
```

### Skills 系统

LLM 自己编写、自己使用的 CLI 工具：

```
workspace/skills/<name>/SKILL.md  ← 全局 skill
<channelId>/skills/<name>/SKILL.md  ← 频道 skill
```

agent.ts 加载时会把宿主路径转换为容器路径。频道级 skill 优先于全局 skill（同名覆盖）。

### Context Compaction

当上下文过长时，pi-coding-agent 的 SessionManager 会自动压缩：

- `auto_compaction_start` → Slack 显示 "Compacting context..."
- 压缩后在 Slack 线程通知完成

### [SILENT] 响应

周期性事件（如 "每天检查邮箱"）如果没有可报告的内容，LLM 回复 `[SILENT]`，mom 会删除整个状态消息和线程，避免频道刷屏。

## 四、设计亮点总结

1. **per-channel 隔离**：每个频道独立的 state/runner/store/session，互不干扰
2. **双日志架构**：context.jsonl (LLM 上下文恢复) + log.jsonl (历史查询 + 消息同步)
3. **消息不丢失**：启动回填 + log.jsonl 同步 + 去重机制
4. **promise 链排序**：所有 Slack API 调用通过 `updatePromise` 链严格顺序执行
5. **自管理哲学**：不预装工具，LLM 自主 `apk add` / `npm install` / 编写 skills
6. **文件驱动事件**：JSON 文件 + fs.watch = 简单可靠的事件调度
