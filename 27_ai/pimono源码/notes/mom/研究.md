# Mom (Master Of Mischief) 源码深度解析

## 一、项目概览

Mom 是一个 **Slack Bot**，背后由 LLM (Claude Sonnet 4.5) 驱动。它的核心理念是 **"自管理"** —— 不预设任何工具，Bot 自己安装依赖、编写脚本、配置环境，逐步把自己打造成用户需要的样子。

### 架构总览

```
用户 (Slack) ←→ SlackBot (slack.ts) ←→ Handler (main.ts) ←→ AgentRunner (agent.ts) ←→ LLM (Claude)
                     ↑                                              ↑
                ChannelStore (store.ts)                    Tools (bash/read/write/edit/attach)
                EventsWatcher (events.ts)                  Sandbox (sandbox.ts)
                Context (context.ts)                       Memory (MEMORY.md)
```

---

## 二、每个文件的职责

### 1. main.ts — 入口与粘合层

**职责**：解析命令行参数、初始化所有组件、定义消息处理流程。

关键设计：

- **per-channel 状态管理**：每个 Slack 频道维护独立的 `ChannelState`（含 runner、store、running 标志）
- **SlackContext 适配器**：main.ts 将 Slack 事件包装为 agent 可消费的统一接口
- **消息更新机制**：用 `accumulatedText` 累积文本，通过 `updatePromise` 链保证 Slack API 调用的顺序性

```typescript
// 核心状态结构
interface ChannelState {
  running: boolean // 是否正在处理
  runner: AgentRunner // LLM agent 执行器
  store: ChannelStore // 持久化存储
  stopRequested: boolean // 用户是否请求停止
}
```

**消息截断**：主消息限制 35K 字符（Slack 限制 40K），线程消息限制 20K。

### 2. agent.ts — Agent 核心（最重要的文件）

**职责**：构建 system prompt、创建 LLM Agent、管理 session、事件订阅、消息流转。

#### System Prompt 构建

agent.ts 是理解 mom 行为的关键。它包含：

| 部分           | 内容                                 |
| -------------- | ------------------------------------ |
| 角色定义       | "You are mom, a Slack bot assistant" |
| Slack 格式指令 | mrkdwn 格式（非 Markdown!）          |
| 频道/用户映射  | 所有频道 ID ↔ 名称、用户 ID ↔ 用户名 |
| 环境信息       | Docker 或 Host 模式描述              |
| 工作区布局     | 目录结构说明                         |
| Skills 系统    | 可扩展的自定义 CLI 工具              |
| Events 系统    | 定时/即时/周期事件调度               |
| Memory 系统    | 持久化记忆                           |
| 工具说明       | bash/read/write/edit/attach          |

#### Agent 生命周期

```
getOrCreateRunner() → 创建Runner(每频道一个，持久化)
    └→ createRunner()
        ├→ 创建 Executor (host 或 docker)
        ├→ 创建 Tools (bash/read/write/edit/attach)
        ├→ 创建 SessionManager (context.jsonl)
        ├→ 创建 Agent (pi-agent-core)
        ├→ 创建 AgentSession (pi-coding-agent)
        └→ 订阅事件 (subscribe)

run() 被调用时:
    1. syncLogToSessionManager() — 同步离线消息
    2. 重载 context.jsonl
    3. 刷新 system prompt（最新 memory + channels + users + skills）
    4. 构建 user message（带时间戳、用户名前缀）
    5. session.prompt() — 调用 LLM
    6. 处理响应/错误/[SILENT] 标记
```

#### 事件订阅（Agent 内部的事件流）

agent.ts 监听以下事件：

- `tool_execution_start` → 在 Slack 显示 `_→ label_`
- `tool_execution_end` → 将工具结果发到线程，错误在主消息显示
- `message_start/end` → 将 LLM 文本响应发给用户
- `auto_compaction_start/end` → 上下文压缩通知
- `auto_retry_start` → 重试通知

### 3. context.ts — 上下文管理

**职责**：管理两个独立的日志文件，确保消息不丢失。

- **context.jsonl**：结构化的 LLM API 消息，SessionManager 用来恢复上下文
- **log.jsonl**：人类可读的频道历史（用户消息 + bot 响应，不含工具调用细节）

#### 核心函数：`syncLogToSessionManager()`

这是**消息同步**的关键。当 mom 重启或者在忙碌时有消息到来，这些消息会被写入 log.jsonl。下次 run() 时，此函数会：

1. 收集 SessionManager 中已有的消息内容（去重用）
2. 读取 log.jsonl，找到不在 context 中的用户消息
3. 按时间排序后追加到 SessionManager

```typescript
// 消息去重的关键：标准化处理
let normalized = content.replace(/^\[\d{4}-\d{2}-\d{2} \d{2}:\d{2}:\d{2}[+-]\d{2}:\d{2}\] /, '')
// 去掉时间戳前缀再比较
```

### 4. store.ts — 持久化存储

**职责**：频道目录管理、附件下载、消息日志。

关键特性：

- **后台异步下载队列**：`processDownloadQueue()` 异步下载 Slack 附件
- **去重机制**：`recentlyLogged` Map 记录最近 60 秒内已记录的消息 ts，防止重复写入
- **附件命名**：`{timestamp_ms}_{sanitized_filename}` 确保唯一性

```typescript
// 下载队列设计
private pendingDownloads: PendingDownload[] = [];
private isDownloading = false;

processAttachments()  → 入队
processDownloadQueue() → 消费（带互斥锁 isDownloading）
downloadAttachment()   → 执行（Bearer token 认证下载）
```

### 5. slack.ts — Slack 集成层

**职责**：Socket Mode 连接、事件路由、消息回填、用户/频道管理。

#### 事件处理流程

```
Slack WebSocket
  ├→ app_mention 事件（频道 @提及）
  │   ├→ logUserMessage() — 同步写日志
  │   ├→ 过滤旧消息（pre-startup）
  │   ├→ "stop" → handleStop()
  │   └→ 排队 → handler.handleEvent()
  │
  └→ message 事件（所有消息 + DM）
      ├→ 跳过 bot 消息、编辑等
      ├→ logUserMessage() — 所有消息都记日志
      ├→ DM → 触发处理
      └→ 非 DM 非 @提及 → 仅记录，不触发
```

#### 消息回填（Backfill）

启动时自动回填所有已交互过的频道的历史消息：

```typescript
backfillAllChannels()
  → 只回填有 log.jsonl 的频道
  → 从最新已记录的 ts 开始拉取（增量）
  → 最多拉 3 页 × 1000 条
  → 写入 log.jsonl
```

#### 频道队列

每个频道有独立的 `ChannelQueue`，**严格顺序执行**，防止同一频道的并发处理。

### 6. events.ts — 事件调度系统

**职责**：文件系统监听驱动的事件调度。

三种事件类型：

| 类型        | 触发方式       | 用途                   |
| ----------- | -------------- | ---------------------- |
| `immediate` | 文件创建即触发 | webhook、外部程序通知  |
| `one-shot`  | 指定时间触发   | 提醒、定时任务         |
| `periodic`  | cron 表达式    | 周期任务（邮件检查等） |

工作原理：

1. `fs.watch()` 监听 `events/` 目录
2. 新 JSON 文件 → 解析 → 根据类型调度
3. `immediate`：立即执行（忽略进程启动前的陈旧事件）
4. `one-shot`：`setTimeout()` 延迟执行，过期则丢弃
5. `periodic`：`croner` 库的 cron 调度
6. 执行时创建合成 `SlackEvent`，通过 `slack.enqueueEvent()` 进入队列

**防抖**：同一文件的变更事件在 100ms 内只处理一次。

### 7. sandbox.ts — 沙箱执行层

**职责**：提供统一的命令执行接口，支持宿主机和 Docker 两种模式。

```typescript
// 两种执行器
HostExecutor  → spawn("sh", ["-c", command])
DockerExecutor → docker exec <container> sh -c '<command>'
```

安全特性：

- 输出限制 10MB（防止内存溢出）
- 支持超时和 AbortSignal
- 进程树清理：`killProcessTree()` 用 `SIGKILL` 杀死整个进程组

### 8. Tools — 工具集

#### bash.ts — 命令执行

- 输出截断：尾部保留 2000 行 或 50KB
- 超限时写入临时文件，告知 LLM 完整输出路径

#### read.ts — 文件读取

- 支持文本和图片（jpg/png/gif/webp → base64 → ImageContent）
- 头部截断（保留前 2000 行 / 50KB）
- 支持 offset/limit 分页读取

#### write.ts — 文件写入

- 自动创建目录
- 使用 `printf '%s'` 加 shell 转义处理特殊字符

#### edit.ts — 精确编辑

- 严格匹配 `oldText`（必须精确唯一）
- 返回 unified diff 格式差异
- 多重校验：不存在、多处匹配、无变化 → 全部报错

#### attach.ts — 文件上传到 Slack

- 通过 `setUploadFunction()` 注入宿主路径转换逻辑

### 9. log.ts — 控制台彩色日志

用 chalk 做彩色输出：绿色=用户消息、黄色=工具调用、红色=错误。

### 10. download.ts — 频道历史导出

`mom --download <channel-id>` 模式：拉取完整频道历史+线程回复，输出为文本。

## 三、核心功能特性深入

### Memory 实现

Mom 的记忆系统是**文件驱动**的，分两级：

```
workspace/
├── MEMORY.md          ← 全局记忆（所有频道共享）
└── <channelId>/
    └── MEMORY.md      ← 频道特定记忆
```

agent.ts 在每次 run() 时读取这两个文件，拼接后注入 system prompt：

```typescript
function getMemory(channelDir: string): string {
  // 1. 读 workspace-level MEMORY.md （全局）
  // 2. 读 channel-specific MEMORY.md
  // 3. 拼接返回，注入到 system prompt 的 "### Current Memory" 区域
}
```

**LLM 自行管理记忆**：system prompt 指示 LLM "Write to MEMORY.md files to persist context across conversations. Update when you learn something important"。LLM 通过 write/edit 工具直接修改 MEMORY.md。

### 完整的消息处理工作流

```
1. 用户在 Slack 发消息 @mom
       ↓
2. slack.ts 的 app_mention 处理器收到事件
       ↓
3. logUserMessage() — 同步写入 log.jsonl + 下载附件
       ↓
4. 检查 startupTs → 老消息只记录不处理
       ↓
5. 检查 "stop" → 直接 abort
       ↓
6. 检查 isRunning → 忙则回复 "Already working"
       ↓
7. 放入 ChannelQueue → handler.handleEvent()
       ↓
8. main.ts handleEvent:
   - 创建 SlackContext 适配器
   - 设置 typing 状态
   - 调用 runner.run(ctx, store)
       ↓
9. agent.ts run():
   a. syncLogToSessionManager() — 补齐离线消息
   b. 重载 context.jsonl
   c. 刷新 system prompt（最新 memory/skills/channels/users）
   d. 构建用户消息 "[timestamp] [username]: text"
   e. session.prompt(userMessage) — 调用 Claude
       ↓
10. Claude 执行循环:
    - 生成文本 → 发到 Slack 主消息
    - 调用工具 → 工具细节发到 Slack 线程
    - 继续推理...
    - 直到返回最终响应
       ↓
11. 处理最终结果:
    - [SILENT] → 删除消息（静默完成）
    - 正常文本 → replaceMessage 更新最终内容
    - 错误 → 显示错误
    - 记录 token 使用量和成本
```

### Skills 系统

LLM 自己编写、自己使用的 CLI 工具：

```
workspace/skills/<name>/SKILL.md  ← 全局 skill
<channelId>/skills/<name>/SKILL.md  ← 频道 skill
```

agent.ts 加载时会把宿主路径转换为容器路径。频道级 skill 优先于全局 skill（同名覆盖）。

### Context Compaction

当上下文过长时，pi-coding-agent 的 SessionManager 会自动压缩：

- `auto_compaction_start` → Slack 显示 "Compacting context..."
- 压缩后在 Slack 线程通知完成

### [SILENT] 响应

周期性事件（如 "每天检查邮箱"）如果没有可报告的内容，LLM 回复 `[SILENT]`，mom 会删除整个状态消息和线程，避免频道刷屏。

## 四、设计亮点总结

1. **per-channel 隔离**：每个频道独立的 state/runner/store/session，互不干扰
2. **双日志架构**：context.jsonl (LLM 上下文恢复) + log.jsonl (历史查询 + 消息同步)
3. **消息不丢失**：启动回填 + log.jsonl 同步 + 去重机制
4. **promise 链排序**：所有 Slack API 调用通过 `updatePromise` 链严格顺序执行
5. **自管理哲学**：不预装工具，LLM 自主 `apk add` / `npm install` / 编写 skills
6. **文件驱动事件**：JSON 文件 + fs.watch = 简单可靠的事件调度

### 1. new.md — 多平台聊天架构重设计

这是 mom 最重要的架构演进文档，核心目标是**解除对 Slack 的强耦合**。

#### 当前痛点

原架构中 Slack 概念深度渗透到各层：

- `SlackContext` 接口暴露了 Slack 特有的概念（线程、typing indicator）
- Agent 代码直接引用 `mrkdwn` 格式、`<@user>` 提及语法
- 存储层使用 Slack 时间戳 `ts` 作为消息 ID
- 如果要接入 Discord，只能整体复制一份代码

#### 新架构三层解耦

```
入口层 (CLI)  →  适配器层 (PlatformAdapter)  →  Agent 层 (MomAgent)  →  存储层 (ChannelStore)
```

**核心思想**：Agent 完全不知道消息来自哪个平台。它只接收统一的 `ChannelMessage`，输出标准 Markdown，由适配器做平台特定的转换。

##### (a) ChannelMessage — 统一消息格式

```typescript
interface ChannelMessage {
  id: string // 平台原生 ID（保留格式）
  channelId: string // 频道/会话 ID
  timestamp: string // ISO 8601 标准时间
  sender: { id; username; displayName?; isBot }
  text: string // 纯文本内容
  rawText?: string // 平台原始格式（调试用）
  attachments: ChannelAttachment[]
  isMention: boolean // 是否直接 @bot
  replyTo?: string // 回复目标（线程化）
  metadata?: Record<string, unknown> // 平台特有数据
}
```

**设计要点**：

- `id` 保留平台原始格式（Slack: `1734567890.123456`，Discord: snowflake），不做归一化
- `rawText` 保留原始文本用于调试（如 Slack 的 `<@U123>` 格式），`text` 是已解析的友好格式
- `metadata` 是逃生舱口，允许适配器传递平台特有数据而不污染核心接口

##### (b) PlatformAdapter — 适配器接口

每个平台实现一个适配器，职责是：

1. 连接平台（WebSocket / HTTP）
2. 将平台事件转为 `ChannelMessage`
3. 接收 `AgentSessionEvent` 并渲染为平台特定 UI

```typescript
interface PlatformAdapter {
  name: string // 用于目录命名，如 "slack-acme"
  start(): Promise<void>
  stop(): Promise<void>
  getChannels(): ChannelInfo[]
  getUsers(): UserInfo[]
}
```

**适配器决定一切 UI**：消息分割方式、是否用线程、markdown→mrkdwn 转换、rate limiting。Agent 完全不管这些。

##### (c) MomAgent — 平台无关的 Agent

```typescript
interface MomAgent {
  handleMessage(
    message: ChannelMessage,
    context: ChannelContext,
    onEvent: (event: AgentSessionEvent) => Promise<void> // 回调！
  ): Promise<{ stopReason: string; errorMessage?: string }>
  abort(channelId: string): void
  isRunning(channelId: string): boolean
}
```

**关键设计：回调模式**。Agent 通过 `onEvent` 回调把事件推给适配器，而不是直接调用 Slack API。这实现了依赖反转——Agent 不依赖任何平台。

##### (d) 存储层命名空间

目录结构变为按适配器分隔：`channels/<adapter-name>/<channelId>/`

```
channels/
├── slack-acme/C0A34FL8PMH/    ← Slack 工作区的频道
└── discord-mybot/123456789/    ← Discord 服务器的频道
```

事件文件中 channelId 也变为 `"slack-acme/C123"` 格式。

##### (e) 安全：Bubblewrap 通道隔离

一个高级特性：在 Linux/Docker 中用 bubblewrap 做 **per-user 的文件系统隔离**。

**原理**：Mom 有 bash 权限可以读所有频道日志。如果用户 A 在私密频道 X 中跟 mom 对话，用户 B 在另一个频道问 mom "读一下频道 X 的日志"——这是安全风险。

**解法**：执行 bash 前，根据请求用户的频道权限，用 bwrap 把无权限的频道目录替换为空 tmpfs：

```typescript
function wrapWithBwrap(command: string, deniedChannels: string[]): string {
  const args = [
    '--bind / /', // 挂载所有文件
    ...deniedChannels.map(
      ch => `--tmpfs /workspace/channels/${ch}` // 用空 tmpfs 覆盖禁止的频道
    ),
    '--die-with-parent'
  ]
  return `bwrap ${args.join(' ')} -- ${command}`
}
```

被隔离的频道在进程看来完全不存在——目录为空。

##### (f) 格式转换

Agent 输出标准 Markdown，适配器负责转换：

```
Markdown      →   Slack mrkdwn:    **bold** → *bold*,  [text](url) → <url|text>
Markdown      →   Discord embed:   保持原样（Discord 原生支持 Markdown）
@username     →   Slack: <@U123>,  Discord: <@123456789>
```

##### (g) 渐进式迁移路径

```
Phase 1: 统一存储格式（不破坏现有功能）
Phase 2: 提取适配器接口（包装现有 SlackBot）
Phase 3: Agent 解耦（删除 agent.ts 中所有 Slack 代码）
Phase 4: 新增 Discord（复用所有存储和 agent 代码）
```

##### (h) Custom Tools — 宿主端工具

Docker 沙箱中无法访问宿主机 API（Gmail、日历等）。解决方案是 **Custom Tool 机制**：

- 工具代码放在 `data/tools/<name>/index.ts`
- Mom 进程用 `jiti`（TS 运行时编译器）在宿主端加载
- Agent 通过 `invoke_tool("gmail", {...})` 调用
- 宿主端执行，结果返回给 Agent

```
Agent → invoke_tool("gmail", {action: "search"})
              ↓
  Mom 进程在宿主端执行 gmail/index.ts
              ↓
  结果返回给 Agent（不经过 Docker）
```

### 2. events.md — 事件调度系统

Mom 可以被"事件"触发执行任务，而不仅仅是用户消息。事件是 `workspace/events/` 目录下的 JSON 文件。

#### 三种事件类型

| 类型          | 触发           | 生命周期          | 典型使用                   |
| ------------- | -------------- | ----------------- | -------------------------- |
| **immediate** | 文件创建即触发 | 执行后删除        | webhook 回调、外部程序通知 |
| **one-shot**  | 指定时间触发   | 执行后/过期后删除 | 提醒、定时任务             |
| **periodic**  | cron 调度      | 持续存在直到被删  | 日报、邮件检查、定期巡检   |

#### 启动行为（防止旧事件重放）

```
immediate：如果 mtime < 进程启动时间 → 陈旧，删除不执行！
one-shot：  如果 at < 当前时间 → 过期，删除不执行！
periodic：  如果错过了排期 → 不补执行！等下一次
```

**为什么 immediate 要检查 mtime？** 假设 mom 宕机 2 小时，期间 webhook 写了 50 个 immediate 事件文件。重启后不应该把这 50 个全执行一遍（因为时效性已过）。

#### 文件系统监听

```
fs.watch("events/") + 100ms 防抖
  ├→ 新文件 → 解析 → 调度
  ├→ 文件修改 → 取消旧调度 → 重新解析调度（允许"改期"）
  └→ 文件删除 → 取消调度
```

##### 解析错误处理

JSON 解析失败 → 指数退避重试（100ms → 200ms → 400ms）→ 仍失败则删除文件。为什么要重试？因为 `fs.watch` 可能在文件写入到一半时触发。

#### 队列集成

```
事件执行 → enqueueEvent() → ChannelQueue（最多 5 个排队）
用户 @mom  → handleEvent() → 如果正在忙拒绝 "Already working"
```

两者的区别：事件永远排队，用户提及在忙时被拒绝。这很合理——事件是程序化的，重要性更高。

#### [SILENT] 静默响应

周期事件的核心设计。例如 "每 15 分钟检查邮箱"：

- 有新邮件 → 正常回复摘要
- 没有新邮件 → LLM 回复 `[SILENT]` → 删除状态消息，什么都不发

避免频道被无意义的 "No new emails" 刷屏。

#### 消息格式

事件触发时，发给 Agent 的消息带有元数据前缀：

```
[EVENT:daily-inbox.json:periodic:0 9 * * 1-5] Check inbox and post summary
```

Agent 能据此知道：事件文件名、类型、调度规则。

### 3. sandbox.md — Docker 沙箱

#### 为什么需要沙箱

Mom 通过 Slack 暴露了一个 LLM 驱动的 bash 工具。Slack 工作区中的任何人都可以通过 prompt injection 让 mom 执行命令。如果 mom 直接在宿主机跑：

```
恶意用户 → @mom 请读取 ~/.ssh/id_rsa 的内容
         → @mom 请执行 rm -rf /
```

Docker 沙箱将工具执行隔离在容器中，只能访问 `/workspace`（挂载的数据目录）。

#### 架构

```
宿主机:
  Mom 进程 (Node.js)
  ├── Slack 连接         ← 宿主端
  ├── LLM API 调用       ← 宿主端
  └── 工具执行 ──→ Docker 容器
                   ├── bash, git, gh 等
                   └── /workspace (mount)
```

- **宿主端**：网络连接（Slack WebSocket）、LLM API 调用
- **容器端**：bash / read / write / edit 等所有工具操作

#### "Mom 管理自己的电脑"

容器被视为 mom 的个人电脑。她自己安装工具、配置凭据。这是"自管理"哲学在安全边界内的体现。

```
用户：@mom 检查 spine-runtimes 仓库
Mom：我需要 gh CLI，正在安装... (apk add github-cli)
Mom：我需要 GitHub token，请提供一个
用户：ghp_xxxx...
Mom：(echo "ghp_xxxx" | gh auth login --with-token)
Mom：好了，正在检查仓库...
```

#### 持久性

容器状态在 `docker stop/start` 和宿主机重启间持久化。已安装的工具和配置保留，直到 `docker rm`。

### 4. artifacts-server.md — Artifact 服务器

让 Mom 创建 HTML 文件，并通过公网 URL 即时分享给用户。

#### 架构

```
Mom 创建 HTML → /workspace/artifacts/files/dashboard.html
                        ↓
              Express 静态文件服务 (port 8080)
                        ↓
              Cloudflare Tunnel (cloudflared)
                        ↓
              https://xxx.trycloudflare.com/dashboard.html
```

#### Live Reload 机制

这是一个巧妙的开发体验设计：

1. **chokidar** 监听 `files/` 目录的所有变化
2. 文件变化 → WebSocket 广播 `{type: "reload", file: "xxx.html"}`
3. HTML 页面注入了 live reload 脚本（通过 `?ws=true` 查询参数启用）
4. 浏览器收到消息 → 自动 `window.location.reload()`

这意味着：Mom 在修改 HTML 时，用户在浏览器中能实时看到变化。适合做 **迭代式 dashboard 开发**。

#### Live Reload 脚本注入

Express 中间件拦截 `.html` 请求，在 `</body>` 前注入 WebSocket 重连脚本。注入只在 URL 带 `?ws=true` 时生效。

#### 安全措施

- **路径遍历防护**：`path.resolve()` 检查解析后的路径是否以 `FILES_DIR` 开头
- **Cache-Control：no-store**：确保每次请求都获取最新文件
- **graceful shutdown**：SIGTERM/SIGINT 时关闭 watcher 和 server

#### Cloudflare Tunnel

使用 `cloudflared` 的免费快速隧道（`trycloudflare.com`），**无需域名、无需公网 IP**。每次启动获得一个随机 URL，适合临时分享。

### 5. slack-bot-minimal-guide.md — Slack Bot 搭建指南

这是一份实操指南，说明如何用最少代码连接 Slack。

#### Socket Mode vs HTTP

Mom 使用 **Socket Mode**（WebSocket 连接），而不是传统的 HTTP webhook：

| 对比       | HTTP Webhook     | Socket Mode        |
| ---------- | ---------------- | ------------------ |
| 公网 IP    | 需要             | 不需要             |
| HTTPS 证书 | 需要             | 不需要             |
| NAT 穿透   | 不行             | 自动               |
| 适用场景   | 生产环境、大规模 | 开发/内部/防火墙后 |

Socket Mode 的核心优势：**零基础设施**。只需要两个 token，运行一个 Node.js 脚本就能连接。

#### 两个 Token

```
xoxb-... (Bot Token)  → 操作权限（发消息、读频道、上传文件等）
xapp-... (App Token)  → WebSocket 连接权限
```

Bot Token 需要 16 个 scope，覆盖了频道读写、DM、文件、用户信息等。

#### 最小事件处理

```javascript
socketClient.on('app_mention', async ({ event, ack }) => {
  await ack() // 必须先确认收到！
  const response = await yourAgent(event.text)
  await webClient.chat.postMessage({ channel: event.channel, text: response })
})
```

`ack()` 是关键——Slack 要求 3 秒内确认收到事件，否则会重试。所以先 ack，再异步处理。

#### 监听策略

- `app_mention`：只在被 @提及时触发（mom 的触发入口）
- `message.channels` + `message.im`：监听所有消息用于写日志（context）
- 过滤 `event.subtype === undefined`，只处理"正常消息"（排除编辑、删除等）

### 6. v86.md — v86 虚拟化方案评估

v86 是一个 **用 JavaScript/WASM 实现的 x86 模拟器**，能在 Node.js/浏览器中运行完整的 Linux。Mom 评估它作为 Docker 的替代沙箱方案。

#### 为什么考虑 v86？

Docker 需要在宿主机安装 Docker daemon（macOS 需要 Docker Desktop），不是所有环境都方便。v86 纯 JS 实现，**零依赖即可运行完整 Linux VM**。

#### 架构

```
宿主 (Node.js)
├── v86 模拟器 (WASM) ─── Alpine Linux (32-bit x86)
│                           ├── Node.js 22
│                           ├── Python 3.12
│                           └── git, curl 等
├── rootlessRelay (WebSocket) ── NAT 网络
│   ├── DHCP
│   ├── DNS 代理
│   └── TCP/UDP NAT
└── 9p 文件系统 ── 宿主↔VM 文件交换
```

#### 三种通信方式

| 方式           | 用途               | 特点                                     |
| -------------- | ------------------ | ---------------------------------------- |
| Serial Console | 发送命令、读取输出 | 最简单，字节级 I/O                       |
| 9p Filesystem  | 文件读写           | `emulator.create_file()` / `read_file()` |
| Network (TCP)  | 访问宿主服务       | 通过 10.0.2.2 网关 → localhost           |

#### 性能特征

| 指标       | 数值       |
| ---------- | ---------- |
| 冷启动     | ~20-25 秒  |
| 状态恢复   | ~2-3 秒    |
| 内存占用   | ~512 MB    |
| 总依赖大小 | ~60 MB     |
| 状态快照   | ~80-100 MB |

**State Save/Restore** 是杀手锏：首次冷启动 25 秒（安装包、配置环境），之后每次从快照恢复只要 2-3 秒。快照包含所有已安装的包和文件。

#### rootlessRelay 网络

v86 的网络通过 WebSocket relay 实现。VM 使用 `10.0.2.x` 内网地址，relay 做 NAT + DNS + DHCP。

**必须打的补丁**：原版 relay 不允许 VM 连接宿主机（10.0.2.2 网关被拦截）。需要修改 relay.js：

1. 禁用网关 IP 的 reverseTCP 处理
2. 重定向网关 TCP/UDP 到 `127.0.0.1`

这样 VM 内 `wget http://10.0.2.2:8080/` 就能访问宿主机的 localhost:8080。

#### 局限性

- **只支持 32 位 x86**（无 64 位支持）
- **无入站连接**（VM 在 NAT 后）
- **无 ICMP ping**
- 性能远不如 Docker（模拟器开销）

#### 典型工作流

```
首次运行:
  1. 启动 rootlessRelay
  2. 冷启动 v86 + Alpine (~25s)
  3. 网络配置 (modprobe virtio-net && ip link set eth0 up && udhcpc)
  4. 安装包 (apk add nodejs npm python3 git)
  5. 保存状态快照

后续运行:
  1. 启动 rootlessRelay
  2. 恢复快照 (~2s)
  3. 直接执行命令
```

#### v86 vs Docker 对比

| 特性     | Docker                 | v86                   |
| -------- | ---------------------- | --------------------- |
| 安装依赖 | Docker daemon          | 无（纯 JS）           |
| 启动速度 | ~1 秒                  | ~2-3 秒（快照恢复）   |
| 性能     | 接近原生               | 模拟器开销大          |
| 架构     | 原生（amd64/arm64）    | 仅 32-bit x86         |
| 隔离级别 | 容器级                 | VM 级（更强）         |
| 跨平台   | Linux / macOS(Desktop) | 任何有 Node.js 的地方 |

**结论**：v86 适合作为 Docker 不可用时的备选方案，或者需要更强隔离的场景。但 Docker 在性能和生态上仍是首选。

## 六、整体架构演进总结

docs 目录揭示了 Mom 从 **"Slack 专用 bot"** 到 **"多平台 AI Agent 框架"** 的演进路线：

```
Phase 0 (当前): Slack-only, 单适配器, host/docker 沙箱
    ↓
Phase 1: 统一消息格式 (ChannelMessage)
    ↓
Phase 2: 提取适配器接口 (PlatformAdapter)
    ↓
Phase 3: Agent 解耦 (标准 Markdown 输出)
    ↓
Phase 4: 多平台 (Slack + Discord + CLI)
    ↓
Phase ∞: 多沙箱 (Docker + v86), Custom Tools, Artifacts Server
```

**核心设计原则**：

1. **适配器模式**：平台差异封装在适配器内，Agent 层完全通用
2. **回调反转依赖**：Agent 通过 `onEvent` 回调推事件，不直接调用任何平台 API
3. **文件系统即 API**：事件调度、记忆、日志全部用文件系统，简单可靠
4. **渐进式安全**：host → Docker → v86 → bubblewrap，按需选择隔离级别
5. **自管理哲学**：不预装工具，LLM 自主管理运行环境
