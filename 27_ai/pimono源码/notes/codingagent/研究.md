# coding-agent 架构全景解析

> 包名：`@mariozechner/pi-coding-agent`，CLI 命令名 `pi`，版本 0.55.4
> 一个终端内的 AI 编码助手，支持交互模式、单次打印、RPC 嵌入三种运行方式。

**coding-agent 是一个终端 AI 编码助手**，核心架构分为 5 层：

1. **入口层** — cli.ts → main.ts：参数解析（两轮）→ 初始化全部管理器 → 选择运行模式
2. **核心层** — `AgentSession` 是大脑，管理对话生命周期、工具调用、模型切换、压缩
3. **工具层** — 7 个内置工具：`bash`/`read`/`write`/`edit`/`grep`/`find`/`ls`
4. **模式层** — 三种运行模式：交互 TUI（30+ 组件）、单次打印、RPC 嵌入
5. **基础设施** — 认证、设置、包管理、扩展系统、资源发现、会话持久化

**最关键的 5 个文件**：

- `agent-session.ts` — 整个系统的状态机和协调者
- `sdk.ts` — 创建 session 的工厂入口
- main.ts — 启动流程和初始化链
- `system-prompt.ts` — 构建发送给 LLM 的系统提示词
- `session-manager.ts` — JSONL 树形会话持久化

## 二、顶层入口文件

### `src/cli.ts` — CLI 入口

- 整个程序的入口点 (`bin: { "pi": "dist/cli.js" }`)
- 仅 3 行代码：设置进程标题 `pi`，调用 `main()`

### `src/main.ts` — 主函数（最重要的文件之一）

- **启动流程**：
  1. 处理包管理命令 (`pi install/remove/update/list`)
  2. 处理 `pi config` 命令
  3. 运行迁移 (`migrations.ts`)
  4. **两轮参数解析**：第一轮获取 `--extension` 路径 → 加载扩展发现自定义 flag → 第二轮完整解析
  5. 初始化：SettingsManager → AuthStorage → ModelRegistry → ResourceLoader → 加载扩展
  6. 解析模型作用域 (`--models`)、创建 SessionManager
  7. 调用 `createAgentSession()` 创建核心 session
  8. 根据模式运行：`InteractiveMode` / `runPrintMode()` / `runRpcMode()`

### `src/config.ts` — 路径与配置常量

- **检测安装方式**：Bun 二进制 / npm / pnpm / yarn / bun
- **包资源路径**：主题目录、HTML 导出模板、package.json、README 等
- **用户配置路径** (`~/.pi/agent/`)：
  - `sessions/` — 会话文件
  - `settings.json` — 用户设置
  - `models.json` — 自定义模型
  - `auth.json` — 认证凭据
  - `prompts/` — 提示词模板
  - `themes/` — 自定义主题
  - `bin/` — 管理的二进制工具 (fd, rg)

### `src/index.ts` — SDK 公开 API

- 对外导出所有核心类型和工厂函数
- 让其他包可以 `import { createAgentSession } from "@mariozechner/pi-coding-agent"`

### `src/migrations.ts` — 版本迁移

- `migrateAuthToAuthJson()`：将旧的 oauth.json 迁移到 auth.json
- `migrateSessionsFromAgentRoot()`：修复 v0.30.0 的 session 放错位置
- `migrateCommandsToPrompts()`：commands/ 目录重命名为 prompts/

## 十一、关键设计模式

### 1. Session 为中心

所有模式（交互/打印/RPC）都围绕 `AgentSession` 工作，模式只负责 I/O 适配。

### 2. 两轮参数解析

先解析基础参数以发现扩展 → 加载扩展获取自定义 flag → 再完整解析。

### 3. 可插拔工具操作

工具支持自定义文件操作（`FileOperations` 接口），使远程执行 (SSH) 成为可能。

### 4. 扩展钩子系统

扩展可以：注册自定义工具、拦截工具调用 (before/after)、注册 Provider、添加 UI 组件。

### 5. JSONL 会话树

会话以 JSONL 存储，支持树形结构（分支/合并），而非简单线性历史。

### 6. 上下文自动压缩

对话过长时自动用 LLM 生成摘要，保留关键信息的同时大幅减少 token。

### 7. 资源层级发现

项目上下文从 CWD 向上遍历寻找 `AGENTS.md`/`CLAUDE.md`，支持全局 + 项目级配置合并。

## 十二、数据流概览

```
1. 用户输入 → CLI 参数 / TUI 编辑器 / 管道 / RPC JSON
2. main.ts 初始化链：Settings → Auth → ModelRegistry → ResourceLoader → Extensions
3. createAgentSession() 创建 AgentSession
4. AgentSession 构建 system prompt (工具+上下文+技能)
5. AgentSession 发送消息给 LLM，流式接收响应
6. LLM 返回工具调用 → AgentSession 执行工具 → 结果回传 LLM
7. 循环直到 LLM 完成响应
8. SessionManager 持久化到 JSONL
9. 如需要，触发 compaction 压缩历史
```

## 一、AgentSession — 核心状态机 (agent-session.ts, 3146 行)

### 1.1 类定位

`AgentSession` 是整个系统最核心的类。所有三种运行模式（交互/打印/RPC）都围绕它工作。
模式层只负责 I/O 适配，所有业务逻辑都在 AgentSession 中。

### 1.2 内部状态

```typescript
class AgentSession {
  // 核心依赖
  readonly agent: Agent // 底层 Agent（来自 pi-agent-core）
  readonly sessionManager: SessionManager // 会话持久化
  readonly settingsManager: SettingsManager // 设置管理

  // 消息队列（三种消息类型）
  private _steeringMessages: string[] // 打断当前运行的消息
  private _followUpMessages: string[] // 等待运行结束后发送的消息
  private _pendingNextTurnMessages: CustomMessage[] // 下一轮附带的上下文消息

  // 自动压缩状态
  private _compactionAbortController?: AbortController
  private _autoCompactionAbortController?: AbortController
  private _overflowRecoveryAttempted: boolean // 防止 overflow 无限循环

  // 自动重试状态
  private _retryAttempt: number // 当前重试次数
  private _retryPromise?: Promise<void> // 等待重试完成的 promise
  private _retryAbortController?: AbortController

  // bash 执行
  private _bashAbortController?: AbortController
  private _pendingBashMessages: BashExecutionMessage[] // 流式中延迟的 bash 结果

  // 扩展系统
  private _extensionRunner?: ExtensionRunner
  private _turnIndex: number

  // 工具注册
  private _toolRegistry: Map<string, AgentTool> // 所有可用工具
  private _baseToolRegistry: Map<string, AgentTool> // 内置工具
  private _toolPromptSnippets: Map<string, string> // 工具描述片段
  private _toolPromptGuidelines: Map<string, string[]> // 工具提示指南

  private _baseSystemPrompt: string // 基础系统提示（不含扩展修改）
}
```

### 1.3 事件处理流程（最重要的逻辑）

AgentSession 订阅底层 Agent 的事件流，每个事件按以下顺序处理：

```
Agent 事件 → _handleAgentEvent()
    │
    ├── 同步：为 agent_end 创建 RetryPromise（防止竞态）
    │
    └── 异步队列：_processAgentEvent()
          │
          ├── 1. 清理消息队列（steering/followUp 消息被消费时移除）
          │
          ├── 2. _emitExtensionEvent()  ← 扩展先收到事件
          │
          ├── 3. _emit() → 通知所有外部监听器（TUI/RPC 等）
          │
          ├── 4. 持久化（message_end 时写入 session）
          │        ├── custom → appendCustomMessageEntry
          │        ├── user/assistant/toolResult → appendMessage
          │        └── assistant 无错误 → 重置重试计数
          │
          └── 5. agent_end 后处理
                   ├── _isRetryableError() → _handleRetryableError()（指数退避重试）
                   └── _checkCompaction()（检查是否需要压缩）
```

### 1.4 prompt() 完整流程

```
用户调用 prompt(text, options)
    │
    ├── 1. 尝试扩展命令 (/command → extension handler)
    │      如果是扩展命令，直接执行并返回
    │
    ├── 2. 扩展 input 事件 (可被拦截/转换)
    │
    ├── 3. 展开 skill 命令 (/skill:name → 读取文件内容)
    │      和提示词模板 (/template → 替换参数)
    │
    ├── 4. 如果正在流式中：
    │      ├── streamingBehavior='steer' → 打断并插入
    │      └── streamingBehavior='followUp' → 排队等完成
    │
    ├── 5. 刷新待处理 bash 消息
    │
    ├── 6. 验证模型和 API Key
    │
    ├── 7. 预检查：是否需要压缩（处理之前中断的回复）
    │
    ├── 8. 构建消息数组：
```

### 1.5 两种消息队列机制

| 类型         | 方法                                      | 行为                                     | 使用场景                   |
| ------------ | ----------------------------------------- | ---------------------------------------- | -------------------------- |

## 四、SessionManager — 会话持久化 (session-manager.ts, 1410 行)

### 4.1 存储格式：JSONL

每个会话是一个 `.jsonl` 文件，存储在 `~/.pi/agent/sessions/<encoded-cwd>/` 目录下。

```
第1行: {"type":"session","version":3,"id":"abc123","timestamp":"...","cwd":"/path"}
第2行: {"type":"thinking_level_change","id":"a1","parentId":null,"thinkingLevel":"medium"}
第3行: {"type":"model_change","id":"a2","parentId":"a1","provider":"anthropic","modelId":"claude-opus-4-5"}
第4行: {"type":"message","id":"a3","parentId":"a2","message":{"role":"user","content":[...]}}
第5行: {"type":"message","id":"a4","parentId":"a3","message":{"role":"assistant","content":[...]}}
...
```

### 4.2 树形结构

**每个条目都有 `id` 和 `parentId`**，链表形式构成一棵树：

```
  a1 → a2 → a3 → a4 → a5 (主分支)
                   │
                   └→ b1 → b2 (分支 fork)
```

- `getBranch(leafId)`: 从叶子节点回溯到根，得到一条路径
- `getTree()`: 返回完整的树结构
- `fork(entryId)`: 从某个节点创建新分支（新 parentId 指向该节点）
- `navigateTree(targetId)`: 切换到树中的另一个节点

### 4.3 buildSessionContext() — 消息重建

从 JSONL 条目重建 LLM 对话上下文：

```
1. 从 leafId 回溯到根节点，收集路径上所有条目
2. 沿途记录最新的 thinkingLevel 和 model
3. 找到最新的 compaction 条目（如果有）
4. 构建消息列表:
   有 compaction:
     [压缩摘要] + [保留的消息(firstKeptEntryId..compaction)] + [compaction之后的消息]
   无 compaction:
     [所有消息(包含 branch_summary 和 custom_message)]
```

### 4.4 条目类型

| 类型                    | 参与 LLM 上下文 | 说明                                                    |
| ----------------------- | :-------------: | ------------------------------------------------------- |
| `message`               |       ✅        | 标准 LLM 消息 (user/assistant/toolResult/bashExecution) |
| `custom_message`        |       ✅        | 扩展注入的消息，display 控制 TUI 渲染                   |
| `branch_summary`        |       ✅        | 分支切换时的摘要                                        |
| `compaction`            |    ✅(摘要)     | 压缩条目，包含 summary 和 firstKeptEntryId              |
| `thinking_level_change` |       ❌        | 元数据                                                  |
| `model_change`          |       ❌        | 元数据                                                  |
| `custom`                |       ❌        | 扩展私有数据，不参与上下文                              |
| `label`                 |       ❌        | 用户书签                                                |
| `session_info`          |       ❌        | 会话名称等元数据                                        |

### 4.5 版本迁移

- v1 → v2: 添加 `id`/`parentId` 树形结构
- v2 → v3: `hookMessage` 角色重命名为 `custom`

## 五、Compaction — 上下文压缩 (compaction.ts, 813 行)

### 5.1 为什么要压缩

LLM 有上下文窗口限制（如 200K token）。长对话会超出限制。
压缩通过 LLM 生成摘要来缩减旧消息，保留近期消息。

### 5.2 触发条件（两种）

```
agent_end 事件后 → _checkCompaction(assistantMessage)
    │
    ├── 情况1: Context Overflow（溢出）
    │   条件: isContextOverflow(message, contextWindow) && 同一模型 && 非前次压缩产物
    │   处理: 删除错误消息 → 自动压缩 → 自动重试
    │   防护: _overflowRecoveryAttempted 防止无限循环
    │
    └── 情况2: Threshold（阈值）
        条件: contextTokens > contextWindow - reserveTokens
        默认: reserveTokens = 16384
        处理: 自动压缩 → 不自动重试（用户继续）
```

### 5.3 压缩算法

```
prepareCompaction(pathEntries, settings)
    │
    ├── 1. 找到上一次压缩的位置 (prevCompactionIndex)
    │
    ├── 2. 估算当前 token 使用量
    │      estimateContextTokens(): 使用最后一条 assistant 的 usage
    │      + 之后消息的 chars/4 估算
    │
    ├── 3. findCutPoint() — 找切割点
    │      从最新消息向前累积 token，超过 keepRecentTokens (默认 20000) 时停止
    │      切割点必须是 user/assistant 消息（不能在 toolResult 处切）
    │
    ├── 4. 分离消息:
    │      ├── messagesToSummarize: 旧的要丢弃的消息
    │      ├── turnPrefixMessages: 如果切在一轮中间，被截断的前半部分
    │      └── 保留的消息: 切割点之后的消息
    │
    └── 5. extractFileOperations(): 从消息中提取文件操作记录

compact(preparation, model, apiKey)
    │
    ├── 1. 生成摘要格式:
    │      ## Goal → ## Constraints → ## Progress → ## Key Decisions → ## Next Steps → ## Critical Context
    │
    ├── 2. 如果有 turnPrefix:
    │      并行生成: [历史摘要] + [轮前缀摘要]
    │      合并为一个 summary
    │
    ├── 3. 迭代更新:
    │      如果有 previousSummary → 使用 UPDATE_SUMMARIZATION_PROMPT 合并
    │      首次 → 使用 SUMMARIZATION_PROMPT 全新生成
    │
    ├── 4. 追加文件操作列表
    │      ## Files Read + ## Files Modified
    │
    └── 5. 返回 CompactionResult { summary, firstKeptEntryId, tokensBefore, details }
```

### 5.4 压缩后的上下文结构

```
[压缩摘要消息(compactionSummary)] → [保留的旧消息] → [新消息]
```

SessionManager 写入一个 `compaction` 条目，包含 `summary` 和 `firstKeptEntryId`。
下次 `buildSessionContext()` 时：从 compaction 摘要开始 + 保留消息 + 后续消息。

## 八、Auto-Retry — 自动重试机制

### 8.1 可重试的错误

正则匹配：`overloaded|rate_limit|429|500|502|503|504|service_unavailable|fetch failed|...`

**不可重试**: Context Overflow（由压缩处理）

### 8.2 重试流程

```
agent_end → _isRetryableError(lastAssistant)
    │
    ├── 同步创建 _retryPromise（防止 prompt() 里 waitForRetry() 竞态）
    │
    └── _handleRetryableError()
         ├── 超过 maxRetries → 报失败，重置
         └── 未超过:
              ├── delayMs = baseDelayMs * 2^(attempt-1)  指数退避
              ├── 从 agent state 移除错误消息（保留在 session 历史中）
              ├── sleep(delayMs)  可被 abort
              └── setTimeout → agent.continue()  重新发送
```

### 8.3 与 prompt() 的协调

```typescript
async prompt(text) {
  // ... 发送 prompt
  await agent.prompt(messages)
  await this.waitForRetry()  // ← 等待重试链完成
}
```

`waitForRetry()` 确保调用者在所有重试完成后才返回。
