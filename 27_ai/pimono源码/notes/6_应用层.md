# 6. 应用层：mom / pods / web-ui 深度源码分析

**mom** —— 极简自管理 Slack Agent

- 5 个原子工具（bash/read/write/edit/attach）+ Docker 沙箱 = 无限能力
- 双层日志（log.jsonl 信源 + context.jsonl LLM 上下文）解决离线消息同步
- 事件系统基于文件系统 watch + cron，Agent 可自建定时任务
- "bash 即 MCP"——反 MCP 协议，用 shell 命令替代工具注册

**pods** —— GPU 集群 LLM 部署

- SSH 作为唯一远程执行层，纯 shell 脚本编排 vLLM 启动
- `models.json` 是经验知识的编码化——预调优的 GPU/模型参数组合
- 智能 GPU 分配：轮询找最低负载 GPU，支持同机多模型部署

**web-ui** —— 浏览器 AI 聊天组件库

- Light DOM + Tailwind 的务实选择（样式一致 > 组件封装）
- 流式渲染分离：MessageList（稳定）+ StreamingMessageContainer（高频更新）
- 沙箱 iframe + `SandboxRuntimeProvider` 实现可插拔的安全执行环境
- Artifacts = 浏览器内虚拟文件系统，构建 "REPL 生成数据 → Artifact 可视化" 的工作流闭环

**贯穿的设计信念**：Agent 不需要"平台"，只需要"原语"——bash、SSH、iframe 沙箱。给最基础的能力，让 Agent 自己组合出复杂行为。

pi-mono 的应用层由三个包组成，分别面向三个场景：**Slack 机器人 (mom)**、**GPU 集群 LLM 部署 (pods)**、**浏览器聊天 UI (web-ui)**。它们共享底层的 `pi-agent-core` 和 `pi-ai`，但各自的架构哲学截然不同。

---

## 一、mom —— "自管理" Slack Bot Agent

### 1.1 核心理念：Minimal + Self-Managing

mom 的设计哲学是 **给 LLM 一个 bash，让它自己搞定一切**。它不预装任何工具、不预定义任何工作流，而是：

- 给 Agent 5 个原子工具：`bash`、`read`、`write`、`edit`、`attach`
- 让 Agent 自己通过 bash 安装 apk/npm/pip 包、编写脚本、配置凭证
- 一切 Agent 创建的脚本以 **Skills（CLI 工具）** 形式组织，下次直接复用

这是一种 **"No MCP"** 的理念——作者明确反对 MCP 协议的复杂性，认为 shell 命令就是最通用的"工具协议"。

### 1.2 架构分层

```
main.ts          入口：解析 CLI 参数、启动 Slack bot、初始化事件系统
  ├── slack.ts   Slack Socket Mode 连接、消息收发、队列化
  ├── agent.ts   构建 Agent 实例、系统提示词、事件订阅
  ├── context.ts 上下文管理：log.jsonl ↔ context.jsonl 同步
  ├── store.ts   频道文件存储：附件下载、日志持久化、去重
  ├── sandbox.ts 沙箱执行：Host 模式 vs Docker 模式
  ├── events.ts  定时事件系统：即时/一次性/周期性事件
  └── tools/     5 个原子工具
```

### 1.3 关键设计决策

#### (1) 每频道隔离的状态模型

```typescript
// main.ts 中的核心数据结构
interface ChannelState {
  running: boolean // 是否正在处理消息
  runner: AgentRunner // 每个频道独立的 Agent 运行器
  store: ChannelStore // 每个频道独立的存储
  stopRequested: boolean // 停止信号
}
const channelStates = new Map<string, ChannelState>()
```

**每个 Slack 频道 = 一个完全独立的 Agent 实例**：独立的对话历史、记忆、工具、工作目录。这是一个关键的架构选择——它使 mom 能同时服务多个频道/用户而互不干扰。

#### (2) 双层日志架构（log.jsonl + context.jsonl）

这是 mom 最精妙的设计之一：

- **log.jsonl** = 信源（Source of Truth）：记录所有频道消息（用户 + bot），不含工具调用细节
- **context.jsonl** = LLM 上下文：从 log.jsonl 同步而来，包含完整的工具调用和返回

`syncLogToSessionManager()` 负责将 log.jsonl 中 "未被 Agent 看到" 的消息（如 Agent 忙碌时的频道聊天）同步到 context。这解决了一个根本问题：**Agent 离线期间的消息也能被纳入上下文**。

去重逻辑通过文本归一化（去掉时间戳前缀、去掉附件部分）来匹配已有消息。

#### (3) Slack 消息的串行队列

```typescript
class ChannelQueue {
  private queue: QueuedWork[] = []
  private processing = false
  // 工作项按顺序执行，一次只处理一个
}
```

每个频道有独立的任务队列。当 Agent 正在运行时，新消息会入队等待。这避免了 Slack API 的并发竞态问题。

#### (4) "Respond + Thread" 双层 Slack 交互

- **主消息**：用户看到的简洁回复（累积追加，用 `...` 表示"正在工作"）
- **线程消息**：包含工具调用详情（参数 + 结果的代码块）

这是优秀的 UX 设计——让用户看到要点，细节按需查看。

#### (5) Docker 沙箱执行

```typescript
class DockerExecutor {
  async exec(command: string, options?: ExecOptions): Promise<ExecResult> {
    const dockerCmd = `docker exec ${this.container} sh -c ${shellEscape(command)}`
    return new HostExecutor().exec(dockerCmd, options)
  }
  getWorkspacePath(): string {
    return '/workspace'
  }
}
```

Docker 模式将所有 bash 命令通过 `docker exec` 隔离执行。注意 `shellEscape` 使用单引号包裹 + 内部单引号转义来防止命令注入。workspace 路径映射为容器内的 `/workspace`。

#### (6) 事件系统（Events）

```typescript
type MomEvent = ImmediateEvent | OneShotEvent | PeriodicEvent
```

基于文件系统的事件调度器：

- **Immediate**：文件一出现就触发（用于外部 webhook/脚本通知）
- **One-shot**：定时触发一次（用于提醒）
- **Periodic**：Cron 调度（用于定期任务如邮件检查）

实现上用 `fs.watch` 监听事件目录，用 `croner` 库做 cron 调度。事件触发后给 Agent 发一条伪消息 `[EVENT:filename:type:time]`。

#### (7) 系统提示词工程

`buildSystemPrompt()` 是一个精心构造的 ~2000 字提示词，包含：

- Slack mrkdwn 格式说明（不是 Markdown！）
- 频道/用户 ID 映射表
- 完整的工作区目录布局
- Skills 系统的创建规范（带 YAML frontmatter 的 SKILL.md）
- Events 系统的完整 JSON schema + cron 语法教程
- 记忆系统（全局 MEMORY.md + 频道 MEMORY.md）
- 日志查询的 jq 命令示例

### 1.4 深刻洞见

1. **"bash 即 MCP"**：mom 证明了一个激进的观点——不需要复杂的工具注册协议，一个 bash 工具 + 文件读写 = 无限能力。Agent 可以自举，编写并调用自己的 CLI 工具。
2. **Agent-as-Infrastructure**：mom 不是传统的"聊天机器人"，它是一个运行在 Docker 中的自治系统，可以安装软件、配置环境、写代码、创建定时任务——本质上是一个 **AI 运维工程师**。
3. **文件系统即数据库**：所有状态（历史、记忆、事件、技能）都是文件。这使得调试极其简单——直接查看文件即可。

## 二、pods —— GPU 集群 LLM 部署 CLI

### 2.1 核心理念：一条命令部署模型

pods 将 "在远程 GPU 机器上部署 vLLM" 这个复杂工作封装为简单的 CLI：

```bash
pi pods setup dc1 "ssh root@1.2.3.4" --mount "sudo mount ... /mnt/hf-models"
pi start Qwen/Qwen2.5-Coder-32B-Instruct --name qwen
```

### 2.2 架构分层

```
cli.ts              命令行解析与分发
  ├── config.ts     配置持久化（~/.pi/pods.json）
  ├── ssh.ts        SSH 执行抽象层
  ├── types.ts      核心类型定义
  ├── model-configs.ts  预定义模型配置
  ├── models.json   模型参数数据库
  └── commands/
      ├── pods.ts   Pod 的 setup/list/active/remove
      ├── models.ts 模型的 start/stop/list/logs
      └── prompt.ts Agent 交互式模式
```

### 2.3 关键设计决策

#### (1) 极简的数据模型

```typescript
interface Pod {
  ssh: string // SSH 命令字符串（如 "ssh root@1.2.3.4"）
  gpus: GPU[] // GPU 列表
  models: Record<string, Model> // 运行中的模型
  modelsPath?: string // 模型存储路径
  vllmVersion?: 'release' | 'nightly' | 'gpt-oss'
}

interface Config {
  pods: Record<string, Pod>
  active?: string // 当前活跃 pod
}
```

整个配置就是一个 JSON 文件。SSH 命令作为字符串存储而非结构化解析——这是故意的简化，保持最大兼容性。

#### (2) "模型配置数据库" 模式

```typescript
// models.json 定义预配置模型
{
  "Qwen/Qwen2.5-Coder-32B-Instruct": {
    "configs": [
      { "gpuCount": 1, "gpuTypes": ["H200"], "args": ["--tensor-parallel-size", "1", ...] },
      { "gpuCount": 2, "gpuTypes": ["H100"], "args": [...] }
    ]
  }
}
```

`getModelConfig()` 根据当前 GPU 类型和数量自动匹配最佳配置。这是经验知识的编码化——不同 GPU 组合下的 tensor-parallel-size、gpu-memory-utilization 等参数已预先调优。

#### (3) 智能 GPU 分配

```typescript
const selectGPUs = (pod: Pod, count: number): number[] => {
  // 统计每个 GPU 被多少个模型使用
  const gpuUsage = new Map<number, number>()
  // 返回使用最少的 GPU
  return sortedGPUs.slice(0, count)
}
```

轮询式 GPU 分配：优先选择负载最低的 GPU。这允许在同一台机器上运行多个模型（如一个大模型占 4 卡，一个小模型占 1 卡）。

#### (4) SSH 作为唯一的远程执行层

所有远程操作都通过 SSH：

- `sshExec`：执行命令并收集输出
- `sshExecStream`：流式输出（用于 setup 和 logs）
- `scpFile`：文件传输
- SSH keepalive（`ServerAliveInterval=30`）防止长时部署断连

没有用 Kubernetes、没有用 Docker API——纯 SSH。这是对简洁性的极致追求。

#### (5) 模型启动的 Shell 编排

模型启动是一个精心编排的流程：

1. 读取本地 `model_run.sh` 模板
2. 替换占位符（MODEL_ID、PORT、VLLM_ARGS）
3. 通过 SSH 的 heredoc 将脚本上传到远程
4. 用 `setsid` + `script` 命令在后台启动（保留颜色输出 + 日志文件）
5. 捕获 PID 写入本地配置

### 2.4 深刻洞见

1. **"SSH over Everything"**：pods 证明了在 AI 基础设施领域，SSH 仍然是最可靠的远程执行方案。相比 K8s 的复杂性，SSH 的优势是零依赖、零配置、通用性强。
2. **经验编码化**：`models.json` 本质上是一个社区智慧的数据库——"这个模型在这种 GPU 上应该用什么参数"。这比文档更有价值，因为它直接可执行。
3. **端口自增策略**：从 8001 开始递增分配端口，每个模型暴露独立的 OpenAI-compatible API——这使得客户端可以无缝切换模型。

## 三、web-ui —— 浏览器端 AI 聊天 UI 组件库

### 3.1 核心理念：可组合的 Web Components

web-ui 不是一个完整的应用，而是一个 **组件库**——基于 Lit + mini-lit 的 Web Components，开发者可以按需组合。

### 3.2 架构分层

```
ChatPanel.ts          顶层组件：编排 AgentInterface + ArtifactsPanel
  ├── AgentInterface   聊天界面：消息列表 + 输入框 + 流式渲染
  │   ├── MessageList  稳定的历史消息列表
  │   ├── StreamingMessageContainer  正在流式生成的消息
  │   ├── MessageEditor  输入框（支持附件）
  │   └── Messages.*   各类消息组件（User/Assistant/Tool）
  ├── ArtifactsPanel   Artifact 面板：文件式持久内容
  │   ├── HtmlArtifact   HTML 沙箱渲染
  │   ├── SvgArtifact    SVG 渲染
  │   ├── MarkdownArtifact  Markdown 渲染
  │   └── ...
  ├── Tools/
  │   ├── javascript-repl  浏览器沙箱 JS 执行
  │   ├── artifacts        文件 CRUD 工具
  │   └── extract-document 文档提取
  ├── Storage/            IndexedDB 持久层
  │   ├── AppStorage     全局存储门面
  │   ├── SessionsStore  会话存储
  │   ├── SettingsStore  设置存储
  │   └── ProviderKeysStore  API Key 存储
  └── Components/Sandbox/  沙箱运行时
      ├── SandboxedIframe     iframe 沙箱
      ├── RuntimeMessageBridge  消息桥
      └── *RuntimeProvider    运行时提供者
```

### 3.3 关键设计决策

#### (1) Light DOM 渲染策略

几乎所有组件都使用 `createRenderRoot() { return this; }`——即 **Light DOM 而非 Shadow DOM**。这意味着：

- 所有组件共享全局 Tailwind CSS 样式
- 没有样式隔离问题
- 可以用简单的 CSS 选择器定制外观

这是一个 **务实的决策**——在 AI 聊天场景中，样式一致性比组件封装更重要。

#### (2) 流式渲染的分离架构

```html
<!-- 稳定消息 —— 仅在消息完成时重渲染 -->
<message-list .messages="${messages}"></message-list>

<!-- 流式消息 —— 高频更新的独立容器 -->
<streaming-message-container .isStreaming="${isStreaming}"></streaming-message-container>
```

这是一个关键的性能优化：将"已完成的消息列表"和"正在流式生成的消息"分离到不同的组件中。`MessageList` 只在消息数量变化时重渲染，而 `StreamingMessageContainer` 处理高频的 token-by-token 更新。

#### (3) 沙箱 iframe 架构（安全执行）

```
主页面                        iframe (sandbox)
  │                              │
  │  postMessage ───────────►  │  执行用户 JS 代码
  │  ◄─────────── postMessage  │  返回结果
  │                              │
  ├── RuntimeMessageRouter      │  console拦截
  ├── ConsoleRuntimeProvider    │  文件返回
  ├── ArtifactsRuntimeProvider  │  Artifact 读写
  └── AttachmentsRuntimeProvider│  附件访问
```

JavaScript REPL 工具的执行通过 `SandboxedIframe` 实现完全隔离：

- iframe 使用 `sandbox="allow-scripts allow-modals"` 属性
- 所有通信通过 `postMessage` API
- `SandboxRuntimeProvider` 接口定义了沙箱能力注入机制

每个 Provider 包含：

- `getData()` → 注入到 window 作用域的数据
- `getRuntime()` → 被 toString() 后注入到沙箱执行的函数
- `handleMessage()` → 双向通信处理器
- `getDescription()` → 自动追加到工具描述的文档

这个设计使得 **沙箱能力可插拔**——添加新的运行时能力只需实现一个 Provider。

#### (4) Artifacts 系统

Artifacts 是 web-ui 最核心的差异化特性：

```typescript
const artifactsParamsSchema = Type.Object({
  command: StringEnum(['create', 'update', 'rewrite', 'get', 'delete', 'logs']),
  filename: Type.String(),
  content: Type.Optional(Type.String()),
  old_str: Type.Optional(Type.String()),
  new_str: Type.Optional(Type.String())
})
```

它本质上是一个 **浏览器内的虚拟文件系统**：

- `create/update/rewrite/delete` = 文件 CRUD
- `get` = 文件读取
- `logs` = 获取 HTML artifact 的 console 输出

不同文件类型自动匹配渲染器（HTML → iframe 沙箱、SVG → 直接渲染、Markdown → mini-lit 渲染等）。

关键的 UX 设计："REPL 生成数据 → 存入 Artifact → HTML Artifact 读取数据并可视化"——这构建了一个 Agent 的持久工作空间。

#### (5) 存储抽象层

```typescript
interface StorageBackend {
  get<T>(storeName: string, key: string): Promise<T | null>
  set<T>(storeName: string, key: string, value: T): Promise<void>
  delete(storeName: string, key: string): Promise<void>
  keys(storeName: string, prefix?: string): Promise<string[]>
  transaction<T>(storeNames, mode, operation): Promise<T>
}
```

`StorageBackend` 是一个多 Store 键值存储抽象。当前实现是 IndexedDB，但抽象层允许替换为远程 API。`AppStorage` 作为门面，统一暴露 Settings/ProviderKeys/Sessions/CustomProviders 四个领域 Store。

#### (6) 自动滚动的精密控制

```typescript
private _handleScroll = () => {
  // 如果用户向上滚动且距离底部 > 50px → 禁用自动滚动
  // 如果接近底部 < 10px → 重新启用
  // 忽略因消息编辑器推高导致的 viewport 缩小
};
```

自动滚动看似简单，实际需要处理：用户主动上滚查看历史、新消息到达时自动滚到底部、编辑器展开时 viewport 缩小不应误触发。这里的 `ResizeObserver` + 滚动距离阈值组合是经验之作。

### 3.4 深刻洞见

1. **"LLM 的 IDE"**：web-ui 不仅是聊天界面，它通过 Artifacts + REPL + 沙箱构建了一个 **Agent 的集成开发环境**。Agent 可以写代码、执行代码、保存结果、构建可视化——形成完整的工作流闭环。
2. **Provider 模式的动态工具描述**：`getDescription()` 让工具描述随运行时能力动态变化——如果有附件就告诉 LLM 可以读附件，如果有 Artifact 就告诉 LLM 可以读 Artifact。这是 "工具文档即 API" 理念的体现。
3. **Message Renderer Registry**：允许外部注册自定义消息渲染器，使 web-ui 成为可扩展框架。

## 四、三包协同关系与设计哲学总结

### 4.1 依赖关系

```
pi-ai          ← 模型抽象、消息类型、流式 API
pi-agent-core  ← Agent 核心（状态机、工具执行、事件系统）
pi-coding-agent ← 编程 Agent（Session、Settings、Skills）
  ↑
  ├── mom (Slack bot)         依赖 pi-coding-agent 的 Session/Skills 系统
  ├── pods (GPU deployment)   仅依赖 pi-agent-core 的 Agent 类型
  └── web-ui (browser UI)     依赖 pi-ai + pi-agent-core
```

### 4.2 贯穿的设计原则

| 原则             | mom                                | pods                        | web-ui               |
| ---------------- | ---------------------------------- | --------------------------- | -------------------- |
| **文件即数据库** | JSONL 日志、MEMORY.md、events JSON | pods.json 配置、models.json | IndexedDB            |
| **最少抽象**     | 5 个工具 + bash                    | SSH + shell 脚本            | Light DOM + 全局 CSS |
| **能力可插拔**   | Skills 系统                        | models.json                 | RuntimeProvider      |
| **Agent 自治**   | 自装工具、自管环境                 | 预定义 + 自定义 `--vllm`    | REPL + Artifacts     |

### 4.3 与 ChatGPT/Claude Web 的对比

- **ChatGPT Artifacts** = web-ui 的 Artifacts（但 web-ui 的更强：支持文件间引用、双向通信）
- **Claude Artifacts** = 类似，但 web-ui 额外支持 REPL ↔ Artifact 的数据管道
- **mom** = 无直接对标产品（Slack + Docker + 自管理 Agent 是独特组合）
- **pods** = Ollama 的远程版本（但聚焦多 GPU vLLM 部署）

### 4.4 最关键的一个洞见

pi-mono 的核心设计信念是：**Agent 不需要 "平台"，Agent 需要 "原语"**。

- mom 给的原语是 bash + 文件系统
- pods 给的原语是 SSH + vLLM
- web-ui 给的原语是 iframe 沙箱 + 虚拟文件系统

不搭建复杂的插件系统、不设计 DAG 工作流、不做可视化编排——而是给 Agent 最基础的能力，让它自己组合出复杂行为。这种 "少即是多" 的哲学在三个包中一以贯之。
