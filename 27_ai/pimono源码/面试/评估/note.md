## Agent 评估体系详解

评估是 Agent 系统最容易被忽视但最关键的环节——**没有评估就没有迭代，没有迭代 Agent 就不会变好**。

---

### 一、评估的分层结构

Agent 是一个复合系统，不能只评估"最终回答好不好"，需要**逐层评估**：

```
┌─────────────────────────────────┐
│  端到端评估（End-to-End）        │  ← 用户视角：任务完成了吗？
├─────────────────────────────────┤
│  生成质量评估                    │  ← 回答准确吗？有幻觉吗？
├─────────────────────────────────┤
│  检索质量评估（RAG）             │  ← 召回的文档相关吗？
├─────────────────────────────────┤
│  规划质量评估（Plan）            │  ← 任务分解合理吗？工具选对了吗？
├─────────────────────────────────┤
│  工具调用评估（Tool）            │  ← 参数填对了吗？调用成功了吗？
└─────────────────────────────────┘
```

---

### 二、离线评估（Offline Evaluation）

在上线前、迭代中，用**预构建的数据集**评估系统表现。

#### 2.1 Golden Dataset（黄金测试集）

| 要素     | 说明                                                                            |
| -------- | ------------------------------------------------------------------------------- |
| **构成** | `(query, expected_answer, expected_retrieved_docs, expected_tool_calls)` 四元组 |
| **来源** | 人工标注、线上日志筛选、LLM 辅助生成 + 人工审核                                 |
| **规模** | 通常 200~2000 条，覆盖核心场景 + 边界 case                                      |
| **维护** | 随知识库/工具更新持续补充，做版本管理                                           |

#### 2.2 分模块评估指标

**检索评估（Retrieval）：**

| 指标                            | 含义                                 | 公式/说明                                         |
| ------------------------------- | ------------------------------------ | ------------------------------------------------- |
| **Recall@K**                    | Top-K 结果中包含正确文档的比例       | 最重要，检索的核心是"别漏"                        |
| **Precision@K**                 | Top-K 中相关文档占比                 | 衡量噪声多不多                                    |
| **MRR**（Mean Reciprocal Rank） | 第一个正确结果排在第几位的倒数的均值 | $MRR = \frac{1}{N}\sum_{i=1}^{N}\frac{1}{rank_i}$ |
| **NDCG@K**                      | 考虑排序位置的相关性加权             | 排在前面的相关文档贡献更大                        |
| **Hit Rate**                    | 至少召回一个正确文档的查询占比       | 最直观的"能不能用"指标                            |

**生成评估（Generation）：**

| 指标                               | 含义                                     | 评估方式                                                   |
| ---------------------------------- | ---------------------------------------- | ---------------------------------------------------------- |
| **Faithfulness（忠实度）**         | 回答是否完全基于检索到的上下文，没有编造 | LLM-as-Judge：把回答拆成若干 claim，逐条验证是否有文档支撑 |
| **Answer Relevance（回答相关性）** | 回答是否切题、完整回答了问题             | LLM-as-Judge 或人工评分                                    |
| **Answer Correctness（正确性）**   | 与标准答案的语义一致性                   | 与 Golden Answer 做语义相似度 + 事实覆盖率                 |
| **Hallucination Rate（幻觉率）**   | 回答中无法被上下文支撑的 claim 比例      | 核心安全指标                                               |
| **Toxicity / Harmfulness**         | 有害/违规内容检测                        | 安全分类器                                                 |

**规划 & 工具调用评估：**

| 指标                        | 含义                                   |
| --------------------------- | -------------------------------------- |
| **Tool Selection Accuracy** | 是否选对了工具                         |
| **Parameter Accuracy**      | 工具参数是否填对                       |
| **Plan Efficiency**         | 步骤数是否合理（太多=冗余，太少=遗漏） |
| **Task Completion Rate**    | 端到端任务完成率                       |

#### 2.3 评估框架

| 框架              | 特点                                                                                            |
| ----------------- | ----------------------------------------------------------------------------------------------- |
| **RAGAS**         | 专注 RAG 评估，内置 Faithfulness / Context Relevancy / Answer Relevancy 等指标，用 LLM 自动打分 |
| **DeepEval**      | 通用 LLM 评估框架，支持自定义指标，集成 CI/CD                                                   |
| **LangSmith**     | LangChain 生态，支持 Trace + 评估一体化，人工标注界面                                           |
| **Arize Phoenix** | 开源可观测 + 评估平台，可视化 Trace                                                             |
| **Braintrust**    | 评估 + 实验管理，版本对比功能强                                                                 |

#### 2.4 LLM-as-Judge（用 LLM 评估 LLM）

核心思想：用一个**强模型**（如 GPT-4 / Claude）来给 Agent 的输出打分。

```
Prompt 示例（Faithfulness 评估）：

你是一个评估专家。请判断以下【回答】中的每个论断是否能被【参考资料】支撑。

【参考资料】: {context}
【回答】: {answer}

对每个论断，输出：
- claim: 具体论断
- verdict: "supported" | "not_supported"
- evidence: 支撑证据（如有）

最终输出 faithfulness_score = supported_count / total_count
```

**优缺点：**

| 优点                             | 缺点                                                  |
| -------------------------------- | ----------------------------------------------------- |
| 大规模自动化评估，成本远低于人工 | Judge 模型自身有偏见（position bias、verbosity bias） |
| 可快速迭代                       | 对于需要精确事实判断的场景不够可靠                    |
|                                  | 需要校准：LLM 打分与人工打分的一致性（Cohen's Kappa） |

**缓解 bias 的方法：**

- **多 Judge 投票**：用多个模型/多次采样取均值
- **Swap 位置**：对比两个回答时交换顺序，取平均
- **结构化评分**：先让 Judge 给出理由，再打分（CoT 评估）

### 三、在线评估（Online Evaluation）

上线后用**真实流量**评估。

#### 3.1 A/B Test

```
流量 ──→ [分流器] ──→ 50% 走 Agent V1（对照组）
                  ──→ 50% 走 Agent V2（实验组）
                           │
                           ▼
                    对比核心指标
```

**关键指标：**

| 类型         | 指标                                                 |
| ------------ | ---------------------------------------------------- |
| **效果指标** | 任务完成率、回答采纳率（用户点👍）、引用点击率       |
| **效率指标** | 平均轮次数（越少越好）、首次响应时间（TTFT）、总延迟 |
| **成本指标** | 平均 Token 消耗、API 调用成本                        |
| **安全指标** | 幻觉投诉率、有害输出拦截率                           |
| **留存指标** | 用户回访率、会话深度（多轮数）                       |

**注意事项：**

- 样本量要足够，必须做统计显著性检验（p-value < 0.05）
- 分流要随机且稳定（同一用户在实验期间始终在同一组）
- 避免新奇效应（Novelty Effect）：跑够时间再下结论

#### 3.2 用户反馈收集

| 类型         | 方式                                                                          |
| ------------ | ----------------------------------------------------------------------------- |
| **显式反馈** | 👍👎 按钮、满意度评分、"回答有误"举报                                         |
| **隐式反馈** | 用户是否追问同一问题（说明没解决）、是否手动修改 Agent 输出、会话是否中途放弃 |
| **标注回流** | 把 bad case 标注后加入 Golden Dataset，形成数据飞轮                           |

#### 3.3 在线监控告警

```
[实时指标] ──→ 幻觉率突升？工具失败率飙高？延迟 P99 超阈值？
              │
              ▼
         [自动告警 + 降级]
```

### 四、Agent 专用 Benchmark

通用 NLP 评测不够用，Agent 需要**任务导向**的评测集：

| Benchmark              | 评测维度                 | 说明                                                                                 |
| ---------------------- | ------------------------ | ------------------------------------------------------------------------------------ |
| **SWE-bench**          | 代码修复                 | 给一个真实 GitHub Issue，Agent 需要定位 bug 并提交 patch。最权威的 Coding Agent 评测 |
| **SWE-bench Verified** | 代码修复（人工验证子集） | 500 条经人工确认的高质量子集                                                         |
| **GAIA**               | 通用 Agent 能力          | 需要使用工具（搜索、计算器等）才能回答的复杂问题                                     |
| **WebArena**           | Web 操作                 | Agent 在真实网站上完成任务（订机票、查信息等）                                       |
| **ToolBench**          | 工具使用                 | 16000+ 真实 API，评估工具选择和编排能力                                              |
| **AgentBench**         | 综合                     | 8 种环境（OS、DB、Web 等），评估 Agent 在不同环境的操作能力                          |
| **HumanEval / MBPP**   | 代码生成                 | 代码正确性：生成代码 → 跑测试用例 → Pass@K                                           |
| **MMLU / GPQA**        | 知识推理                 | 通用知识和专家级推理能力                                                             |

**面试要点：** SWE-bench 的核心衡量指标是 **% Resolved**（成功修复的 Issue 占比），目前顶尖 Agent 在 SWE-bench Verified 上达到 ~70%+。

### 五、评估体系建设的最佳实践

```
┌──────────────────────────────────────────────────┐
│                    评估飞轮                        │
│                                                    │
│   上线 → 收集反馈 → 标注 bad case → 加入测试集    │
│     ↑         → 优化 Prompt/RAG/Tool              │
│     │         → 离线回归测试                       │
│     └──────── → 通过 → 上线                       │
└──────────────────────────────────────────────────┘
```

| 实践               | 说明                                                                       |
| ------------------ | -------------------------------------------------------------------------- |
| **回归测试自动化** | 每次改动跑 Golden Dataset，确保不退步（CI/CD 集成）                        |
| **分层评估**       | 不只看端到端，逐层定位瓶颈（是检索差？还是生成差？）                       |
| **对比实验**       | 每次只改一个变量（如只换 Reranker），控制变量法                            |
| **多维度看**       | 平均分高不代表好——关注长尾 bad case 和特定类别的表现（如事实类 vs 推理类） |
| **人工抽检**       | 再好的自动评估也需定期人工 Spot Check，校准 LLM Judge                      |
| **评估集不泄露**   | 评估集不能出现在训练/微调数据中，否则失去评估意义                          |

### 六、面试常见追问

| 问题                            | 回答要点                                                                                                         |
| ------------------------------- | ---------------------------------------------------------------------------------------------------------------- |
| **离线评估和在线评估的关系？**  | 离线是必要非充分条件——离线指标好是上线的前提，但最终要看线上真实效果。两者指标可能不一致（离线好 ≠ 线上好）      |
| **如何评估 Agent 的规划能力？** | 对比 Agent 的实际执行路径与专家标注的最优路径，看步骤冗余度、工具选择准确率、任务完成率                          |
| **评估集多大算够？**            | 取决于置信度要求。粗略标准：200+ 覆盖核心场景；统计显著性通常需要每组 1000+ 样本                                 |
| **LLM-as-Judge 可靠吗？**       | 有偏见但实用。关键是校准——计算与人工标注的一致性（Cohen's Kappa > 0.6 可接受），并用多 Judge + 位置交换缓解 bias |
| **SWE-bench 分数意味着什么？**  | 衡量 Agent 自主修 bug 的能力。高分需要：理解 Issue → 定位代码 → 写 patch → 通过测试，是端到端能力的综合体现      |
