## RAG 全链路详解：Query 改写 → 检索 → Rerank → 生成

### 1. Query 改写（Query Rewriting）

用户原始 query 往往**模糊、口语化、缺乏上下文**，直接用来检索效果很差。改写的目的是将 query 变成更适合检索的形式。

**核心方法：**

| 方法                                        | 原理                                                                                          | 示例                                                                                        |
| ------------------------------------------- | --------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------- |
| **HyDE**（Hypothetical Document Embedding） | 让 LLM 先生成一个"假设性回答"，再用这个回答去做向量检索。因为假设回答与真实文档在语义空间更近 | Q: "Python 怎么读文件" → 生成假设答案 → 用答案的 embedding 去召回                           |
| **Step-back Prompting**                     | 把具体问题抽象成更高层的问题，先检索背景知识再回答细节                                        | Q: "GPT-4 的 context window 多大" → Step-back: "GPT-4 的架构参数有哪些"                     |
| **Sub-query 分解**                          | 将复杂问题拆成多个子问题，分别检索后合并                                                      | Q: "对比 Redis 和 Memcached 的持久化" → Q1: "Redis 持久化机制" + Q2: "Memcached 持久化机制" |
| **关键词扩展 / 同义词替换**                 | 补充同义词、缩写展开，增加召回覆盖度                                                          | Q: "k8s 部署" → "kubernetes 部署 deployment"                                                |
| **历史上下文融合**                          | 多轮对话时，将 history 注入当前 query 形成自包含 query                                        | User: "它的性能怎么样" → 改写为 "Redis 的性能怎么样"                                        |

**面试要点：** HyDE 的核心直觉是"答案和答案更近，问题和答案可能很远"。

---

### 2. 检索（Retrieval）

改写后的 query 进入检索阶段，从知识库中召回候选文档。

**两大范式：**

| 类型                   | 方法                                                                                             | 优点                                           | 缺点                                 |
| ---------------------- | ------------------------------------------------------------------------------------------------ | ---------------------------------------------- | ------------------------------------ |
| **稠密检索（Dense）**  | 用 Embedding 模型（如 BGE、text-embedding-3）将 query 和文档编码为向量，做 ANN（近似最近邻）搜索 | 语义理解能力强，能匹配意思相同但措辞不同的内容 | 对精确关键词/专有名词/编号匹配弱     |
| **稀疏检索（Sparse）** | BM25、TF-IDF 等基于词频的传统方法                                                                | 精确匹配强，对专有名词友好，速度快             | 无法理解语义，同义词不同词就召回不到 |

**最佳实践 → 混合检索（Hybrid Search）：**

```
Score_final = α × Score_dense + (1-α) × Score_sparse
```

- 用 **RRF（Reciprocal Rank Fusion）** 或加权线性融合合并两路结果
- 典型配置：α = 0.7（偏向语义），根据领域调整

**向量数据库选型：**

| 数据库       | 特点                                  |
| ------------ | ------------------------------------- |
| **Milvus**   | 开源，分布式，支持十亿级向量          |
| **Pinecone** | 全托管 SaaS，开箱即用                 |
| **pgvector** | PostgreSQL 插件，适合已有 PG 基础设施 |
| **Qdrant**   | Rust 实现，高性能，支持过滤           |

**Chunk 策略（构建期决定，影响检索质量）：**

- **Chunk 大小**：太小丢上下文，太大引入噪声。常见 256~1024 token
- **重叠（Overlap）**：相邻 chunk 重叠 10%~20%，避免语义在边界被截断
- **按语义切分**：用句子边界/段落边界切，而非硬截断
- **层次索引**：Parent-Child，先召回 small chunk 精确匹配，再返回 parent chunk 提供完整上下文

---

### 3. Rerank（重排序）

检索返回 Top-K（如 20~50）候选文档，但排序**只基于相似度得分**，精度不够。Rerank 用更精细的模型做**精排**。

**核心区别：**

|      | 检索阶段                            | Rerank 阶段                                      |
| ---- | ----------------------------------- | ------------------------------------------------ |
| 模型 | Bi-Encoder（query 和 doc 分别编码） | **Cross-Encoder**（query 和 doc 拼接后一起编码） |
| 速度 | 快（ANN 毫秒级）                    | 慢（逐对打分）                                   |
| 精度 | 一般                                | **高**（能捕捉 query-doc 交互特征）              |
| 用途 | 从百万文档中粗筛                    | 从几十个候选中精选                               |

**常用 Reranker：**

- **Cohere Rerank**：API 服务，效果好
- **BGE-Reranker**：开源 Cross-Encoder
- **ColBERT**：Late Interaction，兼顾速度和精度
- **LLM as Reranker**：用 GPT/Claude 直接给候选排序（成本高但效果最好）

**Rerank 流程：**

```
检索返回 Top-50 → Cross-Encoder 逐对打分 → 取 Top-5 → 送入生成
```

**面试要点：** 为什么不直接用 Cross-Encoder 检索？因为它需要 O(N) 次前向传播（N=文档总数），不可能对百万文档逐一计算。所以必须先粗筛再精排，这就是**检索-重排两阶段架构**的原因。

---

### 4. 生成（Generation）

将 Rerank 后的 Top-K 文档作为 context，与原始 query 一起送入 LLM 生成最终回答。

**Prompt 模板设计：**

```
你是一个专业助手。请根据以下参考资料回答用户问题。
如果资料中没有相关信息，请说明你不知道，不要编造。

【参考资料】
{doc_1}
{doc_2}
{doc_3}

【用户问题】
{query}

【回答要求】
- 引用具体来源
- 如有不确定，标注置信度
```

**关键设计点：**

| 点                       | 说明                                                                  |
| ------------------------ | --------------------------------------------------------------------- |
| **引用溯源（Citation）** | 要求 LLM 标注每句话来自哪个文档，方便用户验证，减轻幻觉               |
| **Faithfulness 约束**    | Prompt 中明确"仅根据提供的资料回答"，防止模型用自身参数知识编造       |
| **降级策略**             | 如果检索结果相关度都低于阈值 → 告诉用户"未找到相关信息"，而非强行生成 |
| **流式输出**             | 大段回答用 streaming 提升体验                                         |
| **多文档冲突**           | 检索到的多个文档信息矛盾时，需要模型识别并标注分歧                    |

### 全链路一图流

```
用户: "K8s 中 Pod 被 OOMKilled 怎么排查"
           │
           ▼
    ┌──────────────┐
    │  Query 改写   │  → "Kubernetes Pod OOMKilled 排查 内存溢出 troubleshooting"
    └──────┬───────┘
           ▼
    ┌──────────────┐
    │  混合检索     │  → Dense(向量) + Sparse(BM25) → 召回 Top-30
    └──────┬───────┘
           ▼
    ┌──────────────┐
    │  Rerank      │  → Cross-Encoder 精排 → Top-5
    └──────┬───────┘
           ▼
    ┌──────────────┐
    │  LLM 生成    │  → 结合 Top-5 文档 + Query 生成回答 + 引用标注
    └──────────────┘
```

### 面试常见追问

1. **检索失败了怎么办？** → 降级到 LLM 自身知识 + 明确标注"未找到权威来源"
2. **Chunk 太多 token 超限？** → Map-Reduce（每个 chunk 先独立摘要，再合并生成）或 Refine（迭代式精炼）
3. **如何评估 RAG 效果？** → 检索侧看 Recall@K / MRR，生成侧看 Faithfulness / Relevance / Answer Correctness（框架如 RAGAS）
4. **实时性怎么保证？** → 增量索引 + TTL 过期策略 + 检索时加时间过滤
