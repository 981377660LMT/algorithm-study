# pi-mono Agent 架构源码分析

## 整体架构映射

```
用户输入 (TUI Editor / CLI)
   ↓
[Interactive Mode — 意图识别 & Slash Commands]
   ↓
[Agent Session — Context Manager] ←→ [Compaction + Session (记忆)]
   ↓
[Agent Loop — Planner / Reactor] ←→ [Model Registry (Model Router)]
   ↓                                      ↓
[Tool Executor — AgentTool]         [Extension Runner (SubAgent 位)]
   ↓                                      ↓
[自校正 — Auto-Retry + Compaction]
   ↓
[TUI Streaming Render — 输出格式化]
   ↓
[Session Manager — Trace / Log / Persist]
   ↓
用户输出
```

---

## 一、构建期（Build Time）

### 1. SystemPrompt — 动态构建 + 变量插值 + 安全注入

**源码位置**: `coding-agent/src/core/system-prompt.ts`

pi-mono 的 System Prompt 不是静态字符串，而是由 `buildSystemPrompt()` **动态拼装**：

```typescript
// 多段拼接：
1. 基础角色设定："You are an expert coding assistant operating inside pi..."
2. 可用工具列表（根据当前 selectedTools 动态生成）
3. 工具使用指南（条件注入，如 hasRead && hasEdit 时才注入 "先读后改" 指南）
4. 项目上下文（AGENTS.md / CLAUDE.md，从 cwd 向上递归扫描）
5. Skills 列表（从 ~/.pi/agent/skills/ 和 .pi/skills/ 加载）
6. 运行时变量：${dateTime}、${resolvedCwd}
7. 用户自定义追加：appendSystemPrompt
```

**面试关键点**：

- **条件注入**：不同工具组合产生不同的 Guidelines，避免 Prompt 膨胀
- **项目上下文文件**：类似 RAG 的静态注入，无需向量检索
- **Prompt 模板化**：`coding-agent/src/core/prompt-templates.ts` 支持用户定义可复用模板（带参数替换）

---

### 2. Tool — TypeBox Schema + 注册/发现 + Extension 拦截

**源码位置**: `agent/src/types.ts` + `coding-agent/src/core/tools/`

**Tool 定义接口**（TypeBox 作为 JSON Schema 源）：

```typescript
interface AgentTool<TParams extends TSchema, TDetails> {
  name: string // LLM 调用的名称
  label: string // UI 显示名
  description: string // 给 LLM 看的描述
  parameters: TParams // TypeBox Schema（自动转 JSON Schema）
  execute(toolCallId, params, signal, onUpdate): Promise<AgentToolResult>
  // UI 渲染钩子
  renderCall?: (args, theme) => Component
  renderResult?: (result, options, theme) => Component
}
```

**7 个内置工具**：`read`, `bash`, `edit`, `write`, `grep`, `find`, `ls`

**工具注册三通道**：

1. 内置工具集（`codingTools` / `readOnlyTools`）
2. Extension 动态注册：`pi.registerTool({ name, execute, ... })`
3. `createAgentSession()` 传入自定义工具

**参数校验**（`ai/src/utils/validation.ts`）：使用 **AJV** 做运行时校验 + 类型强转：

```typescript
const ajv = new Ajv({ allErrors: true, strict: false, coerceTypes: true })
// LLM 返回 string "5" → 自动转 number 5
```

**Extension 拦截**（`coding-agent/src/core/extensions/wrapper.ts`）：

```typescript
wrapToolWithExtensions(tool, runner):
  1. Before: emit tool_call → 可 block 执行
  2. Execute: 实际工具执行
  3. After: emit tool_result → 可修改返回内容
```

**面试关键点**：Schema 设计用 TypeBox（类型安全 + JSON Schema 双保险）、AJV coerceTypes 容错 LLM 输出类型不精确、Extension 拦截链实现权限控制

### 3. Skill — 文件发现 + Frontmatter 规范 + Prompt 注入

**源码位置**: `coding-agent/src/core/skills.ts`

**Skill = Tool + Prompt 的封装**，以 SKILL.md 文件为载体：

```yaml
---
name: 'browser-tools'
description: 'Interactive browser automation via Chrome DevTools Protocol'
disable-model-invocation: false
---
# Skill 详细文档...
```

**发现机制**：扫描 `~/.pi/agent/skills/` 和 `${CWD}/.pi/skills/` 下的 `SKILL.md`

**注入方式**：在 System Prompt 末尾生成 Skills 列表，LLM 知道有哪些 Skill 可用，可按需调用

**面试关键点**：Skill vs Tool 的区别 — Skill 是更高层的「能力封装」（文档+工具+提示词组合），Tool 是底层的单一可调用函数

---

### 4. Protocol — Provider Registry 模式

**源码位置**: `ai/src/api-registry.ts` + `ai/src/providers/register-builtins.ts`

pi-mono 没有实现 MCP 协议，而是自己实现了一套 **API Provider Registry**：

```typescript
interface ApiProvider<TApi extends Api> {
  api: TApi;                    // "anthropic-messages" | "openai-completions" | ...
  stream: StreamFunction;       // 流式调用
  streamSimple: StreamFunction; // 简化流式调用
}

// 8 个内置 Provider：
registerApiProvider({ api: "anthropic-messages", stream: streamAnthropic, ... });
registerApiProvider({ api: "openai-completions", stream: streamOpenAICompletions, ... });
// + azure, google, vertex, bedrock, codex, gemini-cli
```

**路由机制**：`stream()` 根据 `model.api` 字段从 registry 查找对应 provider，**一切分发逻辑都基于 Model 对象自身携带的 api 类型**。

## 二、运行期（Runtime）

### 5. Context Management — transformContext + convertToLlm 两层管道

**源码位置**: `agent/src/agent-loop.ts` (L200-230)

每次 LLM 调用前经过**两层转换**：

```
AgentMessage[]
   ↓ transformContext()    // 第一层：AgentMessage → AgentMessage（裁剪、注入外部上下文）
   ↓ convertToLlm()       // 第二层：AgentMessage → Message[]（过滤非 LLM 消息、转格式）
   ↓
LLM Context { systemPrompt, messages, tools }
```

**面试关键点**：两层分离设计 — `transformContext` 做业务层裁剪（token 管理、compaction），`convertToLlm` 做协议层适配（过滤掉 UI-only 消息如通知、状态更新）

---

### 6. Memory（记忆）— 三层实现

**源码位置**: `coding-agent/src/core/compaction/` + `coding-agent/src/core/session-manager.ts`

| 记忆层级     | pi-mono 实现                                           | 源码                       |
| ------------ | ------------------------------------------------------ | -------------------------- |
| **短期记忆** | `AgentState.messages[]` — 当前对话的全部消息           | `agent/src/types.ts`       |
| **工作记忆** | Compaction Summary — 压缩后的上下文摘要 + 文件操作追踪 | `compaction/compaction.ts` |
| **长期记忆** | Session JSONL 文件 — 完整会话持久化，支持跨会话恢复    | `session-manager.ts`       |

**Compaction 算法**（核心的记忆压缩）：

```
1. 检测 token 使用量 > reserveTokens（默认 16384）
2. 保留最近 keepRecentTokens（默认 20000）个 token 的消息
3. 将更早的消息序列化为文本，调用 LLM 生成摘要
4. 追踪文件操作（readFiles + modifiedFiles），写入摘要
5. 插入 CompactionSummaryMessage 替换被压缩的消息
```

**自动触发**：每个 turn 结束后 `_isCompactionNeeded(usage)` 检查，超限自动 compact

**Session 持久化格式**（JSONL，每行一个 Entry）：

```
Line 1: SessionHeader { version: 3, id, cwd, parentSession? }
Line 2+: SessionEntry { type: message|compaction|model_change|label|... }
```

**分支导航**：支持 DAG 形式的会话树，`fork()` 创建分支，切换时生成 `BranchSummaryMessage` 保持上下文连贯

**面试关键点**：

- 不用向量数据库，用 LLM 做摘要压缩（成本可控、效果好）
- 文件操作追踪是关键创新 — 即使消息被压缩，仍知道哪些文件被读过/改过
- JSONL append-only 存储，天然支持增量写入和崩溃恢复

---

### 7. Plan — 通过 Extension 扩展实现

**源码位置**: `coding-agent/src/cli/args.ts` (L227)

pi-mono **核心不内置 Plan 模块**，而是提供扩展点：

- Extension 可注册 `--plan` CLI flag
- `before_agent_start` 事件可注入规划指令
- Session 分支（`/tree`）允许探索不同解决路径（类似 ToT）

**实际推理模式**：依赖 LLM 本身的 CoT/Thinking 能力（通过 `thinkingLevel` 控制 thinking budget），而非显式 Plan 模块。这是 **ReAct 范式** — 观察（read）→ 思考（thinking block）→ 行动（tool call）→ 循环。

---

### 8. Model Router — Registry + Cycling + 动态切换

**源码位置**: `coding-agent/src/core/model-registry.ts` + `model-resolver.ts`

**模型选择优先级**：

```
1. --models CLI 参数指定的 scoped models
2. 恢复会话中上次使用的模型
3. settings 中的默认模型
4. 第一个有 API Key 认证的可用模型
5. 兜底：提示用户登录
```

**运行时切换**：`Ctrl+P` 循环切换模型，支持 `models.json` 自定义 provider（包括 baseUrl、apiKey 变量插值、OpenAI 兼容 API）

**成本追踪**（`ai/src/models.ts`）：

```typescript
calculateCost(model, usage):
  input × model.cost.input/M + output × model.cost.output/M
  + cacheRead + cacheWrite
```

### 9. Tool 执行（运行时）— 串行执行 + Steering 中断

**源码位置**: `agent/src/agent-loop.ts` (L280-370)

**执行流程**：

```typescript
for (toolCall of toolCalls) {
  1. emit tool_execution_start
  2. validateToolArguments(tool, toolCall)  // AJV 校验
  3. tool.execute(id, params, signal, onUpdate)  // 实际执行，支持流式更新
  4. emit tool_execution_end
  5. 构造 ToolResultMessage 返回 LLM

  // 关键：每个 tool 执行后检查 steering
  if (getSteeringMessages()) {
    skip remaining tools → 用 "Skipped due to queued user message" 替代
    break
  }
}
```

**面试关键点**：

- **串行执行**（非并行），允许用户中途 steer 打断
- Steering 机制 = Human-in-the-Loop 的实现
- 每个工具都有 `signal: AbortSignal` 支持取消

---

### 10. SubAgent — Extension 工具代理模式

pi-mono 核心**没有内置 SubAgent 编排**。但通过 Extension 的 `registerTool()` 可以实现：

```typescript
pi.registerTool({
  name: 'search_subagent',
  execute: async (id, params) => {
    // 在工具内部调用另一个 LLM / Agent
    const result = await anotherAgent.prompt(params.query)
    return { content: [{ type: 'text', text: result }], details: {} }
  }
})
```

Session 分支（`fork()`）也提供了类似 SubAgent 的探索能力。

## 三、校验期（Verification）

### 11. 自校正 — Auto-Retry + Context Overflow Recovery

**源码位置**: `coding-agent/src/core/agent-session.ts` + `ai/src/utils/overflow.ts`

**Context Overflow 检测**（20+ 正则模式覆盖所有主流 Provider）：

```typescript
const OVERFLOW_PATTERNS = [
  /prompt is too long/i, // Anthropic
  /exceeds the context window/i, // OpenAI
  /input token count.*exceeds the maximum/i, // Google Gemini
  /too many tokens/i // Generic
  // ... 共 14 个 pattern
]
```

**自动恢复链路**：

```
LLM 返回 error
  ↓ isContextOverflow() 检测
  ↓ 触发自动 compact()
  ↓ 压缩上下文后重试 LLM 调用
```

**工具错误处理**：

```typescript
try {
  result = await tool.execute(...)
} catch (e) {
  result = { content: [{ type: "text", text: e.message }], details: {} };
  isError = true;  // 标记为错误，LLM 会看到错误信息并尝试修正
}
```

**面试关键点**：错误不会终止 Agent，而是将错误信息作为 ToolResult 返回给 LLM，让 LLM 自行修正（Self-Correct）

---

### 12. 容错与恢复 — 指数退避 + 死循环防护

**源码位置**: `coding-agent/src/core/agent-session.ts`

- **可重试错误**：rate_limit、overloaded、server error → 指数退避重试
- **不可重试错误**：validation error、auth error → 直接终止
- **maxRetryDelayMs**：服务器要求等待时间超过阈值（默认 60s）→ 快速失败，交给上层处理
- **死循环防护**：Steering 机制允许用户随时打断，`AbortSignal` 支持取消

---

### 13. 安全 — 图片过滤 + 路径校验 + Extension 隔离

- **blockImages**：设置项可全局阻止图片发送到 LLM（防 Prompt Injection via image）
- **Edit 工具**：fuzzy match 确保编辑位置唯一性（>1 match 报错防误编辑）
- **Bash 工具**：进程树 kill、超时控制，防止恶意命令长时间运行
- **Extension 隔离**：Reserved keybindings 不可覆盖、工具执行前 `tool_call` 事件可拦截
- **Unicode 安全**：`ai/src/utils/sanitize-unicode.ts` 处理不合法 Unicode

## 四、运维期（Operation）

### 14. Observability — Event Stream + Session Trace

**源码位置**: `agent/src/types.ts` (AgentEvent)

pi-mono 的可观测性通过 **事件流** 实现：

```typescript
type AgentEvent =
  | { type: 'agent_start' }
  | { type: 'agent_end'; messages }
  | { type: 'turn_start' }
  | { type: 'turn_end'; message; toolResults }
  | { type: 'message_start' | 'message_update' | 'message_end' }
  | { type: 'tool_execution_start' | 'tool_execution_update' | 'tool_execution_end' }
```

- **全链路追踪**：每条消息、每个工具调用都有完整的 start/update/end 生命周期事件
- **Session JSONL**：完整审计日志，包含所有消息、模型切换、compaction 记录
- **TUI 实时渲染**：事件驱动的流式 UI 更新
- **成本追踪**：每次 API 调用的 token 使用量和费用

## 总结：pi-mono 的核心设计哲学

| 设计原则          | 实现方式                                                                                  |
| ----------------- | ----------------------------------------------------------------------------------------- |
| **简单 > 复杂**   | 不用向量 DB 做 RAG，用 LLM 摘要做 Compaction；不内置 Plan，用 ReAct + thinking            |
| **可扩展 > 内置** | Extension 系统提供各种钩子（tool_call、before_agent_start），Plan/SubAgent 都通过扩展实现 |
| **流式 > 批处理** | EventStream 异步迭代器贯穿全链路，从 LLM 响应到 UI 渲染                                   |
| **容错 > 完美**   | AJV coerceTypes 容忍 LLM 类型错误、错误返回给 LLM 自修正、overflow 自动 compact           |
| **两层抽象**      | `agent` 包 = 通用 Agent Loop，`coding-agent` 包 = 编程领域实现（工具、Session、TUI）      |
