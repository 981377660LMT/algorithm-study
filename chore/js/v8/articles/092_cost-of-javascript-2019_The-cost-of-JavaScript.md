# 2019 年 JavaScript 的交付代价：不仅仅是网络大小

# The cost of JavaScript in 2019

这是一场不仅关于流量，更关于执行效率的技术审计。

### 1. 核心观点：JS 的昂贵性远超同尺寸的图片

在 2019 年的移动设备上，1MB 的 JS 压缩包在解压后可能达到 3-5MB。解析（Parsing）和编译（Compiling）这几兆字节的代码在入门级手机上可能导致 2-4 秒的主线程完全阻塞，这种“交付震荡”是图片类资源所不具备的。

### 2. 流式解析 (Streaming) 的普及

V8 倡导开发者使用流式加载。现代浏览器可以在下载的同时在后台线程解析字节码，这使得“大文件”不再一定意味着“长等待”，关键在于如何组织代码结构以利于引擎尽早开工。

### 3. 给开发者的建议：预算管理

通过详细的数据图表展示了 JS 的生命周期：网络 -> 解析/编译 -> 运行 -> 内存占用。V8 团队强调，如果用户的 TTI（可交互时间）过长，优化后的执行逻辑（Runtime）可能不及初次加载时的“代码精简”来得有效。

### 4. 一针见血的见解

JavaScript 是 Web 应用中“最重”的一环。这篇文章是 V8 向业界传达的一个警号：不要被 5G 的速度蒙蔽，真正的瓶颈正在从“带宽”转移到移动设备那有限的“单核解析性能”上。
