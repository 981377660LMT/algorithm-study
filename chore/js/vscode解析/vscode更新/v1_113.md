## VS Code v1.113（2026 年 3 月） 深入分析：Agent 统一化与体验打磨

**发布日期**: 2026 年 3 月 25 日

### 版本定位

如果说 v1.110 是"让 Agent 真正可用"，v1.111 是"Agent 自动驾驶与生态基建"，那么 v1.113 的关键词就是 **"统一"与"收敛"**。这一版没有引入颠覆性的新概念，而是将此前分散在各处的 Agent 能力（MCP、会话管理、调试日志、定制配置）统一到本地 / CLI / Claude 三大运行时中，同时在用户体验层面做了大量"最后一公里"的打磨——统一定制编辑器、模型推理控制、图片预览、浏览器标签管理、新默认主题。

本版的核心哲学：**不是"能做什么新事"，而是"让已有的事做起来更顺"**。

---

### 1. Agent 体验：三大运行时的能力对齐

本版最大的工程主题是消除本地 Agent、Copilot CLI Agent、Claude Agent 三者之间的能力鸿沟。以前很多功能只在本地 Agent 可用，CLI 和 Claude 是"二等公民"。v1.113 系统性地补齐了这些差距。

#### A. MCP 支持扩展到 Copilot CLI & Claude Agent

此前，通过 `mcp.json` 或用户设置注册的 MCP Server 只对本地 Agent 可用。现在 VS Code 将这些 MCP Server **桥接**到 Copilot CLI 和 Claude Agent 运行时。

- **用户定义的 MCP Server**：在 VS Code 设置中注册的全局 MCP Server
- **工作区定义的 MCP Server**：`.vscode/mcp.json` 中声明的项目级别 Server

**💡 深入分析**：这意味着如果你已经为本地 Agent 配置了数据库查询、API 调用、自定义搜索等 MCP 工具，现在**无需额外配置**就能在 CLI 后台任务和 Claude Agent 中获得相同能力。这大幅降低了"配置一次，处处可用"的门槛，也解决了之前用户反馈的最大痛点之一——"后台 Agent 能力太弱"。

#### B. Copilot CLI & Claude Agent 支持 Fork 会话

设置项：`github.copilot.chat.cli.forkSessions.enabled`

之前 `/fork`（分支对话）只在本地 Chat 视图可用（v1.110 引入）。现在 Copilot CLI 和 Claude Agent 同样支持在对话历史的任意节点创建分支副本。

**💡 深入分析**：这对长时间运行的后台任务尤其有价值。例如一个 CLI Agent 正在执行复杂的重构任务，你想在某个中间状态尝试另一种方案，以前只能开一个全新会话从头来。现在可以精确地在决策分叉点 fork，大幅节省上下文重建的时间和 token 消耗。

#### C. Agent Debug Logs 扩展到 CLI 和 Claude 会话（预览）

Agent Debug Log Panel（v1.110 引入的"Agent DevTools"）现在能展示 Copilot CLI 和 Claude Agent 会话的事件流。

**💡 深入分析**：这补上了可观测性的最后一块拼图。v1.110 让本地 Agent 变透明，v1.113 让所有运行时都透明。现在无论你的 Agent 跑在哪里，都能用同一个调试面板查看系统 prompt 组装、工具调用链、hooks 触发时序。

#### D. Claude 会话列表改用官方 SDK API

**之前的问题**：VS Code 通过解析 Claude 在磁盘上的 JSONL 日志文件来列出会话和消息。一旦 Claude SDK 改变了文件格式，就会出现会话丢失或消息不完整的问题。

**现在的方案**：采用 Claude Agent SDK 的官方 API 来获取会话列表和消息内容。

**💡 深入分析**：看似小改动，实则是工程质量的重要提升。依赖文件解析是典型的脆弱集成模式（brittle integration），改用官方 API 意味着 VS Code 与 Claude 的集成从"逆向工程"升级为"正式契约"。如果你之前遇到过 Claude Agent 会话莫名消失的问题，升级到 v1.113 应该能解决。

#### E. 嵌套子代理（Nested Subagents）

设置项：`chat.subagents.allowInvocationsFromSubagents`

此前子代理（Subagent）被硬性限制不能调用其他子代理，以防止无限递归。v1.113 通过新设置解除了这一限制。

**💡 深入分析**：这是 Agent 编排能力的重大升级。考虑这样一个场景：

```
Plan Agent
├── Explore 子代理（搜索代码库，找到相关文件）
│   └── Code Review 子代理（对找到的文件进行安全审查）
└── Implementation 子代理（根据审查结果实施修改）
```

之前这种多层委托是不可能的。现在可以通过 `.agent.md` 构建**层级式工作流**。当然，默认关闭是合理的——递归调用会快速消耗 token 且有失控风险，只有在你理解工作流拓扑结构时才应开启。

这本质上是将 Agent 系统从**扁平调度**升级为**树形编排**，为更复杂的自动化流水线铺平了道路。

#### F. 插件市场管理与 URL 安装

- **`Chat: Manage Plugin Marketplaces`** 命令：集中管理所有已注册的插件市场源，支持浏览、打开本地目录、移除
- **URL Handler 安装**：通过 `vscode://chat-plugin/add-marketplace?ref=<source>` 和 `vscode://chat-plugin/install?source=<source>` 实现一键安装

**💡 深入分析**：这正是 v1.110 引入的 Agent Plugin 生态的自然延伸。URL Handler 的意义在于**降低分发门槛**——插件作者可以在自己的 README、博客、甚至 Discord 消息中放一个安装链接，用户点击即自动安装。类似于 VS Code 扩展的 `vscode://extension/publisher.extension` 模式，但面向的是 Agent 能力生态。

### 2. Chat 体验：定制化的统一入口

#### A. Chat Customizations 编辑器（预览）

这是 v1.113 面向日常用户的最重要改进。

此前，要定制 AI 的行为，你需要在多个地方操作：

- 自定义指令 → `.github/copilot-instructions.md`
- Prompt 文件 → `.github/prompts/*.prompt.md`
- 自定义 Agent → `.github/agents/*.agent.md`
- Agent Skills → `.github/skills/*/SKILL.md`
- MCP Server → `.vscode/mcp.json`
- Agent Plugins → 通过命令面板安装

这些散落在不同文件和入口中，新用户的学习曲线非常陡峭。

**新的 Chat Customizations 编辑器将所有定制统一到一个界面中**：

- **标签页组织**：自定义指令、Prompt 文件、自定义 Agent、Agent Skill 分别在不同 Tab 中
- **内嵌代码编辑器**：带有语法高亮和验证的 Markdown 编辑
- **AI 辅助生成**：可以让 AI 基于你的项目自动生成初始定制内容
- **市场浏览集成**：直接从编辑器中浏览和添加 MCP Server 及 Agent Plugin

入口：Chat 视图的齿轮图标，或 `Chat: Open Chat Customizations`

**💡 深入分析**：这解决了 Agent 定制的最大可用性问题。之前的分散配置方式对于有经验的用户来说尚可接受（因为他们了解文件结构），但对于新用户来说几乎是劝退级别的。集中式编辑器+AI 辅助生成的组合，让"5 分钟内为你的项目配置好 AI"成为可能。

从架构视角看，这也标志着 VS Code 的 Agent 定制体系从**面向文件（file-oriented）**过渡到**面向 UI（UI-oriented）**——底层仍然是文件（保证了 git 可追踪和团队协作），但上层提供了友好的图形化管理。

#### B. 模型选择器中的 Thinking Effort 控制

支持推理的模型（如 Claude Sonnet 4.6、GPT-5.4）现在在模型选择器中直接展示 **Thinking Effort 子菜单**，提供 Low / Medium / High 三档推理深度。

关键细节：

- VS Code 会**按模型记忆**选择的 Effort 级别，跨会话保留
- 模型选择器标签会显示当前 Effort 级别，如 "GPT-5.3-Codex · Medium"
- 不支持推理的模型不显示此子菜单
- 原有的 `github.copilot.chat.anthropic.thinking.effort` 和 `github.copilot.chat.responsesApiReasoningEffort` 设置已**废弃**

**💡 深入分析**：这是一个"把高级功能变成日常功能"的典型设计。之前调整推理深度需要去设置中翻找晦涩的配置项名，而且全局生效（影响所有模型和所有会话）。现在直接在选模型时顺手选 Effort，且每个模型独立记忆。

实际使用建议：

- **Low**：简单的代码补全、格式化、翻译——速度优先
- **Medium**：日常编程辅助——平衡性能和质量
- **High**：复杂的架构设计、调试、算法推理——质量优先

废弃旧设置也说明 VS Code 团队认为 UI 控制是推理深度的正确交互范式，而不是隐藏在 settings.json 中的配置项。

#### C. Chat 图片附件预览器

设置项：`imageCarousel.chat.enabled`、`imageCarousel.explorerContextMenu.enabled`

在 Chat 中涉及的图片（用户附加的截图或 Agent 通过工具调用生成的图片）现在可以点击打开**全功能图片查看器**：

| 功能        | 说明                                                             |
| ----------- | ---------------------------------------------------------------- |
| 导航        | 箭头按钮 / 键盘方向键 / 底部缩略图条                             |
| 分段        | 图片按对话轮次分组                                               |
| 缩放 & 平移 | 点击放大，Option+Click 缩小，滚轮/捏合连续缩放，高倍时可滚动平移 |
| 模态覆盖    | 以浮层形式打开，不占用编辑器 Tab                                 |

此外，Explorer 视图中的图片文件右键菜单新增 `Open in Images Preview`，会以当前文件夹所有图片打开查看器。

**💡 深入分析**：这看似是个小功能，但解决了 Agent 工作流中的真实痛点。回忆 v1.110 引入的 Browser Tools：Agent 可以截图验证页面。但截图后只是在聊天消息中以小缩略图展示，很难看清细节。现在有了全功能查看器，**Agent 截图 → 用户验证**的循环变得更加实用。按对话轮次分组的设计也很贴心——在长对话中，你可以快速找到"Agent 第三次尝试修复后的截图"。

### 3. 编辑器体验：浏览器集成与视觉刷新

#### A. 集成浏览器支持自签名证书

开发 HTTPS 应用时，自签名证书是常见需求。以前集成浏览器遇到不可信证书会直接拒绝加载，无法绕过。

现在的行为与主流浏览器一致：

- 显示证书警告页面，用户可选择"继续访问"
- 信任有效期为**一周**
- URL 栏显示"连接不安全"标识
- 随时可以**撤销信任**（Revoke and Close）

**💡 深入分析**：这是集成浏览器走向"真正可用于开发"的关键一步。之前缺少这个能力意味着任何使用 HTTPS 的本地开发项目都不得不退回外部浏览器，集成浏览器的 Agent 联动优势（截图、DOM 分析等）就完全用不上。安全设计也很合理——一周有效期+可撤销，避免了永久信任带来的安全隐患。

#### B. 浏览器标签管理增强

设置项：`workbench.browser.showInTitleBar`

随着 Agent Browser Tools 带来更多浏览器标签的使用场景，标签管理变得更加重要：

- **Quick Open Browser Tab**：类似 ⌘P 的快速跳转，但面向浏览器标签。支持过滤、聚焦、关闭。快捷键 ⇧⌘A（浏览器聚焦时可用）
- **标题栏浏览器按钮**：当有浏览器 Tab 打开时，标题栏显示地球图标按钮，可通过 `workbench.browser.showInTitleBar` 配置可见性
- **关闭所有浏览器标签**：上下文菜单新增"Close All Browser Tabs"，支持同组关闭或跨组全部关闭

**💡 深入分析**：这些看似小改动反映了 VS Code 对集成浏览器定位的战略性提升。它不再是一个"附加功能"，而是与编辑器标签平起平坐的一等公民。Quick Open 机制的引入（⇧⌘A）是一个明确信号——VS Code 希望开发者把集成浏览器当作主力开发浏览器来用。

#### C. 新默认主题

VS Code 推出了新的默认主题 **"VS Code Light"** 和 **"VS Code Dark"**，替代此前的 "Modern" 系列主题。

关键变化：

- 新用户自动使用新主题
- OS 主题同步（跟随系统深/浅色模式）默认使用新主题
- 现有用户不会被强制切换

**💡 深入分析**：默认主题的更新历来是高关注度事件（用户的审美感知直接挂钩使用体验）。此次更新的定位是"fresh and modern while maintaining familiarity"——在刷新视觉的同时不做过大改动。值得注意的是，上次更换默认主题还是 v1.71 引入 "Modern" 系列，距今已有近 3 年。

### 6. 总结与趋势

| 维度         | v1.111                        | v1.113                                   |
| ------------ | ----------------------------- | ---------------------------------------- |
| Agent 运行时 | 本地为主，CLI/Claude 仍有差距 | **三大运行时能力对齐（MCP/Fork/Debug）** |
| Agent 编排   | 子代理扁平调度                | **嵌套子代理，树形编排**                 |
| Agent 定制   | 分散在多个文件和入口          | **Chat Customizations 统一编辑器**       |
| 模型控制     | settings.json 中的配置项      | **模型选择器内联 Thinking Effort**       |
| 插件生态     | 基础安装能力                  | **市场管理 + URL Handler 一键安装**      |
| 集成浏览器   | 基本可用                      | **自签证书 + 标签管理 = 开发主力**       |
| 视觉         | Modern 系列主题               | **新 VS Code Light/Dark 主题**           |

**核心趋势**：v1.113 是一个典型的**收敛版本（Convergence Release）**。前几个版本引入了大量新概念——Autopilot、Agent Plugins、Browser Tools、Fork、Debug Panel、MCP、子代理——但它们之间存在不一致性（某些功能只在某些运行时可用）和可发现性问题（定制配置散落各处）。v1.113 的工作就是**把这些散落的拼图拼成一幅完整的图**。

从产品成熟度看，这是一个积极信号：**VS Code 的 Agent 平台已经度过了"疯狂出新功能"的阶段，开始进入"让已有功能真正好用"的阶段。** 这通常意味着该产品方向已经稳定，团队对架构有信心，开始投资于打磨和统一——这对于用户来说是最好的消息。
