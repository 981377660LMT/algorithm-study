# 古法编程的遗珠：重识架构本质

几周前，一个兄弟跟我感叹：现在满世界都是大模型的故事，大家都在探索新大陆，而我们还在搞架构设计、做稳定性、扛高并发——活像一群坚持古法编程的手艺人。
这话听着有点扎心。2025 年的今天，大模型的浪潮正压着所有人往前跑，框架、样板、工具链都在被重塑。相比之下，我们这些还在打磨架构、兜底稳定性的老技工，确实显得有些古板。
但现实是，这些古法活计，大模型暂时还没办法 100%替代。系统总要落地，稳定性必须兜底，架构评审也不能缺席。大模型可以是心智的外骨骼，却还不是能托付一切的代工厂。结构、边界、取舍，依然要靠人类自己去掂量。

---

### 1. 📋 总结摘要：快速了解核心观点

这篇文章在探讨一个焦虑与自满并存的问题：**在大模型（LLM）狂奔的时代进行“古法编程”和架构设计，还有意义吗？**

- **架构的本质与必要性：** 架构是“在混沌中建立秩序”，目的是**管理复杂度**。因为人类心智有限，面对无限的思想空间和现实业务，必须通过抽象、封装和分层来进行“认知压缩”。目前 LLM 还无法处理未经切分的整体系统复杂度，因此架构设计依然是必须的。
- **架构师的两大核心动作：**
  1.  **建模（Modeling）：** 对现实有目的的简化。不是克隆现实，而是为了特定目的（如流转、交易）提取核心。核心方法包括 E-R 模型（数据视角）、面向对象（自然视角）、DDD（业务视角）。
  2.  **划分（Partitioning）：** 切割复杂度，追求“高内聚、低耦合”。划分的依据包括技术分层、业务分模块、物理分服务。
- **难点与误区：**
  - **领域归属判定：** 依据谁守护数据一致性、上下文在哪里、变更节奏跟随谁等原则。
  - **中台与业务边界：** 介绍了无前台、大中台小前台、大前台小中台等演进模式，以及二方包 vs 插件 vs API 的耦合策略。
  - **架构 vs 框架：** 框架是工具，架构是决策。不能把“用了 Spring”当成“有了架构”。
- **思考工具：** 强调**第一性原理**。拒绝盲目类比（“大厂都这么做”），而是回归本质（“核心约束是什么”）。不要像“尤西弗罗”一样只看到现象。
- **未来展望：** 当下 LLM 是“外骨骼”，帮我们处理信息和局部推理；但在未来，一旦 LLM 学会了构建全系统的内部地图，人类架构师可能会像排字工一样消失，只剩下“定义系统目标”和“定义人机权力结构”这两件事。

### 2. 🌟 金句提取：扎心与醍醐灌顶

- **关于现状：** “大模型可以说是心智的外骨骼，却还不是能托付一切的代工厂。结构、边界、取舍，依然要靠人类自己去掂量。”
- **关于架构本质：** “架构的本质，是在混沌中建立秩序……管理复杂度，不是为了取悦 CPU 或者内存（任何逻辑正确的代码计算机都能执行），而是为了迁就人的认知。”
- **关于建模：** “若试图百分之百复刻现实，那不是建模，而是克隆……建模的目的，就是选择能够让系统最简单、协作最顺畅的抽象结构，而非寻找某个学科意义上正确的类别。”
- **关于取舍：** “没有完美的架构，只有合适的架构。架构师要清楚地知道【我们为了 A，舍弃了 B】。”
- **关于稳定性：** “复杂度不会消失，只会换位置……你选了哪种接口形态，就等于选了复杂度落点和风险传播路径。”
- **关于伪架构：** “说我们用了某某框架，所以有了某种架构，是把因果关系完全倒过来了。”
- **关于思维惰性（尤西弗罗困境）：** “当我们说‘方案 A 性能好’、‘方案 B 成本低’、‘方案 C 大厂在用’时，我们就像尤西弗罗一样，在用例子、属性和共识来思考。”
- **关于未来：** “这整套争论，很可能会被归类为【手工编程晚期，工匠们对自己的最后一点优化偏执】。”

### 3. 🧠 深度剖析：逻辑、论证与盲点

**逻辑结构：**
文章采用了非常经典的 **What-Why-How** 黄金圈结构，层层递进：

1.  **开篇立论：** 承认焦虑，确立架构在当下的不可替代性（管理复杂度）。
2.  **核心拆解：** 将架构拆解为“建模”和“划分”两个原子动作，并详细展开了 E-R/OO/DDD 和 SOLID/分层/中台策略。
3.  **辨析误区：** 专门讨论了架构与稳定性、架构与框架的区别，旨在正本清源。
4.  **思维升维：** 引入哲学思考（苏格拉底），强调第一性原理。
5.  **终局推演：** 预测 LLM 最终将如何吞噬架构师的岗位，形成闭环。

**论证方法：**

- **隐喻法：** 用“地图 vs 疆域”、“认知压缩”、“外骨骼”等比喻，把抽象概念具象化。
- **对比法：** 尤西弗罗的例子对比了“意见”与“真知”；框架 vs 架构的对比澄清了概念。
- **场景化：** 用“商品查询接口”的大而全还是小而美、“电商拍卖保证金是不是商品”等具体业务场景，极大地增强了说服力。

**潜在盲点/值得商榷之处：**

- **对“中台”的论述偏理想化：** 文中列举了中台的几种模式，但在实际落地中，组织架构斗争（康威定律）往往比技术划分更决定中台的生死。文章侧重技术理性，对组织政治的非理性因素提及较少。
- **对 LLM 的未来预测略显激进：** 作者认为 LLM 一旦建立全局视图，架构师就会像排字工一样消失。这个观点假设了业务需求是可以被“完美翻译”给 AI 的。但现实中，业务需求往往是模糊、矛盾且非逻辑的，人类架构师在“把模糊需求转化为精确逻辑”这一步的博弈沟通价值，可能被低估了。

### 4. 🛠️ 实际应用建议：架构师的生存指南

基于文章内容，给当下的技术人 4 条建议：

1.  **修炼“定义问题”的能力（第一性原理）：**

    - 在评审会上，不要沉迷于列举优缺点（那是 LLM 能做的）。
    - 逼问自己和团队：`核心约束到底是什么？`是交付速度、数据一致性还是运维成本？
    - **行动：** 下次做方案时，先写一个“约束条件清单”，并明确哪个是 No.1。

2.  **不仅要会拆，还要会“不拆”：**

    - 文章提到“复杂度守恒”。不要为了微服务而微服务。
    - **行动：** 在做服务拆分时，运用文中提到的“5 个判定标准”（数据一致性、上下文、变更节奏、权责、语义）。如果拆分后导致数据一致性维护成本剧增，果断合并。

3.  **区分“框架熟练工”与“架构设计者”：**

    - 不要把精通 Spring Cloud 等同于精通架构。
    - **行动：** 尝试脱离具体框架画出业务的领域模型图（E-R 或类图）。如果你的图里全是 `Manager` / `Service` 而没有充血的业务实体，说明你还在做面向过程的编程。

4.  **将 LLM 作为“局部复杂度”的粉碎机：**
    - 不要指望 LLM 现在就给你出全套架构方案。
    - **行动：** 用 LLM 做单点突破，比如：“检查这段代码的死锁风险”、“帮我生成这个模块的单元测试”、“优化这条慢 SQL”。把精力省下来去思考宏观的边界划分。
