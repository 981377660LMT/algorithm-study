7. **设计哲学** — 协议先行、生产消费解耦、同构多粒度、协议产品化

> Parts·造物（以下简称 Parts）是阿里巴巴 / 钉钉宜搭团队基于 LowCodeEngine 打造的「物料生产 + 资产管理 + 物料消费」一体化平台。
> 它解决的核心问题只有一句话：**让"组件"变成"物料"，让"物料"可被任意低代码平台直接使用。**

如果说 LowCodeEngine 是一台「神奇烤箱」，那么 Parts·造物就是这台烤箱背后的**乐高积木工厂 + 物流配送中心 + 上架货架系统**。本文从「它是什么 → 解决什么问题 → 整体架构 → 关键能力 → 工作流 → 与引擎的关系 → 设计哲学」七个维度做彻底的拆解。

## 一、Parts·造物到底是什么

一句话定义：

> **一个面向"低代码物料"的全生命周期工作台。**

可以用一个对照表来理解它的定位：

| 维度     | 普通 React 组件                 | Parts·造物里的「物料 (Part)」                           |
| -------- | ------------------------------- | ------------------------------------------------------- |
| 形态     | 一段 `npm` 包里的 JS/TS 代码    | 代码 + 描述协议 + 配置面板 + 缩略图 + 文档 + 版本元信息 |
| 消费方式 | `import { Button } from 'antd'` | 拖拽到画布、属性面板可视化配置、Schema 落库             |
| 谁能用   | 写代码的前端                    | 任何低代码平台 / 任何业务搭建者                         |
| 升级     | 改包发版本                      | 灰度、回滚、版本绑定、跨应用统一升级                    |
| 资产价值 | 单点复用                        | 跨平台、跨团队、跨应用的**沉淀资产**                    |

**关键词：「造物」= 把组件加工成"可被低代码识别的资产"。**

## 二、为什么需要 Parts：三个真实痛点

### 痛点 1：组件 ≠ 物料

一个 `antd` 的 `Table` 直接拖到低代码画布上是**用不了**的。引擎不知道：

- 它有哪些 props？哪些是必填？默认值是什么？
- `columns` 这种复杂结构应该用什么 setter（设置器）配置？
- 它在画布上长什么样？拖动时显示什么图标？
- 双击它是进入编辑还是切换属性？
- 它的子节点该如何被识别和投放？

要把组件升级成物料，必须额外提供一份 **物料描述协议（meta / configure）**。手写这份协议非常痛苦——通常一个中等组件要写 200~500 行 JSON。**Parts 就是那个把"写协议"自动化、可视化的工厂。**

### 痛点 2：物料散落在各团队，重复建设

A 团队封了一个「审批流程节点」，B 团队又封一个，C 团队还想再封一个，三个版本互不兼容。Parts 提供**统一的物料中心**：发布、检索、版本、订阅、灰度、依赖图谱，让物料像 npm 包一样可被全公司复用。

### 痛点 3：源码物料的研发链路太长

传统流程：写组件 → 写 demo → 写文档 → 写物料协议 → 打包 → 发 npm → 在低代码平台导入 → 调试。每改一行都要走一遍。Parts 把这条链路压缩成 **「在线 IDE → 一键预览 → 一键发布 → 自动同步到所有订阅方」**。

## 三、整体架构：四层模型

Parts 内部可以拆成四层，自底向上：

```
┌─────────────────────────────────────────────┐
│   消费层  Consumer                           │  ← 钉钉宜搭、各类低代码设计器
│   - 物料面板 / 资产包加载 / 版本协商         │
├─────────────────────────────────────────────┤
│   分发层  Distribution                       │  ← CDN、资产包、版本路由
│   - assets.json / view.js / meta.js         │
├─────────────────────────────────────────────┤
│   生产层  Production                         │  ← 在线 IDE、调试器、发布流水线
│   - 模板脚手架 / 在线编辑 / 预览沙箱          │
├─────────────────────────────────────────────┤
│   仓储层  Repository                         │  ← 物料中心、版本管理、权限
│   - 元数据 DB / 包存储 / 检索引擎            │
└─────────────────────────────────────────────┘
```

### 3.1 仓储层：物料的「中央仓库」

存的不是文件，是「**物料三件套**」：

1. **运行时包**（`view.js`）：真正在浏览器里跑的组件代码（UMD/ESM）。
2. **设计态包**（`meta.js`）：给低代码引擎用的协议描述（属性、setter、行为、嵌套规则）。
3. **资源描述**（`assets.json`）：上面两者的清单 + 版本 + 依赖 + 缩略图 + 分类。

仓储层负责：

- **唯一标识**：`scope/name@version`（类 npm）
- **版本树**：支持 `latest / beta / 灰度桶 / 业务定制版`
- **依赖图**：A 物料依赖 B 物料 → 自动级联加载
- **权限矩阵**：组织/团队/应用三级可见性

### 3.2 生产层：在线「物料 IDE」

这是 Parts 体验最有特色的部分。开发者不再需要本地搭脚手架，而是在浏览器里：

- **新建物料** → 选模板（基础组件 / 业务组件 / 区块 / 模板）
- **写代码** → 内置 Monaco + TS + 热更新
- **写协议** → 提供「**协议可视化编辑器**」：可以像填表一样勾选 props 类型、绑定 setter、声明嵌套约束，自动产出 `meta.ts`
- **调试** → 右侧实时跑一个**真实的低代码设计器沙箱**，所见即所得
- **发布** → 走流水线打包 → 上传 CDN → 写入仓储层

**核心价值：把"写协议"从一项苦力活变成一项可视化配置工作。**

### 3.3 分发层：让物料"零成本"被消费

每个物料发布后会生成一份 `assets.json`，长这样（简化版）：

```json
{
  "packages": [
    { "package": "@parts/form-pro", "version": "1.2.0", "urls": ["https://cdn/.../view.js"] }
  ],
  "components": [
    {
      "componentName": "FormPro",
      "title": "高级表单",
      "category": "表单",
      "snippets": [
        {
          "schema": {
            /* 默认插入到画布的 Schema */
          }
        }
      ],
      "configure": {
        /* 属性面板配置 */
      }
    }
  ]
}
```

低代码平台只需要 **一行 `assets.load(url)`**，所有物料的代码、协议、缩略图、setter 配置就全部到位。这是 Parts 能"跨平台流通"的关键。

### 3.4 消费层：低代码平台如何"接入" Parts

宜搭等平台的接入只做三件事：

1. **拉取资产清单**：`GET /parts/asset-bundle?app=xxx`
2. **注入引擎**：`engine.material.setAssets(assets)`
3. **订阅升级**：当 Parts 仓库里某个物料发版，推送给设计器，下一次打开自动获取新版本（或锁版本）。

## 四、关键能力详解

### 4.1 物料类型分层（重要）

Parts 把"物料"细分成 4 个层级，颗粒度从小到大：

| 层级 | 名称                     | 例子                         | 用途                           |
| ---- | ------------------------ | ---------------------------- | ------------------------------ |
| L1   | **原子组件 (Component)** | Button、Input、Table         | 最小可拖单元                   |
| L2   | **区块 (Block)**         | "登录卡片"、"用户信息表"     | 多个组件 + Schema 的预设组合   |
| L3   | **模板 (Template)**      | "后台管理首页"、"工单详情页" | 一整页 Schema                  |
| L4   | **应用 (Solution)**      | "审批中心"、"CRM 套件"       | 多页 + 数据源 + 权限的整套方案 |

四层共用同一套发布、版本、检索机制——这是 Parts 架构上的一个非常优雅的设计：**不为每种粒度造一套轮子。**

### 4.2 协议可视化编辑（造物的灵魂功能）

物料协议（configure）的核心结构是一棵 **PropConfig 树**，每个节点描述一个 prop：

```ts
{
  name: 'columns',
  title: '列配置',
  setter: {
    componentName: 'ArraySetter',
    props: {
      itemSetter: {
        componentName: 'ObjectSetter',
        props: { items: [
          { name: 'title', setter: 'StringSetter' },
          { name: 'dataIndex', setter: 'StringSetter' },
        ]}
      }
    }
  }
}
```

手写极易出错。Parts 提供 **「协议编辑器」**：

- 表格化录入：左侧字段名、中间类型、右侧 setter 选择器
- 类型推断：从 TS 类型反向生成默认协议（`Props` → `configure`）
- 实时预览：右边直接渲染出"属性面板"，所见即所得
- 协议校验：自动检测必填、循环引用、setter 不存在等问题

### 4.3 设计态 / 运行态分离

一个成熟物料常常需要"设计时长一个样，运行时长另一个样"。例如：

- `Tabs` 在画布上要**所有面板同时展开**才方便选中编辑
- `Dialog` 设计时要直接"嵌"在画布里，不能真的弹出去
- `LazyImage` 设计时要立刻显示，不要懒加载

Parts 通过两个入口产物解决：

```
view.js          → 真实运行时
view.design.js   → 设计时增强版（仅在设计器中加载）
```

引擎在设计模式下优先取后者，运行时模式下取前者。**对业务方完全透明。**

### 4.4 灰度与版本路由

每个物料版本可以打标签：

```
@parts/table@1.4.0    → latest
@parts/table@1.5.0-beta → beta
@parts/table@1.5.0    → gray (灰度 10% 应用)
```

消费侧可以指定策略：

- 全量稳定：永远拉 `latest`
- 跟随灰度：按应用 ID hash 分桶
- 锁版本：业务关键应用直接钉死 `1.4.0`

这套机制把"前端组件升级"变成了一个**可治理、可回滚、可观测**的基础设施级动作。

### 4.5 物料关系图谱

Parts 会扫描每个物料的依赖（npm + Parts 内部依赖），构建图谱：

- **影响面分析**：升级 `@parts/icon` 会影响哪些上游？
- **冗余检测**：哪些物料长期没有任何应用引用？
- **版本一致性**：同一应用里 `react` 是否被多个物料锁成不同版本？

这是大型企业落地低代码绕不开的「资产治理」能力。

## 五、典型工作流（端到端）

以"封装一个『审批节点卡片』物料"为例：

```
开发者                              Parts 平台                       宜搭设计器
  │                                    │                                │
  ├── 1. 在线新建物料 (选模板)        ─►│                                │
  │                                    │                                │
  ├── 2. 编辑 view.tsx + meta.ts     ─►│ (沙箱热更新)                  │
  │                                    │                                │
  ├── 3. 在右侧"模拟设计器"中拖拽自测 ──┤                                │
  │                                    │                                │
  ├── 4. 点"发布 v1.0.0-beta"        ─►│─► CI: lint/test/打包/上传 CDN │
  │                                    │─► 写入仓储 + 生成 assets.json │
  │                                    │                                │
  ├── 5. 在 Parts 后台勾选"灰度 10%"  ─►│                                │
  │                                    │─── push 资产更新事件 ────────► │
  │                                    │                                ├── 6. 命中灰度的应用
  │                                    │                                │     重新加载 assets
  │                                    │                                │     新物料出现在面板
  ├── 7. 验证 OK，提升为 latest     ───►│─── 全量推送 ─────────────────► │
```

整个过程：**没有 npm publish、没有本地脚手架、没有平台手动导入**。

## 六、与 LowCodeEngine 的关系

很多人会混淆两者，关键区别：

|      | LowCodeEngine      | Parts·造物             |
| ---- | ------------------ | ---------------------- |
| 角色 | 引擎（烤箱）       | 物料工厂 + 资产中心    |
| 关注 | 编排 / 渲染 / 出码 | 物料的生产、治理、分发 |
| 输入 | Schema + 物料      | 组件源代码             |
| 输出 | 真实页面 / 源码    | 标准化的物料资产包     |
| 范围 | 单个设计器实例     | 跨平台、跨应用、跨团队 |

**协作关系：**

```
   ┌──────────────┐    资产包 (assets.json)    ┌──────────────┐
   │  Parts·造物  │ ─────────────────────────► │ LowCodeEngine│
   │ （生产物料） │                             │ （消费物料） │
   └──────────────┘                             └──────────────┘
         ▲                                              │
         │ 协议规范（物料协议、资产包协议）              │
         └──────────────────────────────────────────────┘
                    都遵循同一份「标准化协议」
```

两者通过**「低代码引擎物料协议规范」**和**「资产包协议规范」**两份标准对接，因此 Parts 生产的物料可以被任何遵循协议的引擎消费，反之亦然。**这就是 lowcode-engine.cn 强调"打破物料孤岛"的本质实现。**

## 七、设计哲学：四个值得借鉴的思想

### 1. 协议先行（Protocol First）

不是先做平台再补协议，而是**先把"什么是物料"定义清楚**，平台只是协议的一个实现。结果：物料可以脱离 Parts 单独存在，引擎可以脱离 Parts 单独工作。

### 2. 生产与消费解耦

通过 CDN + assets.json，**生产侧的任何变化对消费侧只是一次 HTTP 请求**。这让"物料治理"从工程问题变成了配置问题。

### 3. 同一套机制覆盖多种粒度

组件、区块、模板、方案——形态不同，**但发布、版本、检索、权限完全同构**。少一套轮子就少一倍维护成本。

### 4. 把"写协议"产品化

最大的工程师友好之处：把最枯燥的 meta 编写做成可视化产品。这是 Parts 区别于"自己写脚手架 + 发 npm" 的最大体验差异。

---

## 八、一句话总结

> **LowCodeEngine 决定了"低代码引擎能做什么"，Parts·造物 决定了"低代码能用什么"。**
>
> 前者是引擎，后者是引擎的"燃料库 + 加油站 + 油品研发中心"。
> 没有 Parts 这样的物料中台，低代码就只能停留在 Demo；有了它，低代码才有了规模化落地的底气。
