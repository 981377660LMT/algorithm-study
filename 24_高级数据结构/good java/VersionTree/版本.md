Q：多分支，对分支版本等概念的洞察，为什么有的是 DAG，有的就是一个普通快照版本，如何全面系统理解这些产品。
这是一个关于**版本管理数据模型**的核心架构问题。下面从底层数据结构出发，系统梳理不同产品的设计选择。

---

## 核心洞察：版本历史的拓扑结构由「协作模型」决定

版本管理本质上是对**状态变迁图**的建模。不同拓扑结构的选择，取决于两个关键变量：

1. **并行编辑是否被允许？** → 决定了是否需要分支
   > 注意不是协同编辑
2. **分支之间是否需要合并？** → 决定了是否需要 DAG

---

## 三种拓扑模型

### 1. 线性链（Linear Chain）

```
S₁ → S₂ → S₃ → S₄
```

- **特征**：每个版本只有一个前驱、一个后继，无分叉无合并
- **代表产品**：
  - **Google Docs / Notion**：OT/CRDT 实时协同，冲突在字符级实时解决，宏观上看到的是线性历史
  - **数据库 Migration**（Flyway, Alembic）：严格线性，V1 → V2 → V3
  - **Semantic Versioning**：1.0.0 → 1.0.1 → 1.1.0，标签序列本身是线性的
- **适用场景**：单写者，或冲突已在更低层解决

### 2. 树（Tree）

```
S₁ → S₂ → S₃
         ↘ S₄ → S₅
```

- **特征**：允许分叉（fork），但**不允许合并**，每个节点至多一个入边
- **代表产品**：
  - **Docker 镜像层**：`FROM base` 分叉出不同镜像，层之间不合并
  - **ZFS / Btrfs 快照**：从某个时间点 fork 出只读快照，快照之间不合并
  - **Wikipedia 编辑历史**：虽有 revert，但本质是在线性链上追加"回退操作"
  - **游戏存档系统**：存档分支各自发展，不合并
- **适用场景**：需要多版本并存，但各版本独立演化

### 3. DAG（有向无环图）

```
S₁ → S₂ → S₃ → S₆
         ↘ S₄ → S₅ ↗
```

- **特征**：允许分叉 **且** 允许合并，一个节点可以有多个入边（多个 parent）
- **代表产品**：
  - **Git / Mercurial**：commit 可以有多个 parent（merge commit）
  - **Blockchain（以太坊 DAG 变种）**：IOTA Tangle
  - **Perforce Streams**：stream 之间 merge 形成 DAG
- **适用场景**：多人并行开发 + 需要将成果汇聚

## 为什么 Git 是 DAG 而不是树？

关键操作是 **merge**。

```
      A---B---C  (feature)
     /         \
D---E---F---G---H  (main)
```

commit `H` 有两个 parent：`C` 和 `G`。这打破了树结构，形成了 DAG。

Git 的每个 commit 存储的是**完整快照**（snapshot），但 commit 之间的**指向关系**（parent 指针）构成了 DAG。这里要区分两个层次：

| 层次                  | 数据模型                          | 结构 |
| --------------------- | --------------------------------- | ---- |
| **单个 commit 内部**  | 完整的目录树快照（tree of blobs） | 树   |
| **commit 之间的关系** | parent 指针网络                   | DAG  |

所以 Git 既是"快照"又是"DAG"——快照描述**状态**，DAG 描述**演化关系**。

## 统一理解框架

把所有版本管理产品放到一个二维空间中：

|              | **不可合并**                        | **可合并**                    |
| ------------ | ----------------------------------- | ----------------------------- |
| **不可分叉** | 线性链：DB Migration, Google Docs   | —（逻辑上不存在）             |
| **可分叉**   | 树：Docker 镜像, ZFS 快照, 游戏存档 | DAG：Git, Mercurial, Perforce |

再叠加第三个维度——**存储粒度**：

- **全量快照**：Git（每个 commit 存完整 tree）、ZFS（CoW 快照）
- **增量/差异**：SVN（存 delta）、DB Migration（存变更脚本）
- **操作序列**：Google Docs（OT 操作日志）、CRDT（合并操作集合）

## 几个常见困惑的澄清

**Q: SVN 也有分支，为什么不是 DAG？**
SVN 的"分支"本质上是目录拷贝（`svn copy`），merge 时只是把 diff 应用到目标目录，**不记录双 parent**。所以 SVN 的历史本质上是一棵树（甚至更接近线性链——全局 revision 号单调递增）。

**Q: Git rebase 和 merge 有什么拓扑区别？**

- `merge`：保留 DAG 结构，创建多 parent 节点
- `rebase`：将 DAG **压平**成线性链，丢弃原始分叉信息

**Q: 为什么 Docker 不需要 DAG？**
因为镜像构建是**单向**的（`FROM` → `RUN` → `RUN`），不存在"把两个镜像的层合并成一个"的需求。每个 `Dockerfile` 产生一条独立的链。

---

## 一句话总结

> **版本历史的拓扑 = f(是否允许分叉, 是否允许合并)**。线性链、树、DAG 是三种递进的复杂度，产品选择哪种取决于其协作模型对并行与汇聚的需求。Git 的 commit 是快照，commit 间的指向关系是 DAG——这两者不矛盾，分别描述的是「状态」和「演化」。
