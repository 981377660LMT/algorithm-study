将**可持久化问题**转化为**树上路径维护问题**，通过**离线（Offline）处理**来规避复杂的“可持久化线段树”。

从以下三个层面来拆解这个“Git 比喻”：

### 1. 结构比喻：从“版本”到“版本树”

在 Git 中，每一次 `commit` 都记录了对文件的修改，并且每个 `commit` 都有一个父节点。

- **状态（Commit）**：每一次“类型 2”（修改 $A_{X,Y}=Z$）的操作，都会产生一个新的数列状态，相当于一个新 `commit`。
- **指针（Branch/HEAD）**：数组 $A_i$ 并不存储完整数据，它只是一个指向某个 `commit` 的指针（或是 HEAD）。
- **关系（Tree）**：新 `commit` 会指向它的前驱 `commit`。所有的 `commit` 最终构成了一棵**版本树**。

### 2. 操作对照

你可以把这道题的操作看作在 Git 仓库里操作：

| 操作类型   | 题目描述              | Git 比喻                                                                     |
| :--------- | :-------------------- | :--------------------------------------------------------------------------- |
| **Type 1** | $A_X = A_Y$           | `git checkout -B branch_X branch_Y` (将分支 X 重置到分支 Y 的位置)           |
| **Type 2** | $A_{X,Y} = Z$         | 在 `branch_X` 上做了一次修改并 `git commit`。产生新节点，`branch_X` 指向它。 |
| **Type 3** | 查询 $A_X$ 的 $[L,R]$ | 在 `branch_X` 当前指向的 `commit` 节点上进行区间查询。                       |

### 3.核心算法：离线 + 树上遍历 + 回溯

既然所有的操作构成了多棵树（或者一个森林），我们不需要真的去开辟几万个数组空间。

1.  **建树**：
    - 先记录所有操作，建立起节点之间的父子关系。
    - 初始所有 $A_i$ 指向根节点（值为 0）。
    - “类型 2”操作：产生新节点 `new_node`，其父节点是 $A_X$ 当前指向的节点。然后更新 $A_X$ 指向 `new_node`。
    - “类型 3”查询：将查询任务挂在 $A_X$ 当前指向的节点上。

2.  **DFS 遍历（Euler Tour）**：
    - 从根节点开始向下遍历这棵“版本树”。
    - **进入节点（修改）**：当移动到某个节点时，执行该节点对应的修改（如把位置 $Y$ 的值加/减，变为 $Z$）。由于我们要支持撤销，需要记录旧值。通常用 **树状数组（Fenwick Tree）** 维护当前路径上的数值和。
    - **处理查询**：如果该节点上有挂载的查询操作，直接在树状数组上查 $[L, R]$。
    - **离开节点（回溯/Rollback）**：**关键步骤**。将刚才的修改撤销（恢复成父节点的状态），然后返回。

### 总结这种做法的优势

- **空间极省**：不需要可持久化线段树那种 $O(Q \log M)$ 的空间开销（通常涉及大量指针和新节点），只需要 $O(Q + M)$。
- **代码简单**：只需要一个普通的树状数组 + 一次 DFS。
- **跨越式思维**：将“时间（操作顺序）”维度的修改，转化为了“空间（树形结构）”维度的遍历。

**一句话总结**：把每次修改看作 Git 提交，把查询挂在对应的提交记录上，最后通过一次 DFS 遍历所有提交并在树状数组中实时增删修改，就能离线得到所有答案。
